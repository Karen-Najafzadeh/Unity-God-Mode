### ۱. جمع برداری: ترکیب واقعیت‌ها

برای درک جمع برداری، ابتدا باید به «دانش» نحوه مدیریت این آرایه‌های بسته‌بندی‌شده توسط ماشین نگاه کنیم.

#### دانشِ علوم کامپیوتر: اکسپرس SIMD
در یک برنامه استاندارد کامپیوتری، اگر بخواهید دو مجموعه از سه عدد را با هم جمع کنید، CPU ممکن است سه مرحله جداگانه انجام دهد: جمع کردن Xها، سپس Yها، سپس Zها. با این حال، موتورهای بازی مدرن از **SIMD (دستور واحد، داده‌های چندگانه)** استفاده می‌کنند. تصور کنید یک مسیر اختصاصی در بزرگراه وجود دارد که در آن یک دستور «جمع» واحد، تمام سه مؤلفه بردار را به طور همزمان هدف قرار می‌دهد. این کارایی در سطح سخت‌افزار همان دلیلی است که ما می‌توانیم هزاران بردار را در هر فریم بدون لکنت موتور اضافه کنیم.

#### مشکل اصلی: کابوس «مختصات دستی»
پیش از آنکه در موتورها جبر برداری داشته باشیم، اگر می‌خواستید شخصیتی را حرکت دهید، باید به صورت دستی هر محور را به‌روزرسانی می‌کردید. 
* **سناریو:** اگر بازیکنی در $(10, 5, 2)$ بود و به اندازه $(1, 0, -1)$ حرکت می‌کرد، باید می‌نوشتید: `x = x + 1; y = y + 0; z = z - 1;`
* **مشکل:** این کار مستعد خطای انسانی است و حرکت‌های پیچیده (مانند راه رفتن مورب در حالی که توسط باد هل داده می‌شوید) را به یک آشفتگی ریاضی تبدیل می‌کند.

#### جمع چگونه آن را حل می‌کند: نیروی مرکب
جمع برداری به ما امکان می‌دهد تأثیرات مختلف را در یک حرکت نهایی «انباشته» کنیم. لازم نیست محاسبه کنید باد شما را به کجا هل می‌دهد و پاهای شما شما را به کجا می‌برد؛ به سادگی «بردار باد» را با «بردار راه رفتن» جمع می‌کنید تا «بردار سرعت» نهایی خود را به دست آورید.

**نمونه کد: سیستم حرکت ترکیبی**
```csharp
using UnityEngine;

public class CombinedMovement : MonoBehaviour 
{
    void Update() 
    {
        // ۱. اسم: موقعیت فعلی ما (بردار موقعیت)
        Vector3 currentPos = transform.position;

        // ۲. فعل: نیروهای مختلف که بر ما اثر می‌گذارند (بردارهای جابجایی)
        Vector3 walkInput = new Vector3(1, 0, 0); // حرکت به راست
        Vector3 windPush = new Vector3(0, 0, 0.5f); // باد ملایم که به سمت شمال هل می‌دهد

        // ۳. عملیات جبری: جمع برداری
        // ما تمام «فعل‌ها» را با «اسم» جمع می‌کنیم تا واقعیت جدید خود را پیدا کنیم
        transform.position = currentPos + walkInput + windPush;
        
        // نتیجه به صورت (currentPos.x + 1, currentPos.y + 0, currentPos.z + 0.5) است
    }
}
```

### ۲. تفریق برداری: منطق «هدف‌گیری»

اگر جمع درباره جایی است که می‌روید، **تفریق** درباره جایی است که نگاه می‌کنید.

#### دانشِ علوم کامپیوتر: بردار هدایت (Steering)
در برنامه‌نویسی هوش مصنوعی، ما اغلب درباره «هدایت» صحبت می‌کنیم. برداری که توسط تفریق تولید می‌شود، از نظر فنی یک **بردار جابجایی** است که شکاف بین دو نقطه را توصیف می‌کند. این «پل» بین «خود» و «هدف» است.

#### مشکل اصلی: هوش مصنوعی کور
تصور کنید یک نگهبان هوش مصنوعی در یک پست ایستاده است. یک دزد در حال دزدکی رد شدن است.
* **مشکل:** نگهبان مختصات خود $(5, 0, 5)$ و مختصات دزد $(10, 0, 10)$ را می‌داند، اما نمی‌داند برای رهگیری دزد به کدام *سمت* راه برود.
* **فاجعه:** بدون تفریق، باید دستورات `if` پیچیده‌ای می‌نوشتید که بررسی می‌کرد آیا X دزد بزرگتر از X نگهبان است، سپس همین کار را برای Z انجام می‌دادید و غیره.

#### تفریق چگونه آن را حل می‌کند: هدف - خود
تفریق برداری (هدف - خود) بردار جدیدی ایجاد می‌کند که مستقیماً از مبدأ به سمت هدف اشاره می‌کند. این عملیات جهت و فاصله را در یک حرکت واحد ارائه می‌دهد.

**نمونه کد: جهت «نگاه به» هوش مصنوعی**
```csharp
public class AIGuard : MonoBehaviour 
{
    public Transform thief;

    void Update() 
    {
        // عملیات جبری: تفریق
        // همیشه «هدف» منهای «خود» تا جهت به سمت هدف به دست آید
        Vector3 directionToThief = thief.position - transform.position;

        // اکنون می‌توانیم از آن جهت برای چرخش یا حرکت استفاده کنیم
        Debug.DrawRay(transform.position, directionToThief, Color.red);
    }
}
```

### ۳. معیارهای اقلیدسی در برابر sqrMagnitude: «مالیات ریاضی»

در فصل ۵، ما تحلیل می‌کنیم که کامپیوتر واقعاً چگونه طول این بردارها را محاسبه می‌کند.

#### دانشِ علوم کامپیوتر: گلوگاه ریشه دوم
برای یافتن فاصله بین دو نقطه، کامپیوتر از **فرمول فاصله اقلیدسی** استفاده می‌کند: $\sqrt{x^2 + y^2 + z^2}$.
* **مشکل:** محاسبه یک **ریشه دوم (Square Root)** برای سخت‌افزار «گران» است. این یک فرآیند تکراری است که نسبت به ضرب یا جمع ساده، چرخه‌های پردازشی بسیار بیشتری می‌گیرد.

#### راه‌حل «God Mode»: sqrMagnitude
وقتی فقط نیاز به مقایسه فواصل داریم (مثلاً «آیا بازیکن در فاصله ۵ متری است؟»)، می‌توانیم ریشه دوم را حذف کنیم. به جای بررسی اینکه $Distance < 5$، بررسی می‌کنیم که $Distance^2 < 25$.
* **مزیت:** این یک اجرای سخت‌افزاری $O(1)$ است، به این معنی که با حداکثر سرعتی که سیلیکون اجازه می‌دهد اتفاق می‌افتد.

**نمونه کد: بررسی فاصله با عملکرد بالا**
```csharp
void DetectionCheck() 
{
    float detectionRadius = 10f;
    Vector3 offset = player.position - transform.position;

    // کند: از Vector3.Distance استفاده می‌کند که یک ریشه دوم انجام می‌دهد
    if (Vector3.Distance(transform.position, player.position) < detectionRadius) { /* منطق */ }

    // GOD MODE: از sqrMagnitude استفاده می‌کند (بدون ریشه دوم)
    // ما در برابر شعاع به توان دو مقایسه می‌کنیم (10 * 10 = 100)
    if (offset.sqrMagnitude < 100f) 
    {
        Debug.Log("تشخیص با سرعت بالا فعال شد!");
    }
}
```

### ۴. نرمال‌سازی برداری: پروتکل «جهت خالص»

#### دانشِ علوم کامپیوتر: باگ سرعت مورب
در روزهای اولیه بازی‌های دوبعدی، اگر «راست» را فشار می‌دادید، با سرعت ۱ حرکت می‌کردید. اگر «بالا» را فشار می‌دادید، با سرعت ۱ حرکت می‌کردید.
* **مشکل:** اگر هر دو را فشار می‌دادید، به صورت مورب حرکت می‌کردید. به دلیل قضیه فیثاغورث، سرعت شما تقریباً ۱.۴۱ می‌شد. بازیکنان عمداً به صورت مورب حرکت می‌کردند تا «تقلب» کنند و سریع‌تر از حد انتظار حرکت کنند.

#### راه‌حل: بردار واحد
**نرمال‌سازی** عملیات جبری تقسیم یک بردار بر طول خودش است. این منجر به یک **بردار واحد (Unit Vector)** می‌شود - برداری با طول دقیقاً ۱.۰. این عملیات جهت را حفظ می‌کند اما اندازه را دور می‌ریزد، و اطمینان می‌دهد که چه مستقیم حرکت کنید و چه مورب، سرعت شما همیشه ثابت است.

**نمونه کد: حرکت نرمال‌شده**
```csharp
void MoveCharacter() 
{
    float horizontal = Input.GetAxis("Horizontal");
    float vertical = Input.GetAxis("Vertical");

    // این بردار ممکن است در صورت حرکت مورب طول ۱.۴۱ داشته باشد
    Vector3 rawInput = new Vector3(horizontal, 0, vertical);

    // عملیات جبری: نرمال‌سازی
    // ما طول را به ۱.۰ «صاف» می‌کنیم تا سرعت ثابت باشد
    Vector3 pureDirection = rawInput.normalized;

    transform.Translate(pureDirection * speed * Time.deltaTime);
}
```

### ۵. فرمالیسم ریاضی و اثبات‌ها

برای درک واقعی *چرایی* کارکرد این عملیات، باید فراتر از کد برویم و به اصول موضوعه فضای برداری در $\mathbb{R}^n$ نگاه کنیم.

#### ۵.۱ ویژگی‌های جمع برداری
فرض کنید $\vec{a} = (a_1, a_2, \dots, a_n)$ و $\vec{b} = (b_1, b_2, \dots, b_n)$.

**خاصیت جابجایی:** $\vec{a} + \vec{b} = \vec{b} + \vec{a}$
*اثبات:*
$$\vec{a} + \vec{b} = (a_1+b_1, \dots, a_n+b_n)$$
طبق خاصیت جابجایی جمع اعداد حقیقی، $a_i + b_i = b_i + a_i$.
$$\vec{a} + \vec{b} = (b_1+a_1, \dots, b_n+a_n) = \vec{b} + \vec{a}$$

#### ۵.۲ اندازه بردار و نرمال‌سازی
برای هر بردار $\vec{v} = (x, y, z)$، اندازه آن به صورت $||\vec{v}|| = \sqrt{x^2 + y^2 + z^2}$ (از قضیه فیثاغورث) تعریف می‌شود.

**نرمال‌سازی:**
یک بردار نرمال‌شده $\hat{v}$ باید در $||\hat{v}|| = 1$ صدق کند.
با داشتن $\hat{v} = \frac{\vec{v}}{||\vec{v}||}$:
$$||\hat{v}|| = ||\frac{\vec{v}}{||\vec{v}||}|| = \frac{1}{||\vec{v}||} \cdot ||\vec{v}|| = 1$$
این اثبات می‌کند که عملیات نرمال‌سازی با موفقیت هر بردار را بر روی ابرکره واحد تصویر می‌کند، فارغ از اندازه اصلی آن.

#### ۵.۳ تفریق برداری به عنوان جمع معکوس
تفریق به عنوان افزودن معکوس جمعی تعریف می‌شود: $\vec{a} - \vec{b} = \vec{a} + (-\vec{b})$، که در آن $-\vec{b} = (-b_1, -b_2, \dots, -b_n)$.
هنگام محاسبه جهت بین دو نقطه $\vec{P}$ (هدف) و $\vec{S}$ (خود)، بردار جابجایی $\vec{D} = \vec{P} - \vec{S}$ نشان‌دهنده بردار منحصربه‌فردی است که وقتی با $\vec{S}$ جمع شود، $\vec{P}$ را می‌دهد:
$$\vec{S} + \vec{D} = \vec{S} + (\vec{P} - \vec{S}) = (\vec{S} - \vec{S}) + \vec{P} = \vec{0} + \vec{P} = \vec{P}$$

### خلاصه عملیات فصل ۵
با تسلط بر این عملیات جبری—**جمع** برای نیروهای ترکیبی، **تفریق** برای محاسبه اهداف، **sqrMagnitude** برای فاصله بهینه‌شده از نظر عملکرد، و **نرمال‌سازی** برای جهت ثابت—شما **شهود فیزیکی** به دست می‌آورید. شما دیگر حدس نمی‌زنید اشیاء شما به کجا می‌روند و شروع به تعریف دقیق مرزهای ریاضی دنیای خود می‌کنید.
