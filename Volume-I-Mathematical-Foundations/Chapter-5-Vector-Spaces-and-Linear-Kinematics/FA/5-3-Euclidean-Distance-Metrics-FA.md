# معیارهای فاصله اقلیدسی
---
[Read in English](../5-3-Euclidean-Distance-Metrics.md)

در **جلد اول، فصل ۵: فضاهای برداری و سینماتیک خطی**، ما از «دستور زبان» کد به قوانین بنیادی واقعیت دیجیتال گذر می‌کنیم. در قلب این فصل، مطالعه **معیارهای فاصله اقلیدسی (Euclidean Distance Metrics)** قرار دارد—نوار اندازه‌گیری ریاضی که برای تعیین شکاف دقیق بین دو نقطه در فضای سه‌بعدی استفاده می‌شود.

برای توسعه‌دهنده‌ای که آرزوی «God Mode» را دارد، درک این موضوع فقط به معنای دانستن نحوه اندازه‌گیری یک خط نیست؛ بلکه درک «مالیات ریاضی» پنهانی است که کامپیوتر هر بار که از او سوالی درباره فاصله می‌پرسید، می‌پردازد.

---

### دانشِ علوم کامپیوتر: گلوگاه «ریشه دوم» و سرعت سخت‌افزاری $O(1)$
برای درک معیارهای اقلیدسی، کتابداری را تصور کنید که می‌تواند کتاب‌ها را فوراً جمع، تفریق و ضرب کند. با این حال، اگر از این کتابدار بخواهید «ریشه دوم» را پیدا کند، باید به اتاق پشتی برود و یک سری طولانی و تکراری از حدس‌ها و بررسی‌ها را انجام دهد تا به پاسخ برسد.

در دانش علوم کامپیوتر، محاسبه طول یک بردار (**اندازه**) به **قضیه فیثاغورث** در سه‌بعد نیاز دارد: $\sqrt{x^2 + y^2 + z^2}$. «ریشه دوم» ($\sqrt{}$) گلوگاه است. در حالی که پردازنده‌های مدرن بسیار سریع هستند، ریشه دوم همچنان یک دستورالعمل «سنگین» در مقایسه با ضرب ساده است.

وقتی در زمینه `sqrMagnitude` درباره **اجرای سخت‌افزاری $O(1)$** صحبت می‌کنیم، می‌گوییم که کامپیوتر می‌تواند «مجذور فاصله» ($x^2 + y^2 + z^2$) را با سرعت خام و حداکثری سیلیکون محاسبه کند، بدون اینکه مجبور باشد آن مرحله اضافی و کندِ یافتن ریشه دوم را انجام دهد.

---

### مشکل اصلی: کابوس «بررسی مجاورت»
تصور کنید در حال ساخت یک میدان نبرد عظیم با ۱۰۰۰ سرباز هستید. هر سرباز باید بررسی کند که آیا دشمن در «محدوده حمله» ۱۰ متری او قرار دارد یا خیر.

* **مشکل:** اگر از فرمول استاندارد فاصله اقلیدسی برای هر ۱۰۰۰ سرباز استفاده کنید، کامپیوتر باید در هر فریم (۶۰ بار در ثانیه) ۱۰۰۰ محاسبه ریشه دوم انجام دهد.
* **نتیجه:** این «مالیات ریاضی» به سرعت جمع می‌شود. نرخ فریم بازی شما ممکن است از ۶۰ FPS روان به ۳۰ FPS لکنت‌دار کاهش یابد، صرفاً به این دلیل که پردازنده تمام وقت خود را صرف انجام ریاضیات گرانِ ریشه دوم می‌کند به جای رندر گرافیک زیبا یا پردازش هوش مصنوعی پیچیده.

---

### راه‌حل: پروتکل «فاصله مجذورشده»
در اکثر سناریوهای توسعه بازی، شما واقعاً نیازی به دانستن فاصله «دقیق» (مثلاً «۷.۴۳۲ متر») ندارید. شما فقط باید بدانید که آیا فاصله **کمتر از** یک آستانه مشخص است (مثلاً «آیا کمتر از ۱۰ است؟»).

**نوآوری ریاضی:**
به جای مقایسه **فاصله** با **محدوده**، ما **فاصله مجذورشده** را با **محدوده مجذورشده** مقایسه می‌کنیم.
* اگر $Distance < 10$ باشد، آنگاه $Distance^2 < 100$ است.
* با مجذور کردن آستانه خود تنها یک بار (۱۰ * ۱۰ = ۱۰۰)، می‌توانیم فاصله هر ۱۰۰۰ سرباز را با استفاده از `sqrMagnitude` بررسی کنیم که ریشه دوم را کاملاً حذف می‌کند. این به سخت‌افزار اجازه می‌دهد تا بررسی را در یک پالس واحد و سریع انجام دهد.

---

### مثال دقیق: مین مجاورتی «هوشمند»
بیایید نگاهی بیندازیم که چگونه یک «Unity God» اسکریپتی برای یک مین مجاورتی می‌نویسد که وقتی بازیکن نزدیک می‌شود منفجر می‌شود. ما روش «آماتور» (کند) را با روش «معمار سیستم» (سریع) مقایسه می‌کنیم.

**روش کند (فاصله اقلیدسی):**
این نسخه پردازنده را مجبور می‌کند در هر فریم یک ریشه دوم محاسبه کند.
```csharp
using UnityEngine;

public class AmateurMine : MonoBehaviour
{
    public Transform player;
    public float triggerRange = 5f;

    void Update()
    {
        // Vector3.Distance به طور داخلی از فرمول ریشه دوم استفاده می‌کند
        // این «مالیات ریاضی» است که ۶۰ بار در ثانیه پرداخت می‌شود
        float dist = Vector3.Distance(transform.position, player.position);

        if (dist < triggerRange)
        {
            Explode();
        }
    }

    void Explode() { /* بوم */ }
}
```

**روش God Mode (بهینه‌سازی sqrMagnitude):**
این نسخه از مسیر **اجرای سخت‌افزاری $O(1)$** استفاده می‌کند و گلوگاه را حذف می‌کند.
```csharp
using UnityEngine;

public class GodModeMine : MonoBehaviour
{
    public Transform player;
    public float triggerRange = 5f;
    private float _sqrTriggerRange;

    void Start()
    {
        // ما محدوده را یک بار در ابتدای بازی مجذور می‌کنیم
        // 5 * 5 = 25. اکنون دیگر هرگز مجبور نیستیم ریشه دوم بگیریم!
        _sqrTriggerRange = triggerRange * triggerRange;
    }

    void Update()
    {
        // ما بردار جابجایی خالص را محاسبه می‌کنیم
        Vector3 offset = player.position - transform.position;

        // sqrMagnitude فقط (x*x + y*y + z*z) است
        // بدون ریشه دوم = حداکثر عملکرد
        if (offset.sqrMagnitude < _sqrTriggerRange)
        {
            Explode();
        }
    }

    void Explode() { /* بوم */ }
}
```

### اثبات مفهوم: چرا SqrMagnitude برتر است

برای اثبات رسمی اینکه مقایسه فواصل مجذورشده معادل مقایسه فواصل است، فاصله $d = \sqrt{x^2+y^2+z^2}$ و آستانه $r$ را در نظر بگیرید.

**قضیه:** $d < r \iff d^2 < r^2$ برای $d, r > 0$.

*اثبات:*
تابع $f(t) = t^2$ برای $t > 0$ **اکیداً صعودی** است.
اگر $0 < d < r$ باشد، اعمال تابع اکیداً صعودی $f(t) = t^2$ ترتیب نابرابری را حفظ می‌کند:
$$d^2 < r^2$$
برعکس، اگر $d^2 < r^2$ باشد، گرفتن ریشه دوم مثبت ($\sqrt{\cdot}$ که برای مقادیر مثبت نیز اکیداً صعودی است) نتیجه می‌دهد:
$$\sqrt{d^2} < \sqrt{r^2} \implies |d| < |r|$$
از آنجا که فاصله و آستانه غیرمنفی هستند ($d, r \ge 0$)، این به $d < r$ ساده می‌شود.

**استدلال پیچیدگی محاسباتی:**
فرض کنید $C_{\sqrt{}}$ تعداد چرخه‌های CPU مورد نیاز برای محاسبه یک ریشه دوم باشد.
فرض کنید $C_{mul}$ تعداد چرخه‌ها برای ضرب باشد.

* **بررسی اقلیدسی ($d < r$):** نیازمند محاسبه $d = \sqrt{x^2+y^2+z^2}$ است که $O(C_{\sqrt{}} + 3C_{mul} + 2C_{add})$ است.
* **بررسی مجذورشده ($d^2 < r^2$):** نیازمند محاسبه $d^2 = x^2+y^2+z^2$ است که $O(3C_{mul} + 2C_{add})$ است.

از آنجا که $C_{\sqrt{}} \gg C_{mul}$ است، بررسی مجذورشده به طور قابل توجهی کارآمدتر است، به خصوص هنگامی که در یک حلقه فشرده اجرا می‌شود (مثلاً به‌روزرسانی هزاران موجودیت).

### بافتار وسیع‌تر: سینماتیک کاربردی و بهینه‌سازی
در بافتار وسیع‌تر فصل ۵، **معیارهای فاصله اقلیدسی** پایه و اساس **سینماتیک کاربردی** و **بردارهای هدایت** هستند. چه در حال محاسبه مسیر برای تعقیب هوش مصنوعی باشید و چه در حال تعیین انتقال با سرعت ثابت یک پرتابه، معیارهای فاصله به موتور می‌گویند که چگونه نیروهای خود را مقیاس‌بندی کند.

با تسلط بر تفاوت بین `magnitude` و `sqrMagnitude`، شما در حال تمرین **مهندسی عملکرد** هستید. این یک موضوع اصلی است که در **جلد هفتم** ادامه می‌یابد، جایی که ما این محاسبات را به **سیستم شغلی C#** و **کامپایلر Burst** منتقل می‌کنیم تا میلیون‌ها مورد از این بررسی‌های فاصله بهینه‌شده را به طور همزمان در تمام هسته‌های CPU با استفاده از **بردارسازی SIMD** انجام دهیم.

در "Unity God Mode"، شما فقط مسئله را حل نمی‌کنید؛ بلکه آن را به گونه‌ای حل می‌کنید که به محدودیت‌ها و نقاط قوت سخت‌افزار احترام می‌گذارد.
