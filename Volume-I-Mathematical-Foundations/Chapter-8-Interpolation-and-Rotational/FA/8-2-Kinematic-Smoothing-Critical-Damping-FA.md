# نرم‌سازی کینماتیک و میرایی بحرانی (Kinematic Smoothing & Critical Damping)

---

[Read the English version](../8-2-Kinematic-Smoothing-Critical-Damping.md)

در معماری یک بازی با وفاداری بالا (High-fidelity)، تفاوت بین یک کنترل‌کننده با احساس "ارزان" و یک تجربه "پریمیوم" AAA اغلب در چند خط ریاضیات کینماتیک نهفته است. چه در حال ارکستراسیون ردیابی دوربین، نوسان رویه‌ای سلاح، یا بازخورد دینامیک رابط کاربری باشید، توانایی شما در انتقال یک سیستم از وضعیت A به وضعیت B، وزن، شخصیت و پاسخگویی بازی شما را تعریف می‌کند.

بیشتر توسعه‌دهندگان به سراغ ساده‌ترین ابزار موجود می‌روند: درون‌یابی خطی (Linear Interpolation). اما در قلمرو مهندسی God Mode، جایی که ما هم وفاداری فیزیکی و هم پایداری مستقل از نرخ فریم را می‌طلبیم، روش‌های خطی اساساً شکسته هستند.

برای دستیابی به حرکت نرم، طبیعی و واقعاً "زنده"، باید از درون‌یابی ساده فراتر رفته و به قلمرو معادلات دیفرانسیل مرتبه دوم وارد شویم—به‌طور خاص، هنر **میرایی بحرانی (Critical Damping)**.

## مبحث ۱: نفرین Lerp خطی (گلوگاه احساس)

اگر زمانی را در Unity سپری کرده باشید، بدون شک به `Vector3.Lerp` یا `Mathf.Lerp` تکیه کرده‌اید. برای عناصر رابط کاربری یا انتقال‌های ساده و غیرتعاملی، این قابل قبول است. با این حال، لحظه‌ای که `Lerp` را به یک کنترل‌کننده دوربین یا هدف حرکت کاراکتر اعمال می‌کنید، با **گلوگاه احساس (Feel Bottleneck)** مواجه می‌شوید.

### مغالطه وابسته به نرخ فریم
مسئله اصلی با `Lerp` این است که از نظر ریاضی نسبت به سرعت و شتاب بی‌تفاوت است. وقتی کدی مانند `target = Vector3.Lerp(current, destination, speed * Time.deltaTime)` می‌نویسید، یک سیستم حرکت فیزیکی تعریف نمی‌کنید؛ بلکه یک رویکرد مبتنی بر درصد تعریف می‌کنید.

با نزدیک شدن `current` به `destination`، سرعت انتقال کاهش می‌یابد زیرا فاصله باقی‌مانده کوچک می‌شود. این یک اثر "slow-in, slow-out" ایجاد می‌کند، اما این کار را بدون هیچ توجیه فیزیکی انجام می‌دهد. از آنجا که ریاضیات مستقیماً به `Time.deltaTime` فریم فعلی گره خورده است، هر نوسانی در نرخ فریم—یک فریم افت‌کرده، یک لرزش یا یک جهش ناگهانی—باعث یک "جهش سرعت" قابل مشاهده می‌شود. شیء به طور موثر پیشرفت خود را متناسب با مدت زمان فریم "تله‌پورت" می‌کند و توهم حرکت سیال و پیوسته را می‌شکند.

### پاسخ غیرطبیعی
فراتر از مسائل پایداری، `Lerp` احساس "لاستیکی" می‌دهد. از آنجا که هیچ مفهومی از تکانه (momentum) ندارد، سیستم نمی‌تواند به تغییرات ناگهانی در سرعت هدف واکنش نشان دهد. اگر شیء هدف به طور ناگهانی جهت خود را تغییر دهد، یک سیستم مبتنی بر `Lerp` برای چندین فریم به سمت هدف قدیمی حرکت می‌کند و احساس تأخیر ناگزیری از "کشیده شدن" پشت نیت بازیکن ایجاد می‌کند.

برای ساخت سیستم‌هایی که احساس می‌کنند دارای وزن، اینرسی و هوش هستند، باید رویکردهای خطی را کاملاً دور بریزیم و با گره‌های هدف خود مانند اشیاء فیزیکی موجود در یک شبیه‌سازی دینامیک و متعادل از نظر انرژی برخورد کنیم.

## مبحث ۲: موتور فیزیک هسته: مدل جرم-فنر-دمپر

برای جایگزینی درون‌یابی خطی با چیزی که دارای "وزن" فیزیکی و هوش است، پارادایم خود را از درون‌یابی هندسی به شبیه‌سازی فیزیکی تغییر می‌دهیم. استاندارد معماری برای حرکت AAA—که در همه چیز از کنترل‌کننده‌های دوربین گرفته تا پارچه رویه‌ای و هدایت AI استفاده می‌شود—**سیستم جرم-فنر-دمپر (Mass-Spring-Damper system)** است.

### شهود فیزیکی
تصور کنید شیئی که می‌خواهید حرکت دهید (دوربین شما، نشانه سلاح، یا گره کاراکتر) توسط یک فنر کشسان و نامرئی به مقصد هدف خود متصل است. این فنر شیء را با نیرویی متناسب با فاصله به سمت هدف می‌کشد. با این حال، اگر فقط از یک فنر استفاده می‌کردیم، شیء برای همیشه در اطراف هدف به جلو و عقب نوسان می‌کرد. برای جلوگیری از این، یک "دمپر" یا میراکننده مجازی اضافه می‌کنیم که در برابر سرعت مقاومت کرده و انرژی را از سیستم خارج می‌کند.

### بنیاد ریاضی
این سیستم توسط یک معادله دیفرانسیل خطی مرتبه دوم توصیف می‌شود که نیروی وارد بر شیء را با موقعیت، سرعت و شتاب آن مرتبط می‌کند:

$$m \cdot a = k(x_{target} - x) - c \cdot v$$

که در آن:
* $m$: **جرم** (تعریف می‌کند شیء چقدر "اینرسی" دارد؛ جرم بیشتر در برابر حرکت سریع مقاومت می‌کند).
* $a$: **شتاب** ($\frac{d^2x}{dt^2}$).
* $k$: **سفتی** (ثابت فنر؛ سیستم چقدر محکم به سمت هدف می‌کشد).
* $x_{target} - x$: **جابجایی** (فاصله از هدف).
* $c$: **ضریب میرایی** (اصطکاکی که از عبور شیء از هدف و نوسان آن جلوگیری می‌کند).
* $v$: **سرعت** ($\frac{dx}{dt}$).

### اجرای عملی: یک حل‌کننده ساده فنر
برای نشان دادن، در اینجا نگاهی ابتدایی به نحوه نمایش این در #C آورده شده است. اگرچه این یک رویکرد انتگرال‌گیری اویلر ساده است—که در مبحث ۴ آن را با یک حل تحلیلی قوی‌تر جایگزین خواهیم کرد—نحوه تعامل نیروها در حلقه کد را نشان می‌دهد:

```csharp
public class SimpleSpring : MonoBehaviour
{
    public Transform target;
    public float stiffness = 100f;
    public float damping = 10f;
    public float mass = 1f;

    private Vector3 velocity;

    void Update()
    {
        // محاسبه جابجایی (نیرو = k * جابجایی)
        Vector3 displacement = target.position - transform.position;
        Vector3 springForce = displacement * stiffness;

        // محاسبه نیروی میرایی (نیرو = c * سرعت)
        Vector3 dampingForce = -velocity * damping;

        // F = ma -> a = F / m
        Vector3 acceleration = (springForce + dampingForce) / mass;

        // انتگرال‌گیری (به‌روزرسانی سرعت و سپس موقعیت)
        velocity += acceleration * Time.deltaTime;
        transform.position += velocity * Time.deltaTime;
    }
}
```

## مبحث ۳: نقطه طلایی میرایی بحرانی ($\zeta = 1$)

اکنون که یک سیستم فنر-دمپر داریم، چگونه آن را تنظیم کنیم؟ رفتار سیستم توسط رابطه بین سفتی ($k$)، جرم ($m$) و ضریب میرایی ($c$) تعیین می‌شود. ما این روابط را در یک مقدار واحد و بدون واحد به نام **نسبت میرایی ($\zeta$)** ادغام می‌کنیم.

### سه حالت حرکت
با تنظیم $\zeta$، "شخصیت" فیزیکی حرکت شیء را تغییر می‌دهیم:

* **زیرمیرایی (Underdamped) ($\zeta < 1$):** میرایی خیلی ضعیف است. شیء به سمت هدف حرکت می‌کند، از آن عبور می‌کند، به جلو و عقب نوسان می‌کند و در نهایت می‌ایستد. این مانند یک ژله شل و لرزان یا یک دوربین که به درستی تنظیم نشده و پس از توقف بازیکن همچنان می‌لرزد، به نظر می‌رسد.
* **بیش‌میرایی (Overdamped) ($\zeta > 1$):** میرایی خیلی قوی است. شیء به سمت هدف می‌خزد و آنقدر در برابر حرکت مقاومت می‌کند که رسیدن به هدف ابدیت طول می‌کشد. این احساس کند، غیرپاسخگو و به طرز ناخوشایندی "سنگین" دارد.
* **میرایی بحرانی (Critically Damped) ($\zeta = 1$):** این **نقطه طلایی God Mode** است. سیستم با سریع‌ترین سرعت ممکن بدون *هیچ‌گونه* عبور از هدف یا نوسان به سمت هدف بازمی‌گردد.

### چرا میرایی بحرانی استاندارد طلایی است؟
میرایی بحرانی هدف نهایی برای تقریباً تمام سیستم‌های بازی تعاملی است. وقتی یک دوربین بازیکن را ردیابی می‌کند، می‌خواهید فوراً به موقعیت بازیکن برسد (پاسخگو)، اما مطلقاً نمی‌خواهید از آن عبور کند و بازگردد (منحرف‌کننده). دستیابی به $\zeta = 1$ تضمین می‌کند که سیستم *سریع‌ترین پاسخ ممکن* را ارائه می‌دهد که پایداری آن تضمین شده است.

### تنظیم سیستم
برای دستیابی به این حالت، ضریب میرایی ($c$) باید با جرم ($m$) و سفتی ($k$) طبق فرمول زیر مرتبط باشد:

$$c = 2\sqrt{m \cdot k}$$

### مثال‌های رفتار میرایی
برای تجسم این حالت‌ها در محیط واقعی موتور، یک دوربین که یک کاراکتر بازیکن را ردیابی می‌کند در نظر بگیرید:

* **مثال زیرمیرایی:** بازیکن یک چرخش تند انجام می‌دهد. دوربین دنبال می‌کند، اما وقتی بازیکن می‌ایستد، دوربین از پشت بازیکن "عبور می‌کند"، دوباره به جلو می‌چرخد و با لرزش می‌ایستد. این اغلب در بازی‌های با سرعت بالا به عنوان "شناور" یا "بیمارگونه" توصیف می‌شود.
* **مثال بیش‌میرایی:** بازیکن می‌چرخد. دوربین لحظه‌ای کند و قابل‌توجه طول می‌کشد تا به جهت جدید بازیکن برسد. احساس می‌شود که دوربین دارای "اینرسی" سنگین است یا زیر آب است. اگرچه نرم است، اما غیرپاسخگو احساس می‌شود.
* **مثال میرایی بحرانی:** بازیکن می‌چرخد. دوربین شتاب می‌گیرد تا بازیکن را ردیابی کند و در همان لحظه که بازیکن می‌ایستد، دوربین دقیقاً پشت کاراکتر هم‌تراز می‌شود. این احساس "قفل‌شده" یا "سفت" دارد—تعریف یک احساس پریمیوم و پاسخگو.

## مبحث ۴: حل تحلیلی بسته (دور زدن انتگرال‌گیری اویلر)

در مبحث ۲، سیستم فنر-دمپر را با استفاده از **انتگرال‌گیری اویلر** پیاده‌سازی کردیم. نیروها را اضافه کردیم، شتاب را محاسبه کردیم، سرعت را به‌روز کردیم و سپس موقعیت را به‌روز کردیم. این برای کارهای ساده کار می‌کند، اما اساساً شکننده است.

### مشکل انتگرال‌گیری عددی
انتگرال‌گیری اویلر یک تقریب است. فرض می‌کند شتاب در طول مدت یک فریم (`Time.deltaTime`) ثابت است. اگر نرخ فریم پایین باشد یا نیروها به سرعت در حال تغییر باشند (که در یک سیستم فنر اینطور است)، انتگرال‌گیری اویلر مقادیر زیادی خطای انرژی وارد می‌کند. این دلیلی است که چرا کنترل‌کننده‌های فیزیک "ساده" اغلب ناپایدار به نظر می‌رسند.

### راه حل God Mode: حساب تحلیلی
برای تسلط واقعی بر حرکت، انتگرال‌گیری عددی را کاملاً دور می‌زنیم. برای یک سیستم میرایی بحرانی، یک **راه حل تحلیلی (closed-form solution)** برای معادله دیفرانسیل وجود دارد. این بدان معناست که می‌توانیم وضعیت فعلی (موقعیت، سرعت) و وضعیت هدف خود را وارد کنیم و ریاضیات دقیقاً به ما می‌گوید بعد از هر گام زمانی دلخواه ($dt$)، فارغ از نرخ فریم، کجا باید باشیم.

برای یک سیستم با میرایی بحرانی که به سمت هدف حرکت می‌کند، موقعیت ($x$) در زمان ($t$) توسط فرمول زیر داده می‌شود:

$$x(t) = (c_1 + c_2 \cdot t) \cdot e^{-\omega_0 \cdot t}$$

### پیاده‌سازی حل‌کننده تحلیلی
برای استفاده از این در Unity، نیازی به انجام انتگرال‌گیری در هر فریم نداریم. در عوض، می‌توانیم از یک حل‌کننده تحلیلی استفاده کنیم. در زیر یک پیاده‌سازی ساده و آماده تولید از یک فنر با میرایی بحرانی آورده شده است که کاملاً مستقل از نرخ فریم است.

```csharp
public static Vector3 GetCriticallyDampedPosition(
    Vector3 currentPos, Vector3 targetPos, 
    ref Vector3 velocity, float frequency, float dt)
{
    // تبدیل فرکانس به فرکانس زاویه‌ای (امگا)
    float omega = frequency * 2f * Mathf.PI;
    
    // راه حل تحلیلی بسته برای یک فنر با میرایی بحرانی
    float exp = Mathf.Exp(-omega * dt);
    
    Vector3 displacement = currentPos - targetPos;
    Vector3 currentVelocity = velocity;
    
    // محاسبه موقعیت جدید
    Vector3 newPos = targetPos + (displacement + (currentVelocity + displacement * omega) * dt) * exp;
    
    // محاسبه سرعت جدید
    velocity = (currentVelocity + displacement * omega - (currentVelocity + displacement * omega) * omega * dt) * exp;
    
    return newPos;
}
```

## مبحث ۵: میثاق استقلال از نرخ فریم (اصلاح Delta-Time)

در مبحث قبلی، ثابت کردیم که حل‌کننده تحلیلی ما به `dt` متکی است. اگرچه این کار ریاضیات را قوی می‌کند، اما مسئولیت معماری جدیدی را ایجاد می‌کند: اطمینان از اینکه پارامترهای تنظیم ($\omega_0$، سفتی، جرم) هنگام نوسان نرخ فریم معنی‌دار باقی می‌مانند.

### فرکانس طبیعی ($\omega_0$)
به جای فکر کردن به "سفتی" و "جرم"—که تنظیم بصری آن‌ها دشوار است—به **فرکانس طبیعی ($\omega_0$)** فکر می‌کنیم. این نرخی است که سیستم در صورت عدم میرایی نوسان می‌کرد. وقتی دوربین را تنظیم می‌کنید، اساساً تعریف می‌کنید: *"چقدر سریع می‌خواهم این واکنش نشان دهد؟"* یک $\omega_0$ بالاتر به معنای دوربین سریع‌تر و "سریع‌تر" (snappier) است؛ یک $\omega_0$ پایین‌تر به معنای حس سینمایی کندتر و "سنگین‌تر" است.

### فیزیک پایداری
درک حیاتی در مهندسی God Mode این است که $\omega_0$ یک **ثابت فیزیکی سیستم حرکت شماست**، نه یک متغیر وابسته به فریم.

از آنجا که ما از فرمول تحلیلی `GetCriticallyDampedPosition` استفاده می‌کنیم، سیستم ما اساساً از نرخ فریم جدا شده است.

### میثاق
"میثاق" استقلال از نرخ فریم ساده است: **هرگز برای *تأثیرگذاری* بر پارامترهای شبیه‌سازی فیزیکی به `Time.deltaTime` تکیه نکنید.** از `dt` فقط برای *پیشبرد وضعیت* حل تحلیلی استفاده کنید. اگر دوربین شما در ماشین‌های مختلف متفاوت احساس می‌شود، خطا در نرخ فریم نیست؛ در مدیریت `dt` شماست. اطمینان حاصل کنید که حل‌کننده فیزیک در یک حلقه به‌روزرسانی قطعی در حال اجراست و متوجه خواهید شد که "احساس" فیزیکی بازی شما یکسان باقی می‌ماند.

## مبحث ۶: راز معماری: SmoothDamp در پس‌زمینه

اگر به دنبال معادل "God Mode" برای حل‌کننده تحلیلی سفارشی ما بوده‌اید، احتمالاً با `Vector3.SmoothDamp` برخورد کرده‌اید. این ابزار استاندارد صنعتی برای نرم‌سازی کینماتیک در Unity است، اما اغلب به اشتباه به عنوان یک "Lerp با نام فانتزی" شناخته می‌شود. در واقع، این یک پیاده‌سازی تخصصی از همان فیزیکی است که ما به تازگی استخراج کردیم.

### چرا SmoothDamp بهتر از پیاده‌سازی‌های ساده عمل می‌کند؟
1. **تطبیق سرعت:** برخلاف یک فنر ساده، `SmoothDamp` به شما اجازه می‌دهد یک ارجاع `currentVelocity` ارسال کنید. این بدان معناست که می‌تواند سرعت هدف را در لحظه ورود مطابقت دهد.
2. **محدودیت سرعت:** یک پارامتر داخلی `maxSpeed` را پیاده‌سازی می‌کند. اگر نیروی فنر باعث شود شیء از این سرعت فراتر رود، سرعت را کلیمپ (Clamping) می‌کند.
3. **کش کردن وضعیت داخلی:** از `ref Vector3 currentVelocity` برای حفظ وضعیت مشتق در طول فریم‌ها استفاده می‌کند.

## مبحث ۷: پیاده‌سازی‌های عملی (ردیابی سینمایی در مقابل محدوده مرده ورودی)

تئوری میرایی بحرانی جهانی است، اما کاربرد آن بسته به قصد حرکت تغییر می‌کند.

### ردیابی سینمایی (وزن زیاد، فرکانس کم)
هنگام ردیابی سینمایی بازیکن، می‌خواهیم سیستم "سنگین" احساس شود. می‌خواهیم دوربین فقط به اندازه‌ای از بازیکن عقب بماند که حس حجم فضایی ایجاد کند، بدون اینکه احساس پاسخ‌ناپذیری ایجاد کند.

### پاسخ ورودی (فرکانس بالا، میرایی شدید)
در اکشن‌های سریع یا رقابتی، "سنگینی" یک مسئولیت است. شما نیاز دارید که گره‌های هدف شما (مانند نشانه) بلافاصله احساس شوند. از یک $\omega_0$ بالا استفاده کنید.

### نوسان رویه‌ای سلاح (گره‌های زوج‌شده)
سیستم‌های نوسان سلاح پیشرفته از یک سلسله‌مراتب "پیروی" استفاده می‌کنند. مدل سلاح باید از جهت‌گیری دوربین عقب بماند. با داشتن سلاح در فاصله‌ای از مرکز دوربین، منطق فنر-دمپر تغییرات چرخشی جزئی را به قوس‌های سلاح دراماتیک و آگاه از فیزیک تبدیل می‌کند.

## مبحث ۸: کد آماده تولید: هسته مرتبه دوم ضمنی

برای پایان این ماژول، یک حل‌کننده با میرایی بحرانی کاملاً قوی، قابل استفاده مجدد و آماده تولید را پیاده‌سازی می‌کنیم.

```csharp
using UnityEngine;

public class GodModeSpring : MonoBehaviour
{
    [Header("پیکربندی حل‌کننده")]
    [Tooltip("بالاتر = سریع‌تر، پایین‌تر = کندتر")]
    [SerializeField] private float frequency = 5f;
    [Tooltip("1.0 = میرایی بحرانی")]
    [SerializeField] private float dampingRatio = 1.0f;

    [Header("ردیابی")]
    [SerializeField] private Transform target;

    private Vector3 _velocity;

    void Update()
    {
        if (target == null) return;

        // هدایت موقعیت با استفاده از حل‌کننده تحلیلی
        transform.position = CalculateCriticallyDampedPosition(
            transform.position, 
            target.position, 
            ref _velocity, 
            frequency, 
            dampingRatio, 
            Time.deltaTime
        );
    }

    public static Vector3 CalculateCriticallyDampedPosition(
        Vector3 currentPos, Vector3 targetPos, ref Vector3 velocity, 
        float frequency, float dampingRatio, float dt)
    {
        float omega = frequency * 2f * Mathf.PI;
        
        if (dampingRatio < 1.0f) // زیرمیرایی
        {
            float wD = omega * Mathf.Sqrt(1.0f - dampingRatio * dampingRatio);
            float exp = Mathf.Exp(-dampingRatio * omega * dt);
            float cos = Mathf.Cos(wD * dt);
            float sin = Mathf.Sin(wD * dt);
            
            Vector3 diff = currentPos - targetPos;
            Vector3 v = velocity + dampingRatio * omega * diff;
            
            Vector3 newPos = targetPos + exp * (diff * cos + v * sin / wD);
            velocity = exp * (velocity * cos - (diff * omega + dampingRatio * velocity) * sin / wD);
            return newPos;
        }
        else // میرایی بحرانی (zeta == 1)
        {
            float exp = Mathf.Exp(-omega * dt);
            Vector3 diff = currentPos - targetPos;
            Vector3 v = velocity + omega * diff;
            
            Vector3 newPos = targetPos + (diff + v * dt) * exp;
            velocity = (velocity - omega * v * dt) * exp;
            return newPos;
        }
    }
}
```

این پیاده‌سازی پایه حرکت سطح بالاست. با جایگزینی فراخوانی‌های `transform` با دستکاری مستقیم ماتریس یا ترکیب این در یک سیستم Job-based DOTS، می‌توانید این را به هزاران شیء با صفر افت کارایی مقیاس دهید.
