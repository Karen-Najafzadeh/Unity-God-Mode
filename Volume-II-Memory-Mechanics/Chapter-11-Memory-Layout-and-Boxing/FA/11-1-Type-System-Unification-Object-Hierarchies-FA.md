

### فصل ۱۱: چیدمان حافظه و بوکسینگ (Boxing)

#### موضوع ۱: یکپارچه‌سازی سیستم تایپ و سلسله‌مراتب اشیاء

به فصل ۱۱ خوش آمدید. در مسیر قبلی‌مان، یک خط معماری محکم در سرتاسر جهان ترسیم کردیم و داده‌های شما را به دو قلمرو مجزا تقسیم نمودیم: **تایپ‌های مقداری (Value Types)** (توکن‌های سریع و خود‌پاک‌کننده در حافظه Stack) و **تایپ‌های مرجع (Reference Types)** (کانتینرهای حمل بار سنگین و انعطاف‌پذیر در حافظه Heap).

اما این تقسیم‌بندی، یک بحران وجودی برای یک زبان برنامه‌نویسی مدرن و ظریف ایجاد می‌کند. اگر داده‌ها به دو رفتار مکانیکی کاملاً متفاوت تقسیم شده باشند، چگونه می‌توان یک تکه کد همه‌منظوره نوشت که بتواند هر چیزی را مدیریت کند؟ چگونه سیستم را به‌گونه‌ای طراحی می‌کنید که یک عدد صحیح ساده ۴ بایتی (int) و یک بافت رندرینگ (Texture) پیچیده ۵۰۰ مگابایتی توسط یک منطق سازمانی واحد مدیریت شوند؟

دانشمندان کامپیوتر برای حل این مسئله، یک معماری هوشمندانه و فریبنده به نام **یکپارچه‌سازی سیستم تایپ (Type System Unification)** طراحی کردند.

---

#### ۱. مفاهیم بنیادی علوم کامپیوتر: والد کیهانی (System.Object)

در دوران باستان و اولیه طراحی زبان‌های برنامه‌نویسی، زبان‌ها به‌شدت چندپاره بودند. در زبان‌هایی مانند ++C، تایپ‌های داده پایه‌ای مانند اعداد (int، float) عناصر خام و سریع سخت‌افزاری بودند، در حالی که اشیای تعریف‌شده توسط کاربر (class) ساختارهای پیچیده‌ای داشتند. آن‌ها به یک زبان مشترک صحبت نمی‌کردند. شما نمی‌توانستید به‌راحتی با یک عدد مانند یک شیء رفتار کنید، و همچنین نمی‌توانستید به‌سادگی سیستم‌های مجموعه فراگیر (مانند لیست‌ها یا صف‌ها) بسازید که بدون پناه بردن به اشاره‌گرهای حافظه ناامن و مدیریت‌نشده، هر دو تایپ را به‌طور ایمن بپذیرند.

هنگامی که معماران زبان‌های مدرن مبتنی بر ماشین مجازی (مانند جاوا و #C) در اواخر دهه ۱۹۹۰ برای ترسیم محیط‌های اجرای خود گردهم آمدند، خواستار دستیابی به **چندریختی همه‌جاخاضر (Omnipresent Polymorphism)** بودند؛ یعنی توانایی اینکه تک‌تک عناصر در سیستم تایپ، مجموعه‌ای از ویژگی‌های پایه را به اشتراک بگذارند.

آن‌ها می‌خواستند تضمین کنند که فارغ از اینکه یک متغیر، یک مقدار بولی (true/false) فوق‌العاده سبک باشد یا یک گره بزرگ از شبکه عصبی هوش مصنوعی، شما بتوانید دستورات پایه‌ای زیر را روی آن فراخوانی کنید:

* **`ToString()`**: تبدیل خود به یک رشته متنی قابل خواندن.
* **`GetType()`**: توصیف ساختار داخلی و قوانین حافظه خود در زمان اجرا (Runtime).
* **`Equals()`**: مقایسه داده‌های خود با داده‌های متغیری دیگر.

معماران برای دستیابی به این هدف، یک قانون معماری کیهانی وضع کردند: **تک‌تک تایپ‌ها در کل زبان برنامه‌نویسی باید از یک اجداد نهایی و واحد ارث‌بری کنند.** در #C، این ریشه کیهانی `System.Object` (یا نام مستعار کوتاه آن یعنی `object`) نام دارد.

```
                       [ System.Object ]
                              |
        +---------------------+---------------------+
        |                                           |
[ Reference Types ]                         [ System.ValueType ]
(Classes, Strings, Arrays)                          |
                                            [ Value Types ]
                                       (int, float, bool, Structs)

```

با مجبور کردن کل جهان داده‌ها به مشتق شدن از این ریشه واحد، زبان به یک هارمونی کامل دست می‌یابد. یک آرایه از `[]object` تئوریکاً می‌تواند هر چیزی را که در جهان هستی وجود دارد در خود جای دهد. اما همان‌طور که خواهیم دید، این توهم ساختاری زیبا هنگام برخورد با سخت‌افزار فیزیکی، با یک سیستم مالیاتی پنهان و ویران‌گر همراه می‌شود.

#### ۲. مسئله اصلی و راهکار یکپارچه

##### مسئله

تصور کنید در حال ساخت یک سیستم رابط کاربری (UI) اختصاصی برای بازی خود هستید. این سیستم دارای یک گزارش اعلان (Notification Log) بسیار انعطاف‌پذیر است که باید رویدادهای مختلفی را روی صفحه چاپ کند:

* "Player scored 100 points!" (یک تایپ مقداری پایه‌ای از نوع int)
* "System Check: True" (یک تایپ مقداری پایه‌ای از نوع bool)
* "Item Found: Excalibur" (یک تایپ مرجع از نوع string)

اگر زبان دارای یک سیستم تایپ چندپاره بود، شما مجبور بودید برای تک‌تک ترکیب‌های انواع داده، لیست‌های ردیابی، سیستم‌های مرتب‌سازی و اسکریپت‌های نمایش کاملاً متفاوتی بنویسید. شما به یک `IntNotificationLog`، یک `BoolNotificationLog` و یک `StringNotificationLog` نیاز پیدا می‌کردید. این امر کوهی از کدهای تکراری و غیرقابل‌پشتیبانی ایجاد می‌کرد.

##### راهکار یکپارچه

زبان #C با یکپارچه کردن سیستم تایپ تحت لوای `System.Object`، به شما اجازه می‌دهد تا یک مکانیسم واحد و ژنریک ایجاد کنید. از آنجا که یک `int` از نظر ساختاری با یک `object` یکپارچه شده است، شما می‌توانید یک `int` را مستقیماً به جایگاهی که یک `object` می‌طلبد، پاس بدهید.

پشت صحنه، کامپایلر به‌آرامی این تایپ‌ها را از طریق یک کلاس واسط پنهان به نام `System.ValueType` هدایت می‌کند. کلاس `System.ValueType` رفتار پیش‌فرض `System.Object` را بازنویسی (Override) می‌کند تا اطمینان حاصل کند که اگرچه استراکت‌ها (Structs) و اعداد صحیح بخشی از این درخت بزرگ و یکپارچه هستند، اما هنگام استفاده به‌صورت محلی، مکانیک سخت‌افزاری فوق‌بهینه‌شده خود (کپی از طریق مقدار یا copy-by-value) را حفظ کنند.

#### ۳. بررسی عمیق نمونه‌های مکانیکی

بیایید مستقیماً نگاهی بیندازیم به اینکه این طبقه‌بندی یکپارچه چگونه در حافظه RAM کامپیوتر شما رفتار می‌کند. فرض کنید یک مخزن ذخیره‌سازی انتزاعی برای نگهداری یک آیتم ایجاد کرده‌ایم و می‌خواهیم ببینیم کامپیوتر چگونه یک کلاس و یک عدد صحیح خام را در آن جای می‌دهد.

##### نمونه الف: ذخیره‌سازی یک تایپ مرجع (کلاس به شیء)

هنگامی که یک نمونه کلاس را به یک متغیر مرجعِ `object` اختصاص می‌دهید، ماشین مجازی بدون هیچ مشکلی آن را می‌پذیرد؛ این یک انتقال بی‌نقص است. شیءِ موجود در حافظه Managed Heap از قبل دارای یک هدر ردیابی شیء (Object Header) است. متغیر روی پشته (Stack) صرفاً برچسب تایپ خود را از نام کلاس اختصاصی به نوع اشاره‌گر ژنریک `object` تغییر می‌دهد. داده‌های واقعی هرگز جابه‌جا نمی‌شوند و هیچ بار اضافی برای حافظه ایجاد نمی‌شود.

```
[Stack Pointer: genericObject] ---> [Heap Data: Object Header + Class Fields]

```

##### نمونه ب: ذخیره‌سازی یک تایپ مقداری (داده پایه‌ای به شیء)

هنگامی که تلاش می‌کنید یک `int` خام ۴ بایتی (تایپ مقداری) را به یک متغیر مرجع `object` اختصاص دهید، یک برخورد معماری شدید رخ می‌دهد! یک متغیر `object` باید یک آدرس اشاره‌گر ۸ بایتی باشد که به یک موجودیت کاملاً شکل‌یافته و ردیابی‌شده در حافظه Managed Heap اشاره می‌کند. اما `int` خام شما صرفاً ۴ بایت داده مستقیم است که روی پشته (Stack) نشسته است؛ بدون هدر شیء، بدون توصیف‌کننده تایپ و بدون هیچ حضوری در حافظه Heap!

ماشین مجازی برای حفظ این توهمِ یکپارچه‌سازی، مجبور است یک تخصیص حافظه اضطراری در زمان اجرا انجام دهد. ماشین مجازی اجرای کد را متوقف کرده، به‌سرعت به سراغ حافظه Managed Heap می‌رود، یک پوسته ردیابی شیءِ (Object Tracking Shell) کاملاً جدید تخصیص می‌دهد، عدد صحیح ۴ بایتی خام شما را در مرکز آن پوسته کپی (Clone) می‌کند و آدرس آن را به متغیر اشاره‌گر `object` شما تحویل می‌دهد.

دانشمندان کامپیوتر این تغییر شکل دراماتیک را **بوکسینگ (Boxing یا بسته‌بندی)** می‌نامند؛ و این موضوع، پایه و اساس مبحث بعدی ماست.

#### ۴. پیاده‌سازی اسکریپت #C با دقت بالا

بیایید به یک سناریوی جامع در یونیتی نگاه کنیم که به‌وضوح ساختار دستوریِ یکپارچه‌سازی سیستم تایپ را نشان می‌دهد و نمایش می‌دهد که چگونه تایپ‌های مختلف می‌توانند به‌صورت پویا به ریشه کیهانی `System.Object` تبدیل (Cast) شوند.

```csharp
using UnityEngine;
using System;

public class TypeUnificationTester : MonoBehaviour
{
    // یک تایپ مقداری سفارشی که نمایانگر یک بردار مختصات دوبعدی سریع است
    public struct GameCoordinate
    {
        public int x;
        public int y;
        
        public GameCoordinate(int x, int y)
        {
            this.x = x;
            this.y = y;
        }
    }

    // یک تایپ مرجع سفارشی که نمایانگر یک زیرسیستم موتور است
    public class CombatSubsystem
    {
        public string systemName = "Acheron AI Core";
    }

    void Start()
    {
        ExecuteUnificationDemonstration();
    }

    private void ExecuteUnificationDemonstration()
    {
        Debug.Log("=== UNIFIED TYPE SYSTEM INSPECTION ===");

        // ۱. نمونه‌سازی از سه موجودیت معماری کاملاً متفاوت
        int rawScore = 4500;                             // تایپ مقداری پایه‌ای (مستقر در Stack)
        GameCoordinate rawPos = new GameCoordinate(10, 5); // استراکت سفارشی / تایپ مقداری (مستقر در Stack)
        CombatSubsystem aiCore = new CombatSubsystem();  // تایپ مرجع سفارشی (مستقر در Managed Heap)

        // ۲. به دلیل یکپارچه‌سازی سیستم تایپ، یک آرایه از System.Object می‌تواند همه آن‌ها را ببلعد!
        object[] cosmicUniversalArray = new object[3];

        // کامپایلر در اینجا به‌طور خودکار تبدیل یکپارچه را اعمال می‌کند
        cosmicUniversalArray[0] = rawScore; // یکپارچه‌سازی از طریق Boxing ضمنی!
        cosmicUniversalArray[1] = rawPos;   // یکپارچه‌سازی از طریق Boxing ضمنی!
        cosmicUniversalArray[2] = aiCore;   // یکپارچه‌سازی از طریق تبدیل مرجع استاندارد (بدون بوکسینگ)

        // ۳. بررسی ویژگی‌های یکپارچه ارث‌بری‌شده از System.Object
        for (int i = 0; i < cosmicUniversalArray.Length; i++)
        {
            object activeElement = cosmicUniversalArray[i];

            // هر آیتمی در زبان به دلیل داشتن والد کیهانی، به ()ToString و ()GetType پاسخ می‌دهد
            string elementStringRepresentation = activeElement.ToString();
            Type underlyingRuntimeType = activeElement.GetType();

            Debug.Log($"Array Slot [{i}] | Derived String: '{elementStringRepresentation}' | True Hardware Type: {underlyingRuntimeType.Name}");
            
            // بررسی اینکه آیا معماری تایپ ادعای اصالت از System.ValueType دارد یا خیر
            if (underlyingRuntimeType.IsValueType)
            {
                Debug.Log($"--> Slot [{i}] is a VALUE TYPE that has been unified via Boxing.");
            }
            else
            {
                Debug.Log($"--> Slot [{i}] is a NATIVE REFERENCE TYPE. No structural alteration needed.");
            }
        }
    }
}

```

#### ۵. ماتریس طرح حافظه معماری

برای هدایت تصمیمات طراحی خود هنگام گروه‌بندی داده‌ها در کامپوننت‌های بازی، این نقشه طبقه‌بندی سیستمی را به خاطر بسپارید:

| شاخص تشخیصی | داده‌های پایه‌ای (int, bool) | استراکت‌های سفارشی موتور (Vector3, struct) | کلاس‌های مدیریت‌شده (class, MonoBehaviour) |
| --- | --- | --- | --- |
| **ریشه اصالت کیهانی** | System.ValueType $\rightarrow$ System.Object | System.ValueType $\rightarrow$ System.Object | تبار مستقیم به System.Object |
| **تخصیص متغیر استاندارد** | داده خالص به‌صورت درون‌خطی (Inline) روی Stack. | چیدمان بلوک داده فشرده به‌صورت درون‌خطی روی Stack. | اشاره‌گر ۸ بایتی روی Stack، بار اصلی (Payload) روی Heap. |
| **رفتار تحت تبدیل یکپارچه (Universal Casting)** | تغییریافته. به اجبار از طریق تخصیص Boxing به حافظه Heap منتقل می‌شود. | تغییریافته. کل چیدمان ساختاری به حافظه Heap باکس (بسته‌بندی) می‌شود. | بدون تغییر. تبدیل اشاره‌گر بومی؛ هزینه تخصیص حافظه صفر بایت است. |
| **قابلیت چندریختی (Polymorphic)** | فقط اینترفیس‌های استاتیک. بدون ارث‌بری در پیاده‌سازی. | فقط اینترفیس‌های استاتیک. بدون ارث‌بری در پیاده‌سازی. | ارث‌بری پویا، اوررایدها (Overrides) و چندریختی کامل. |
