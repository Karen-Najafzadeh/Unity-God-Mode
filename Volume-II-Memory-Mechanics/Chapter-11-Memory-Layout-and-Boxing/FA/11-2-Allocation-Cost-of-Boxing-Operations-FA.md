
# هزینه تخصیص حافظه در عملیات Boxing (تخصیص جعبه‌ای)

اکنون که از سفرهای قبلی خود درک محکمی از استک (Stack - میز کار محلی، فوق‌سریع و آنی شما) و هیپ (Heap - انبار دوردست و پیچیده حافظه) به دست آورده‌اید، می‌توانیم پرده را از روی یکی از فریبنده‌ترین قاتلان عملکرد (Performance) در مهندسی بازی‌های یونیتی برداریم: **تخصیص‌های ناشی از Boxing**.

---

## افسانه‌شناسی علوم کامپیوتر: یکپارچه‌سازی سیستم تایپ و لفاف همه‌منظوره (Universal Wrapper)

در تاریخ اولیه زبان‌های برنامه‌نویسی، مهندسان با یک دوراهی بزرگ مواجه بودند. اگر آن‌ها تایپ‌های داده‌ای کوچک (مانند اعداد و مقادیر بولین) را کاملاً از شیءهای پیچیده (مانند ساختارهای داده و کلاس‌های سفارشی بازی) جدا نگه می‌داشتند، مجبور بودند برای مدیریت هر کدام، نسخه‌های تکراری از کد را بنویسند. برای حل این مشکل، سی‌شارپ مفهوم **یکپارچه‌سازی سیستم تایپ (System Type Unification)** را معرفی کرد.

معماران این زبان تصمیم گرفتند که *همه‌چیز*—از یک بیت خام تک گرفته تا یک موجودیت هوش مصنوعی غول‌آسا برای باسِ بازی (Boss AI)—از یک کلاس جد بزرگ واحد ارث‌بری کند: `System.Object`.

این نوآوری به توسعه‌دهندگان اجازه داد تا سیستم‌های بسیار تعمیم‌یافته و زیبایی بنویسند که می‌توانستند یک `object` را به عنوان پارامتر بپذیرند و با مطلقاً هر چیزی کار کنند. با این حال، این زیبایی با یک معاوضه مهندسی تاریک همراه بود. سخت‌افزار نمی‌تواند به طور بومی با یک مقدار اولیه خام که روی استک تخصیص یافته است، مانند یک شیء کلاس تخصیص‌یافته روی هیپ رفتار کند. برای حفظ این توهمِ هماهنگی و یکپارچگی مطلق، محیط زمان اجرا (Runtime) باید به شکلی خشن نحوه وجود یک تایپ مقداری (Value Type) را در حافظه تغییر دهد.

---

## مسئله اصلی: جریمه مالی جعبه مقوایی

همان‌طور که می‌دانید، یک تایپ مقداری (مانند یک `int` یا یک `struct` سفارشی) صرفاً داده‌ای خام است که به طور فشرده روی استک شما قرار گرفته است. این داده هیچ بار اضافی یا فیلد پنهانی ندارد؛ بلکه دقیقاً نشان‌دهنده همان مقدار خود است.

هنگامی که آن مقدار خام را به سیستمی پاس می‌دهید که انتظار یک `object` را دارد، عملیات **Boxing (جعبه‌گیری)** رخ می‌دهد. بیایید دقیقاً بررسی کنیم که در آن میکروثانیه حساس چه اتفاقی برای سخت‌افزار می‌افتد:

1. **تفحص هیپ و تخصیص حافظه (Heap Speculation & Memory Allocation):** پردازنده (CPU) کاری را که در حال انجام آن است متوقف کرده و از تخصیص‌دهنده هیپ مدیریت‌شده (Managed Heap Allocator) یک بلوک حافظه درخواست می‌کند. این بلوک باید به اندازه کافی بزرگ باشد تا داده‌های خام شما را به همراه دو فیلد بالاسری پنهان و اجباری در خود جای دهد: **سرایند شیء یا Object Header** (مورد استفاده برای بلوک‌های همگام‌سازی) و **اشاره‌گر جدول متدهای تایپ یا Type Method Table Pointer** (که به موتور بازی می‌گوید این چه نوع داده‌ای است).
2. **کپی حافظه (The Memory Copy):** پردازنده به طور فیزیکی مقدار خام را از روی استک می‌خواند، به آدرس تازه تخصیص‌یافته روی هیپ می‌رود و بیت‌ها را درون آن کادر یا جعبه کپی می‌کند.
3. **تولید ارجاع (Reference Generation):** پردازنده آدرس مکان شروع حافظه آن جعبه روی هیپ را می‌گیرد و آن را به عنوان یک اشاره‌گر (Pointer) روی استک شما بازنویسی می‌کند.

### کمرشکن پنهان: انباشت زباله نسلی (Generational Garbage Accumulation)

اگر این کار را یک بار انجام دهید، کامپیوتر شما حتی پلک هم نخواهد زد. اما بازی‌ها در یک حلقه با فرکانس بالا (اغلب ۶۰ تا ۱۲۰+ فریم در ثانیه) اجرا می‌شوند. اگر یک اسکریپت در داخل حلقه `Update()` شما حتی باعث یک عملیات Boxing شود، شما در هر فریم یک جعبه مقوایی کاملاً جدید و دورریختنی را به درون هیپ پرتاب می‌کنید.

از آنجا که این جعبه‌ها بلافاصله پس از پایان فریم، ارجاع خود را از دست می‌دهند، فوراً به **زباله (Garbage)** تبدیل می‌شوند. این امر زباله‌روب یونیتی (Garbage Collector یا به اختصار GC) را مجبور می‌کند تا به طور مکرر فعال شود. زباله‌روب برای اسکن هیپ، محاسبه اینکه کدام جعبه‌ها رها شده‌اند و بازپس‌گیری فضای حافظه آن‌ها، باید رشته اصلی اجرای بازی شما را متوقف کند (پدیده "Stop-The-World"). این اتفاق مستقیماً باعث ایجاد ریزتِپق‌ها (Micro-stutters) و افت فریم‌های ترسناکی می‌شود که تجربه بازیکن را در طول یک گیم‌پلی روان خراب می‌کند.

## نمونه‌های جامع و واقعی در یونیتی

بیایید سناریوهای متمایز و نوآورانه‌ای را بررسی کنیم که در آن‌ها Boxing به آرامی به کد بازی نفوذ می‌کند، نحوه تحلیل هزینه آن را یاد بگیریم و بدانیم چگونه آن‌ها را برای رسیدن به عملکردی با تخصیص صفر (Zero-Allocation) بازنویسی کنیم.

### نمونه ۱: تله ثبت‌کننده رویدادهای واسط کاربری (فرمت‌دهی رشته‌ای چندریختی)

یک گزارش مبارزه (Combat Log) درون بازی را تصور کنید که رویدادهای ترکیبی را نشان می‌دهد (به عنوان مثال: صدمه زدن بازیکن، دریافت امتیاز تجربه یا XP، یا برداشتن آیتم‌ها). یک رویکرد رایج، استفاده از یک متد پیام‌رسانی یکپارچه است که شیءهای عمومی (Generic Objects) را می‌پذیرد.

#### ❌ رویکرد سنگین از نظر تخصیص حافظه (باعث ایجاد Boxing می‌شود)

```csharp
using UnityEngine;
using UnityEngine.UI;

public class CombatLogSystem : MonoBehaviour
{
    public Text logText;

    public void LogCombatEvent(string eventDescription, object eventValue)
    {
        // پارامتر 'eventValue' به عنوان یک object مشخص شده است تا انواع int، float یا string را بپذیرد.
        // اگر یک int (مانند مقدارِ damage) پاس داده شود، دقیقاً در همین‌جا مجبور به تبدیل شدن به Boxing می‌شود.
        logText.text += $"\n{eventDescription}: {eventValue}";
    }

    void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
        {
            int damageDealt = 125; // یک تایپ مقداری روی استک
            
            // عملیات Boxing به طور خودکار رخ می‌دهد؛ چرا که مقدار damageDealt (از نوع int) 
            // به پارامتر 'object eventValue' کست (Cast) می‌شود.
            LogCombatEvent("Player swung sword", damageDealt); 
        }
    }
}

```

#### معماری با تخصیص صفر (بدون Boxing)

برای حل این مشکل بدون نیاز به نوشتن متدهای جداگانه برای هر تایپ اولیه، می‌توانیم از **جایگاه‌های عمومی (Generics)** در سی‌شارپ همراه با تبدیل‌های ساختاری یا اورلودهای (Overloads) ساده و ایمن از نظر تایپ استفاده کنیم.

```csharp
using UnityEngine;
using UnityEngine.UI;

public class OptimizedCombatLogSystem : MonoBehaviour
{
    public Text logText;

    // با پیاده‌سازی یک پارامتر جنریک <T>، کامپایلر در زمان اجرا یک نسخه سفارشی و
    // ایمن از نظر تایپ از این متد را به طور ویژه برای تایپ‌های مقداری تولید می‌کند.
    public void LogCombatEventOptimized<T>(string eventDescription, T eventValue)
    {
        // اگر T از نوع int باشد، به عنوان یک int روی استک باقی می‌ماند و هیچ Boxing رخ نمی‌دهد.
        // ما به طور صریح از .ToString() استفاده می‌کنیم تا به خودِ تایپ مقداری اجازه دهیم داده متنی‌اش را تبدیل کند.
        logText.text += $"\n{eventDescription}: {eventValue.ToString()}";
    }

    void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
        {
            int damageDealt = 125;
            
            // کامپایلر یک مسیر بهینه‌سازی‌شده برای <int> ایجاد می‌کند. زباله تولیدی روی هیپ: صفر!
            LogCombatEventOptimized<int>("Player swung sword", damageDealt);
        }
    }
}

```

### نمونه ۲: مطابقت‌دهی وضعیت ساختاری غیرجنریک (ماتریس دستاوردها)

یک سیستم ردیاب دستاوردها (Achievement Tracker) را در نظر بگیرید که بررسی می‌کند آیا آمارهای مشخصی به اهداف مورد نیاز رسیده‌اند یا خیر. توسعه‌دهندگان اغلب به دنبال انعطاف‌پذیری ساختاری هستند، اما استفاده از الگوهای قدیمی کلکیسیون‌ها در سی‌شارپ یا ساختارهای غیرجنریک، نرخ فریم بازی را نابود می‌کند.

#### ❌ رویکرد سنگین از نظر تخصیص حافظه (باعث ایجاد Boxing می‌شود)

```csharp
using UnityEngine;
using System.Collections; // مورد نیاز برای ساختارهای غیرجنریک قدیمی

public class AchievementTracker : MonoBehaviour
{
    // مجموعه Hashtable کلیدها و مقادیر را به عنوان ارجاع‌های خام 'object' ذخیره می‌کند.
    private Hashtable playerStats = new Hashtable(); 

    void Start()
    {
        // هم کلید (int) و هم مقدار (float) درون شیءها Box می‌شوند!
        playerStats.Add(101, 45.5f); 
    }

    void Update()
    {
        // در هر فریم که این را می‌خوانیم یا به‌روزرسانی می‌کنیم، عملیات Unboxing و Boxing به طور مکرر رخ می‌دهند.
        if (playerStats.ContainsKey(101))
        {
            float currentProgress = (float)playerStats[101]; // جریمه ناشی از Unboxing
            if (currentProgress < 100.0f)
            {
                playerStats[101] = currentProgress + Time.deltaTime; // تخصیص مجدد ناشی از Re-boxing!
            }
        }
    }
}

```

#### معماری با تخصیص صفر (بدون Boxing)

با جایگزین کردن آن کلکسیون قدیمی با یک مجموعه جنریک، مدرن و با تایپ قوی (Strongly-typed)، بهینه‌سازی چیدمان حافظه به شکلی بی‌نقص حاصل می‌شود.

```csharp
using UnityEngine;
using System.Collections.Generic; // استفاده از مجموعه‌های جنریک مدرن را ممکن می‌سازد

public class OptimizedAchievementTracker : MonoBehaviour
{
    // دیکشنری به طور صریح هم کلیدها و هم مقادیر را به تایپ‌های قطعی با اندازه مشخص روی استک متصل می‌کند.
    private Dictionary<int, float> optimizedPlayerStats = new Dictionary<int, float>();

    void Start()
    {
        // داده‌ها مستقیماً وارد یک چیدمان حافظه پیوسته و بدون Box (Unboxed) می‌شوند.
        optimizedPlayerStats.Add(101, 45.5f);
    }

    void Update()
    {
        // عملیات کاملاً مقداری در تمام سطوح. کاملاً پاکیزه و بهینه.
        if (optimizedPlayerStats.TryGetValue(101, out float currentProgress))
        {
            if (currentProgress < 100.0f)
            {
                // بازنویسی یک تایپ مقداری در داخل یک دیکشنری جنریک، داده‌ها را 
                // مستقیماً در محل خود روی چیدمان آرایه استک تغییر می‌دهد. تخصیص هیپ: صفر.
                optimizedPlayerStats[101] = currentProgress + Time.deltaTime;
            }
        }
    }
}

```

## نگاهی سریع به پروفایل‌های حافظه

برای تجسم تاثیر معماری این عادات بر روی چیدمان فیزیکی سخت‌افزار پلتفرم شما:

```
[تخصیص BOXING به جای تایپ مقداری]

STACK (استک)                             HEAP (هیپ مدیریت‌شده)
+------------------------+           +----------------------------------------+
| Pointer to Box -------+----------> | [Object Header] (32/64-bit Sync Block) |
+------------------------+           | [Type Method Table Pointer]            |
| Primitive Raw Value    |           | [Actual Payload Data] (e.g., int 125)  |
+------------------------+           +----------------------------------------+
                                       ^--- پس از اتمام کار، به عنوان زباله باقی می‌ماند!

```

با اعمال محدودیت‌های سخت‌گیرانه برای ایمنی تایپ (Type-safety)، بهره‌گیری از **Generics**، و رفتار با تایپ‌های مقداری به عنوان متغیرهای خالص استک، شما حافظه هیپ را به طور کامل دور می‌زنید؛ امری که زباله‌روب یونیتی را در وضعیت خواب نگه می‌دارد و تضمین می‌کند که بازی شما به شکلی روان و بدون ریزتپق اجرا شود.
