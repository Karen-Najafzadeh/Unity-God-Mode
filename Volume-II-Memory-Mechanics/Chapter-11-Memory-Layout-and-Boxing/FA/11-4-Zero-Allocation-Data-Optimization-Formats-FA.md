

# قالب‌های بهینه‌سازی داده‌ها با تخصیص صفر (Zero-Allocation Data Optimization Formats)

ما به طور گسترده آناتومی این دام تله‌گونه را بررسی کرده‌ایم: اینکه چگونه باکسینگ داده‌های بی‌گناه را مجبور به قرارگیری در کانتینرهای گران‌قیمت مقوایی بر روی حافظه هیپ می‌کند، و چگونه آن‌باکسینگ پردازنده (CPU) را ناچار می‌سازد تا در یک ایستگاه بازرسی گمرکی توقف کرده و مانیفست‌های ساختاری را دو بار بررسی کند. اکنون، ما به فاز ترمیم و اصلاح فعال وارد می‌شویم.

برای دستیابی به بهینه‌سازی در سطح «مد خدایگان» (God Mode) در یونیتی، شما نمی‌توانید صرفاً به یک گلوگاه عملکردی نگاه کرده و انعطاف‌پذیری سیستم خود را قربانی کنید. شما باید بر **قالب‌های بهینه‌سازی داده‌ها با تخصیص صفر (Zero-Allocation Data Optimization Formats)** مسلط شوید. این‌ها پارادایم‌های معماری و الگوهای زبان مدرن هستند که به منظور حفظ جریان روان داده‌های شما بر روی میز کار فوق‌سریع استک (Stack) طراحی شده‌اند و تضمین می‌کنند که نمودار تخصیص زمان اجرای شما در یک خط مبنای کاملاً صاف و بدون زباله باقی بماند.

## ۱. تاریخچه علوم کامپیوتر: جداسازی بزرگ تایپ و چیدمان حافظه

در معماری کلاسیک علوم کامپیوتر، اگر مهندسی به دنبال حداکثر کارایی و عملکرد سخت‌افزاری بود، باید از زبان‌های سطح پایین مانند C یا اسمبلی استفاده می‌کرد و آفست‌های بایت خام را به صورت دستی مستقیماً در حافظه آرایش می‌داد. زمانی که زمان‌های اجرای مدیریت‌شده مدرن مانند سی‌شارپ (محیط .NET) از راه رسیدند، وعده انتزاع امن (Safe Abstraction) را از طریق موتورهای چیدمان خودکار حافظه به همراه آوردند. همان‌طور که کشف کردیم، وجه مصالحه این بود که زمان اجرا برای دستیابی به رفتارهای چندریختی و عمومی، ترجیح می‌داد از اشیاء یکپارچه و سراسری استفاده کند.

پیشرفتی که برنامه‌نویسی مدیریت‌شده با کارایی بالا را نجات داد، مفهوم **جداسازی چیدمان داده‌ها از طریق جنریک‌های کامپایلری (Data-Layout Decoupling via Compilational Generics)** بود.

به جای رفتار با یک توکن جنریک (مانند `T`) به عنوان یک راهنمای زمان اجرا که داده‌ها را مجبور می‌کند به شکل پایه و اولیه `object` سقوط کنند، کامپایلرهای مدرن عملیاتی را انجام می‌دهند که به نام **مُنومورفیزاسیون (Monomorphization)** شناخته می‌شود. وقتی کامپایلر با یک قالب جنریکِ محدودشده به تایپ برخورد می‌کند، چیدمان کد ماشین زیرین را دقیقاً برای همان پهنای ساختاری کلون (شبیه‌سازی و تکثیر) می‌کند. اگر یک عدد صحیح به آن بدهید، یک اسلات ۳۲ بیتی با نقشه‌برداری دقیق باقی می‌گذارد؛ اگر یک استраکت فیزیکی سفارشی به آن بدهید، آن اسلات را به صورت درون‌خطی (Inline) گسترش می‌دهد. معماران کامپیوتر با تغییر نحو ساختاری در زمان کامپایل، محدودیت و مصالحه ناشی از `System.Object` سراسری را به طور کامل دور زدند و به مهندسان سیستم آزادی ساختاری دادند، بدون اینکه سیلیکون پردازنده را مجبور کنند حتی یک بایت زباله روی حافظه تخصیص دهد.

## ۲. مسئله اصلی: طراحی برای ساختار تخصیص صفر

هنگام ساخت بازی‌های ویدیویی پیچیده و بی‌درنگ (Real-Time)، قالب‌های داده باید سه نیاز متضاد و چالش‌برانگیز را برآورده کنند:

1. **توسعه‌پذیری پویا (Dynamic Extensibility):** توانایی مدیریت ساختارهای داده‌ای متغیر (به عنوان مثال، پارس کردن آمار بازیکن، تغییرات موجودی انبار یا به‌روزرسانی‌های شبکه).
2. **پیمایش چندریختی (Polymorphic Traversal):** اجازه دادن به حلقه‌های اجرایی مختلف برای پارس، قالب‌بندی و تعامل با این تایپ‌های سفارشی بدون نیاز به هزاران کپی متد دستی و هاردکد شده.
3. **عدم تغییرپذیری هیپ (Zero Heap Mutability):** دستیابی به دو نیاز اول بدون راه‌اندازی تخصیص‌های جدید (`new`) بر روی حافظه هیپ یا آغاز عملیات باکسینگ.

برای حل این مسئله، مکانیک‌های سطح پایین بازی از قالب‌های پیشرفته ایمنی تایپ بهره می‌برند. ما از **محدودیت‌های تایپ اینترفیس (Interface Type Constraints)**، **شمارنده‌های استراکت (Struct Enumerators)** و **بافرهای نقشه‌برداری‌شده با مقدار (Value-Mapped Buffers)** استفاده می‌کنیم تا کامپایلر را مجبور کنیم ابعاد تایپ را پیش از شروع اجرای بازی به صورت ایستا (Static) حل و فصل کند.

```
[ THE MONOMORPHIZED STRUCT PIPELINE / خط لوله استراکت منومورفیزه‌شده ]

کد برنامه:
  ProcessData<HealthStruct>(myHealth);
  ProcessData<ManaStruct>(myMana);

تولید کامپایلر (تفکیک دقیق در سطح کد ماشین):
  +-----------------------------------+-----------------------------------+
  |   Generated Native Method A       |   Generated Native Method B       |
  |   (Hardcoded 32-bit Stack Slot)   |   (Hardcoded 64-bit Stack Slot)   |
  |   [Reads Health Struct Directly]  |   [Reads Mana Struct Directly]    |
  +-----------------------------------+-----------------------------------+
  ^--- درخواست هیپ: صفر مطلق! تعقیب پوینتر: صفر! حلقه‌های اعتبارسنجی: صفر!

```

## ۳. قالب‌های پیشرفته کد در یونیتی: ارتقای سیستم‌ها به تخصیص صفر

بیایید دو قالب معماری بسیار نوآورانه را کالبدشکافی کنیم که الگوهای رایج اشیاء چندریختی را با پارادایم‌های کاملاً بدون تخصیص و در سطح پرفورمنس بالا جایگزین می‌کنند.

### قالب ۱: مدیریت‌کننده رویداد مُنومورفیزه‌شده (جنریک‌های محدودشده)

یک سیستم تلمتری (سنجش از راه دور) سراسری را تصور کنید که برای هدایت ویژگی‌های رزمی با فرکانس بالا (ضرایب ضربه بحرانی، داده‌های موقعیتی، نرخ سقوط آیتم‌ها) به زیرماژول‌های مختلف زمان اجرا استفاده می‌شود. به جای استفاده از مرزهای شیء خام، ما از استراکت‌های جنریک استفاده می‌کنیم که به طور سفت و سخت به اینترفیس‌های ساختاری استاندارد متصل شده‌اند.

#### ❌ قالب تخصیص متعارف (باعث راه‌اندازی باکسینگ و تخصیص حافظه می‌شود)

```csharp
using UnityEngine;

public interface ITelemetryData { string GetHeader(); }

public struct CombatDamageData : ITelemetryData
{
    public int damageAmount;
    public string GetHeader() => "COMBAT_DMG";
}

public class TelemetryPipeline : MonoBehaviour
{
    // تله اصلی: پاس دادن یک اینترفیس به طور مستقیم به عنوان پارامتر تابع،
    // با استراکت به عنوان یک مرز مرجع رفتار می‌کند و باعث باکسینگ پنهان می‌شود!
    public void BroadcastTelemetry(ITelemetryData data)
    {
        // باکسینگ در اینجا رخ می‌دهد زیرا 'data' به عنوان یک ارجاع اینترفیس ضمنی پاس داده شده است
        Debug.Log($"[{data.GetHeader()}] Processing item on heap.");
    }

    void Update()
    {
        CombatDamageData liveStrike = new CombatDamageData { damageAmount = 950 };
        
        // هزینه تخصیص پنهان: متغیر liveStrike به لایه تایپ اینترفیس باکس می‌شود
        BroadcastTelemetry(liveStrike); 
    }
}

```

#### قالب تخصیص صفر (ساختار محدودشده به تایپ)

با بازسازی پروفایل اجرایی برای پذیرش یک توکن ساختاری `T` که توسط یک محدودیت تایپ زمان کامپایل (`where T : struct, ITelemetryData`) قفل شده است، به یونیتی دستور داده می‌شود تا متدهای بومی ایستای متمایزی تولید کند که تبدیل مرجع اینترفیس را به طور کامل دور می‌زنند.

```csharp
using UnityEngine;

public interface IOptimizedTelemetry { string GetHeader(); }

public struct OptimizedDamageData : IOptimizedTelemetry
{
    public int damageAmount;
    public string GetHeader() => "OPTIMIZED_DMG";
}

public class ZeroAllocationPipeline : MonoBehaviour
{
    // با اعمال محدودیت 'where T : struct'، کامپایلر مسیرهای اسمبلی اختصاصی و
    // نوع مقداری را تولید می‌کند که مستقیماً بر روی چیدمان حافظه استک نقشه‌برداری می‌شوند.
    public void BroadcastTelemetryOptimized<T>(T data) where T : struct, IOptimizedTelemetry
    {
        // ساختار، متدهای خود را به صورت درون‌خطی و بدون بسته‌بندی در کانتینر اینترفیس اجرا می‌کند
        Debug.Log($"[{data.GetHeader()}] Processing zero-allocation structural telemetry data.");
    }

    void Update()
    {
        OptimizedDamageData liveStrike = new OptimizedDamageData { damageAmount = 950 };
        
        // فراخوانی مقدار خالص. حل ساختاری ایستا، زباله صفر و آن‌باکسینگ صفر را تضمین می‌کند.
        BroadcastTelemetryOptimized(liveStrike);
    }
}

```

### قالب ۲: بافر حافظه با آدرس‌دهی مستقیم (آرایه‌های واریانت استراکت)

یک موتور مسیریابی سفارشی یا یک دفتر کل موجودی انبار را در نظر بگیرید که مجموعه‌ای متغیر از ویژگی‌های آیتم را به‌روزرسانی می‌کند. به جای تخصیص ماتریس‌های شیء منحصربه‌فرد یا تکیه بر تغییر اندازه آرایه‌های جنریک، ما یک استراکت واریانت با چیدمان صریح تعریف می‌کنیم که چندین نوع داده را به صورت بومی در داخل یک بلوک چیدمان پیش‌بینی‌پذیر نشان می‌دهد.

#### ❌ قالب تخصیص متعارف (آرایه‌های کلاس چندریختی)

```csharp
using UnityEngine;
using System.Collections;

public class InventoryLedger : MonoBehaviour
{
    // آرایه‌ای از اشیاء برای قرار دادن رشته‌ها، اعداد صحیح و مقادیر سفارشی.
    // هر عدد صحیح اولیه وارد شده، به فضای کاری حافظه هیپ مدیریت‌شده کست می‌شود.
    private object[] ledgerCache = new object[3];

    void Update()
    {
        // راه‌اندازی تخصیص‌های باکسینگ مداوم در داخل به‌روزرسانی‌های با فرکانس بالا
        ledgerCache[0] = 50;              // نوع int باکس شد
        ledgerCache[1] = 1500.75f;        // نوع float باکس شد
        ledgerCache[2] = true;            // نوع bool باکس شد
    }
}

```

#### قالب تخصیص صفر (ساختارهای هم‌پوشان با چیدمان صریح)

برای حفظ کامل ذخیره‌سازی چندتایپی بدون تخصیص فضای هیپ یا باکسینگ، می‌توانیم از چیدمان‌های حافظه سطح پایین سی‌شارپ (`System.Runtime.InteropServices`) بهره ببریم. با مجبور کردن ویژگی‌ها به هم‌پوشانی فیزیکی در همان مکان‌های بایت دقیق در حافظه، ما یک ساختار متغیر و گران‌قیمتِ مستقر در استک ایجاد می‌کنیم که به عنوان یک **یونیون (Union)** شناخته می‌شود.

```csharp
using UnityEngine;
using System.Runtime.InteropServices; // کنترل‌های چیدمان حافظه سطح پایین را فعال می‌کند

// ما یک enum برای رهگیری اینکه کدام بار داده در سلول ما فعال است اعلام می‌کنیم
public enum VariantType : byte { Empty, Integer, Float, Boolean }

// چیدمان صریح به کامپیوتر دستور می‌دهد: شما آفست‌های بایت دقیق را کنترل می‌کنید!
[StructLayout(LayoutKind.Explicit)]
public struct ZeroAllocVariant
{
    [FieldOffset(0)] public VariantType typeIndicator; // بایت شماره 0 را اشغال می‌کند

    // تمام متغیرهای داده در آفست بایت 4 هم‌پوشانی دارند که با فاصله‌گذاری حافظه مطابقت دارد!
    [FieldOffset(4)] public int integerPayload;
    [FieldOffset(4)] public float floatPayload;
    [FieldOffset(4)] public bool booleanPayload;
}

public class OptimizedInventoryLedger : MonoBehaviour
{
    // یک آرایه نوع مقداری پیوسته که مستقیماً در فضای کاری استک قفل شده است.
    // پوینترهای مرجع صفر، جمع‌آوری‌کننده‌های زباله صفر، نوشتن فوری داده‌ها.
    private ZeroAllocVariant[] optimizedLedger = new ZeroAllocVariant[3];

    void Update()
    {
        // نوشتن بار داده صحیح (Integer) به طور مستقیم در اسلات ساختاری 0
        optimizedLedger[0].typeIndicator = VariantType.Integer;
        optimizedLedger[0].integerPayload = 50; // کپی مستقیم مقدار!

        // نوشتن بار داده اعشاری (Float) به طور مستقیم در اسلات ساختاری 1
        optimizedLedger[1].typeIndicator = VariantType.Float;
        optimizedLedger[1].floatPayload = 1500.75f; // کپی مستقیم مقدار!

        // نوشتن بار داده بولین (Boolean) به طور مستقیم در اسلات ساختاری 2
        optimizedLedger[2].typeIndicator = VariantType.Boolean;
        optimizedLedger[2].booleanPayload = true; // کپی مستقیم مقدار!
        
        // خواندن مجدد مقادیر کاملاً آنی و بدون بررسی‌های تایپ آن‌باکسینگ است:
        if (optimizedLedger[0].typeIndicator == VariantType.Integer)
        {
            int scoreValue = optimizedLedger[0].integerPayload; // دسترسی تمیز و فوری
        }
    }
}

```

## ۴. مقایسه ساختاری: استراتژی‌های بهینه‌سازی ساختاری

حفظ یک پایگاه کد با تخصیص صفر مستلزم هماهنگی دقیق قالب‌های داده با قابلیت‌های سخت‌افزار مکانیکی است:

| استراتژی قالب | طرح اولیه پیاده‌سازی | ردپای حافظه | تاثیر اجرای پردازنده (CPU) |
| --- | --- | --- | --- |
| **جنریک‌های محدودشده** | استفاده از مرزهای `where T : struct` در کنار اینترفیس‌های کامپوننت ساختاری. | 📦 **صفر مطلق** (به طور کامل در فریم‌های استک فعال ذخیره می‌شود). | ⚡ **فوری** (بهینه‌سازی کامپایلر کد اسمبلی بومی بسیار هدفمندی را خروجی می‌دهد). |
| **یونیون‌های استراکت صریح** | به‌کارگیری `LayoutKind.Explicit` برای هم‌پوشانی متغیرهای اولیه در یک فضای کش مشترک. | 📦 **صفر مطلق** (پهنای ساختار ثابت به آرایه‌های بومی نقشه‌برداری می‌شود). | ⚡ **فوری** (اعتبارسنجی را دور می‌زند؛ فیلدها فوراً از طریق آفست‌های بایت هاردکد شده خوانده می‌شوند). |
| **اشیاء چندریختی** | پاس دادن مقادیر به پارامترهای گسترده `System.Object` یا اینترفیس‌های استاندارد. | ⚠️ **تولید زباله بالا** (تخصیص‌های نسلی مداوم روی حافظه هیپ مدیریت‌شده). | 🐢 **گلوگاه شدید** (اجبار به تعقیب مداوم پوینترها، کش‌میس‌ها و اعتبارسنجی‌های تایپ). |

با اتخاذ ساختارهای جنریک منُمورفیزه‌شده و چیدمان‌های صریح نقشه‌برداری‌شده با حافظه، شما یونیتی را به طور کامل از توانایی تولید مخفیانه زباله روی هیپ محروم می‌کنید. این کار حلقه‌های اجرایی شما را کاملاً سبک نگه می‌دارد، تضمین می‌کند که نرخ فریم بازی شما ثابت بماند و قدرت پردازش کامپیوتر را برای رسیدگی به محاسبات واقعی گیم‌پلی آزاد می‌گذارد.
