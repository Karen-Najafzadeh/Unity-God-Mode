## فصل ۱۱۲: مدیریت خودکار حافظه و الگوی مخزن اشیاء (Object Pooling)

به محیط کارگاهی موتور بازی‌سازی خوش آمدید!بیایید بار دیگر از زاویه خلاقانه و ملموس به موضوع نگاه کنیم. در خلاصه‌های قبلی، مفهوم کلی زباله‌روب (Garbage Collector یا به اختصار GC) را به عنوان یک سرایدار تنبل بررسی کردیم که گاهی برای جارو کردن آشغال‌ها فریاد می‌زند «دنیا متوقف شود!» (Stop The World). همچنین الگوی مخزن اشیاء (Object Pooling) را مانند یک کشوی بازیافت قطعات دانستیم که برای تمیز نگه داشتن محیط کار به کمک ما می‌آید.

اکنون قصد داریم وارد دفتر برنامه‌ریزی و اتاق کنترل این سرایدار شویم. ما دقیقاً بررسی خواهیم کرد که این فرآیند پاک‌سازی چگونه در قالب ۵ مکانیسم کلیدی عمل می‌کند، سیستم چگونه با انواع مختلف زباله‌ها برخورد می‌نماید و چگونه می‌توانیم یک خط لوله بازیافت فوق‌پیشرفته بسازیم تا هرج‌ومرج ناشی از انباشت داده‌ها را کاملاً دور بزنیم.

### ۱. معماری زباله‌روب (Garbage Collector Architecture)

#### 📜 [افسانه‌ها و تاریخچه علوم کامپیوتر: ظهور سرایدار خودکار](./12-1-Garbage-Collector-Architecture-FA.md)

در دوران باستانِ زبان‌های برنامه‌نویسی مانند سی‌بلس‌بلس (++C)، توسعه‌دهندگان قدرت مطلق و بدون محدودیتی بر کف انبار کامپیوتر (حافظه هیپ یا Heap) داشتند. اگر می‌خواستید یک ساختار داده ایجاد کنید، به طور صریح درخواست یک بلوک از حافظه را ارسال می‌کردید. اما نکته حیاتی اینجا بود که پس از پایان کار با آن داده، خودتان باید کدی برای نابود کردن و آزادسازی آن می‌نوشتید.

اگر یک برنامه‌نویس حتی پاک‌سازی یک قطعه کوچک از داده‌ها را فراموش می‌کرد، آن بخش از حافظه برای همیشه اشغال باقی می‌ماند. به این پدیده **نشت حافظه (Memory Leak)** می‌گویند. اگر یک بازی به طور مداوم دچار نشت حافظه می‌شد، در نهایت تمام رم (RAM) کامپیوتر را می‌بلعید و سیستم‌عامل مجبور می‌شد بازی را به طور ناگهانی ببندد و کرش (Crash) کند. برای حذف این بار ذهنی و دغدغه همیشگی، دانشمندان علوم کامپیوتر زباله‌روب (GC) را اختراع کردند؛ یک سیستم خودکار که به صورت دوره‌ای کف انبار را اسکن می‌کند تا زباله‌های واقعی را شناسایی کرده و آن‌ها را با امنیت کامل دور بریزد.

#### ⚠️ مسئله اصلی: یافتن اشیاء شبه‌مانند (Ghost Objects)

یک ماشین از کجا می‌فهمد که دیگر به یک قطعه داده نیازی نیست؟ ماشین که نمی‌تواند ذهن شما را بخواند. اگر زباله‌روب به اشتباه شیئی را پاک کند که کدهای بازی شما هنوز در حال تلاش برای خواندن آن هستند، بازی فوراً با یک خطای مهلک کرش می‌کند. بنابراین، ماشین به یک روش کاملاً ایمن و خطاناپذیر نیاز دارد تا تشخیص دهد کدام داده «زنده» و کدام داده «مرده» است.

#### 💡 نحوه حل مسئله: سیستم ردگیری ریشه‌ها (Root Tracking)

سیستم GC این مشکل را با استفاده از فرآیندی به نام **علامت‌گذاری و جاروب (Mark and Sweep)** حل می‌کند.

تصور کنید روی میز کار سریع و سازمان‌یافته شما (حافظه استک یا Stack)، نخ‌هایی قرار دارند که به جعبه‌های مستقر در کف انبار (حافظه هیپ یا Heap) متصل شده‌اند. این نقاط شروع روی میز کار شما — به همراه سیستم‌های سراسری و ریشه‌ای بازی — **ریشه‌های زباله‌روب (GC Roots)** نامیده می‌شوند.

وقتی سرایدار فرآیند پاک‌سازی را آغاز می‌کند، یک معماری دو مرحله‌ای را به اجرا می‌گذارد:

1. **مرحله علامت‌گذاری (Mark Phase):** زباله‌روب از «ریشه‌ها» شروع کرده و تک‌تک نخ‌های متصل را به سمت اشیاء درون انبار دنبال می‌کند. هر شیئی که زباله‌روب موفق به لمس آن شود، یک «برچسب سبز درخشان» دریافت می‌کند. اگر شیء الف به شیء ب اشاره کند، شیء ب نیز برچسب می‌خورد.
2. **مرحله جاروب (Sweep Phase):** زباله‌روب به صورت خطی قدم به قدم کف انبار را طی می‌کند. هر شیئی که برچسب سبز *نداشته باشد*، به عنوان یک شیء غیرقابل دسترس و شبه‌مانند در نظر گرفته می‌شود. زباله‌روب آن اشیاء را متلاشی کرده و آن فضای انبار را برای داده‌های آینده به عنوان فضای «خالی» علامت‌گذاری می‌کند.

### ۲. [مدل زباله‌روب نسلی و باندهای طول عمر اشیاء (Generational GC Model)](./12-2-Generational-GC-Model-Object-Lifetime-Bands-FA.md)

#### 📜 افسانه‌ها و تاریخچه علوم کامپیوتر: فرضیه مرگ در کودکی

دانشمندان علوم کامپیوتر هنگام تحلیل رفتار نرم‌افزارها متوجه یک الگوی آماری شگفت‌انگیز شدند: **بیشتر داده‌ها در سنین بسیار پایین می‌میرند**. اکثریت قاطع متغیرهایی که درون یک تابع ایجاد می‌شوند، تنها برای کسری از ثانیه مورد نیاز هستند. در مقابل، اگر شیئی بتواند از چند دقیقه اول حضور خود در انبار جان سالم به در ببرد، به احتمال زیاد تا پایان اجرای برنامه باقی خواهد ماند. این واقعیت علمی **فرضیه نسل ضعیف (Weak Generational Hypothesis)** نام دارد.

#### ⚠️ مسئله اصلی: جستجوی کل انبار در هر بار پاک‌سازی

اگر بازی شما دو ساعت است که در حال اجراست، انبار شما پر از هزاران شیء دائمی شده است (مانند سیستم‌های اصلی رابط کاربری، داده‌های ذخیره‌شده بازیکن و نقشه‌های مرحله). اگر سرایدار مجبور باشد در هر بار جستجوی زباله، تمام نخ‌ها را ردیابی کند و تک‌تک این اشیاء دائمی را بازرسی نماید، فرآیند پاک‌سازی هر بار طولانی‌تر شده و باعث افت فریم‌های ناگهانی و غیرقابل‌پیش‌بینی (Lag Spikes) در بازی می‌شود.

#### 💡 نحوه حل مسئله: تفکیک سنی (نسل‌های ۰، ۱ و ۲)

برای بهینه‌سازی این فرآیند، زبان سی‌شارپ حافظه هیپ را به سه منطقه تکاملی مجزا به نام **نسل‌ها (Generations)** تقسیم می‌کند:

* **نسل ۰ (مهد کودک):** این بخش محل قرارگیری تمام اشیاء کاملاً جدید است. از آنجا که بیشتر اشیاء جوانمرگ می‌شوند، این منطقه به طور مکرر و در بازه‌های زمانی کوتاه اسکن می‌شود. حجم این ناحیه کوچک است، بنابراین اسکن آن تقریباً هیچ زمانی نمی‌برد.
* **نسل ۱ (منطقه بافر یا برزخ):** اگر زباله‌روب نسل ۰ را پاک‌سازی کند و متوجه شود شیئی *هنوز زنده است* (مانند افکت یک جادو که انیمیشن آن در جریان است)، آن شیء به نسل ۱ ارتقا می‌یابد. این منطقه مانند یک اتاق انتظار میان داده‌های کوتاه‌مدت و بلندمدت عمل می‌کند.
* **نسل ۲ (خانه سالمندان):** اگر شیئی از پاک‌سازی نسل ۱ نیز جان سالم به در ببرد، به نسل ۲ منتقل می‌شود. اینجاست که سیستم‌های دائمی شما جا خوش می‌کنند. زباله‌روب فرض می‌کند این آیتم‌ها ماندگار هستند، به همین دلیل نسل ۲ را بسیار به‌ندرت اسکن می‌کند.

> **نکته در یونیتی:** ساختارهای سنتی یونیتی به طور پیش‌فرض از یک نسخه اصلاح‌شده و بدون نسل (تغییریافته از زباله‌روب Boehm) استفاده می‌کردند، اما در محیط‌های مدرن یونیتی و زمان‌اجراهای جدید دات‌نت (NET.)، درک این باندهای طول عمر برای جلوگیری از پیر شدن ناخواسته داده‌ها و ورود آن‌ها به بخش‌های ماندگار حافظه حیاتی است.

### ۳. [معیارهای هیپ اشیاء کوچک در مقابل هیپ اشیاء بزرگ (SOH vs. LOH)](./12-3-Small-Object-Heap-vs-Large-Object-Heap-Metrics-FA.md)

#### 📜 افسانه‌ها و تاریخچه علوم کامپیوتر: هزینه محموله‌های سنگین

وقتی مدیر انبار می‌خواهد فضایی را به آیتم‌ها اختصاص دهد، ترجیح می‌دهد آن‌ها را به طور مرتب و شانه به شانه کنار هم بچیند. با این حال، اگر یک محموله غول‌پیکر از راه برسد، رفتار کردن با آن مانند یک بسته کوچک، تمام نظم مسیرهای توزیع انبار را به هم می‌زند. به همین دلیل، سیستم‌های زمان‌اجرا انبار را به دو طبقه فیزیکی کاملاً متفاوت تقسیم می‌کنند: **هیپ اشیاء کوچک (SOH)** و **هیپ اشیاء بزرگ (LOH)**.

#### ⚠️ مسئله اصلی: تکه‌تکه‌شدن انبار (Memory Fragmentation)

تصور کنید SOH مانند یک قفسه برای جعبه‌های کفش است. اگر یک جعبه را بردارید، یک شکاف کوچک باقی می‌ماند و می‌توانید به‌راحتی جعبه کفش دیگری را درون آن شکاف سُر دهید.

اما چه اتفاقی می‌افتد اگر یک ساختار داده عظیم ایجاد کنید — مثلاً یک آرایه بافت تصویری بزرگ 4K یا یک بافر شبکه چندنفره غول‌آسا؟ در سی‌شارپ، هر شیء واحد که حجمی **برابر یا بیشتر از ۸۵,۰۰۰ بایت** داشته باشد، به عنوان یک غول پیکر طبقه‌بندی می‌شود.

اگر این شیء غول‌پیکر را روی قفسه معمولی بگذارید و سپس آن را حذف کنید، یک حفره عمیق و وسیع به جا می‌گذارید. به مرور زمان، تخصیص و حذف این غول‌ها باعث می‌شود کف انبار حافظه شما شبیه به یک پنیر سوئیسی پر از سوراخ شود. به این پدیده **تکه‌تکه‌شدن حافظه (Memory Fragmentation)** می‌گویند. در این حالت ممکن است شما در مجموع ۵۰ مگابایت فضای خالی داشته باشید، اما چون این فضا به میلیون‌ها شکاف میکروسکوپی تقسیم شده است، نمی‌توانید حتی یک شیء بزرگ جدید را در هیچ‌کجای آن جا دهید؛ امری که منجر به کرش ناگهانی بازی به دلیل اتمام حافظه (Out of Memory) می‌شود.

#### 💡 نحوه حل مسئله: بارانداز کالاهای سنگین (LOH)

برای حل این مشکل، سیستم با هیپ اشیاء بزرگ (Large Object Heap) به شکل کاملاً متفاوتی رفتار می‌کند:

* اشیاء مساوی یا بزرگ‌تر از ۸۵,۰۰۰ بایت مهدکودک را دور زده و مستقیماً به LOH فرستاده می‌شوند.
* سیستم به طور خودکار LOH را بخشی از **نسل ۲** در نظر می‌گیرد.
* نکته کلیدی اینجاست که برای حفظ کارایی و عملکرد سیستم، **زباله‌روب در طول پاک‌سازی‌های استاندارد، اشیاء درون LOH را فشرده یا بازآرایی نمی‌کند**؛ چرا که جابه‌جا کردن بلوک‌های عظیمی از داده‌ها در حافظه، بار پردازشی فوق‌العاده سنگینی به پردازنده (CPU) تحمیل می‌کند. در عوض، سیستم یک دفترچه راهنما از بلوک‌های خالی نگه می‌دارد و غول‌های جدید را دقیقاً در جاهایی قرار می‌دهد که با ابعاد آن‌ها همخوانی داشته باشند.

### ۴. [خطوط لوله زباله‌روب تدریجی و توقف‌های اجرایی (Incremental GC)](./12-4-Incremental-GC-Pipelines-Execution-Pauses-FA.md)

#### 📜 افسانه‌ها و تاریخچه علوم کامپیوتر: تکامل همزیستی

در موتورهای بازی‌سازی اولیه، فرآیند زباله‌روبی کاملاً **همگام (Synchronous)** بود. منطق بازی و سرایدار هر دو از یک دست فیزیکی مشترک استفاده می‌کردند؛ یا بازی می‌توانست اجرا شود، یا سرایدار انبار را جارو می‌کرد، اما هر دو هرگز نمی‌توانستند همزمان کار کنند. این امر منجر به فریزها و قفل شدن‌های معروف بازی می‌شد. علوم کامپیوتر مدرن این چالش را با طراحی خطوط لوله تدریجی (Incremental) و همزمان (Concurrent) حل کرد.

#### ⚠️ مسئله اصلی: سد ۶۰ فریم بر ثانیه

برای حفظ یک بازی روان با نرخ ۶۰ فریم بر ثانیه، موتور بازی دقیقاً **۱۶.۶ میلی‌ثانیه** فرصت دارد تا همه‌چیز را پردازش کند: فیزیک، هوش مصنوعی، رندر گرافیکی و منطق گیم‌پلی. اگر یک پاک‌سازی استاندارد GC حدود ۳۰ میلی‌ثانیه طول بکشد تا انبار را اسکن کند، حفظ نرخ فریم روان از نظر فیزیکی غیرممکن خواهد بود. بازی دچار لکنت یا تپق (Stutter) می‌شود و غوطه‌وری بازیکن در دنیای بازی کاملاً از بین می‌رود.

#### 💡 نحوه حل مسئله: قاچ کردن فرآیند جاروب

یونیتی از یک **زباله‌روب تدریجی (Incremental Garbage Collector)** استفاده می‌کند. این سیستم به جای اینکه بازی را منجمد کند و کل انبار را در یک اقدام طولانی و زجرآور اسکن نماید، عملیات پاک‌سازی را به کارهای کوچک و مینیاتوری تقسیم می‌کند.

اگر سرایدار برای تمیز کردن انبار در مجموع به ۲۰ میلی‌ثانیه زمان نیاز داشته باشد، زباله‌روب تدریجی در هر فریم تنها **۱ یا ۲ میلی‌ثانیه** کار می‌کند، کار خود را دقیقاً در میانه انبار متوقف کرده و کنترل را به موتور بازی برمی‌گرداند تا فریم را به‌موقع رندر کند؛ سپس در فریم بعدی، ادامه جاروب خود را از همان نقطه از سر می‌گیرد.

### ۵. [الگوی ساختاری مخزن اشیاء (Object Pooling Structural Pattern)](./12-5-Object-Pooling-Structural-Pattern-FA.md)

#### 📜 افسانه‌ها و تاریخچه علوم کامپیوتر: جام مقدسِ تخصیص صفر

اگر سیستم‌های خودکار زبان برنامه‌نویسی شما ذاتاً هنگام پاک‌سازی زباله‌ها خطرات عملکردی ایجاد می‌کنند، راهکار معماری نهایی بسیار ساده است: **اصلاً زباله تولید نکنید**. اگر هیچ‌چیز را دور نریزید، سرایدار هرگز مجبور به بیدار شدن نیست، خط لوله تدریجی نیازی به قاچ کردن کارهایش ندارد و نمودار کارایی سیستم به یک خط کاملاً صاف و کاملاً قابل پیش‌بینی تبدیل می‌شود.

#### ⚠️ مسئله اصلی: ناامیدی در بازی‌های پرتابه‌ای (Bullet Hell)

یک بازی آرکید سبک رگبار گلوله را تصور کنید که در آن بازیکن در هر ثانیه ۵۰ تیر لیزری شلیک می‌کند. اگر هر تیر لیزری یک شیء جدید روی حافظه هیپ ایجاد کند و هنگام خروج از صفحه آن را نابود سازد، شما در هر دقیقه هزاران شیء تولید می‌کنید. شما عملاً در حال دفن کردن انبار خود زیر بهمنی از جعبه‌های مقوایی مصرف‌شده هستید؛ پدیده‌ای که زباله‌روب را مجبور به اضافه کاری مداوم کرده و پاشنه آشیل فریم ریت بازی می‌شود.

#### 💡 نحوه حل مسئله: دفتر ثبت بازیافت فعال

ما به جای اینکه اجازه دهیم یک شیء بمیرد، آن را به حالت خواب زمستانی عمیق می‌بریم. وقتی دیگر به یک آیتم نیاز نیست، آن را خاموش کرده و آدرس آن را درون یک مخزن اختصاصی به نام **مخزن اشیاء (Object Pool)** ذخیره می‌کنیم. هرگاه به یک آیتم جدید نیاز داشتیم، ابتدا مخزن را بررسی می‌کنیم. اگر شیء بیکاری وجود داشته باشد، آن را بیدار کرده، متغیرهایش را تغییر می‌دهیم و بدون اینکه حتی یک بایت جدید روی حافظه هیپ تخصیص دهیم، آن را به دنیای بازی بازمی‌گردانیم.

در ادامه نحوه پیاده‌سازی یک مخزن اشیاء فوق‌بهینه با استفاده از سیستم داخلی و توکار یونیتی آورده شده است:

```csharp
using UnityEngine;
using UnityEngine.Pool;

public class HighPerformanceLaserPool : MonoBehaviour
{
    [SerializeField] private GameObject laserPrefab;
    
    // موتور اصلی الگوی ساختاری مخزن اشیاء
    private ObjectPool<GameObject> _pool;

    private void Awake()
    {
        // تنظیم قوانین بازیافت در ابتدای کار:
        _pool = new ObjectPool<GameObject>(
            createFunc: CreateNewLaser,          // اگر کشو خالی بود، چگونه یک لیزر جدید بسازد
            actionOnGet: ActivateLaser,         // هنگام بیرون کشیدن لیزر از کشو چه کاری انجام دهد
            actionOnRelease: DeactivateLaser,   // هنگام بازگرداندن لیزر به داخل کشو چه کاری انجام دهد
            actionOnDestroy: DestroyLaserInstance, // پشتیبان نابودی سخت در صورتی که مخزن بیش از حد بزرگ شد
            collectionCheck: true,              // بررسی ایمنی برای جلوگیری از انداختن دوجانبه یک آیتم تکراری به مخزن
            defaultCapacity: 50,                // تعداد اشیایی که در زمان شروع بازی به طور پیش‌فرض آماده می‌شوند
            maxSize: 200                        // سقف مطلق برای جلوگیری از مصرف بیش از حد حافظه انبار
        );
    }

    // ۱. کارخانه پشتیبان: تنها زمانی اجرا می‌شود که مخزن ما کاملاً خالی شده باشد
    private GameObject CreateNewLaser()
    {
        GameObject projectile = Instantiate(laserPrefab);
        // یک جزء ردیاب پنهان به شیء می‌دهیم تا بداند چطور به این مخزن بازگردد
        projectile.AddComponent<PooledObjectTracker>().LinkPool(_pool);
        return projectile;
    }

    // ۲. بیدار کردن: بیرون کشیدن یک آیتم از خواب زمستانی
    private void ActivateLaser(GameObject laser)
    {
        laser.SetActive(true);
    }

    // ۳. خواب زمستانی: قرار دادن امن آیتم در داخل کشوی بازیافت
    private void DeactivateLaser(GameObject laser)
    {
        laser.SetActive(false);
    }

    // ۴. امحاء سخت: تنها زمانی استفاده می‌شود که از سقف مجاز نگهداری انبار فراتر رفته باشیم
    private void DestroyLaserInstance(GameObject laser)
    {
        Destroy(laser);
    }

    // یک API عمومی برای اسلحه‌های شما تا به جای متد پیش‌فرض Instantiate یونیتی از آن استفاده کنند
    public GameObject FireLaser(Vector3 position, Quaternion rotation)
    {
        GameObject laser = _pool.Get();
        laser.transform.position = position;
        laser.transform.rotation = rotation;
        return laser;
    }
}

// یک اسکریپت ردیاب کوچک متصل به لیزر برای مدیریت فرآیند بازیافت خود
public class PooledObjectTracker : MonoBehaviour
{
    private ObjectPool<GameObject> _associatedPool;

    public void LinkPool(ObjectPool<GameObject> pool)
    {
        _associatedPool = pool;
    }

    // لیزر به جای فراخوانی متد مخرب (Destroy)، هنگام برخورد با دیوار این متد را صدا می‌زند
    public void Recycle()
    {
        if (_associatedPool != null)
        {
            _associatedPool.Release(this.gameObject);
        }
    }
}
```
