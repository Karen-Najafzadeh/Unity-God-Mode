

# معماری بیتی و سیستم‌های فلگ‌های باینری (Bitwise Architecture & Binary Flags Systems)

---

### ۱. مقدمه: ماتریس برد سوئیچ (The Switchboard Matrix)

به فصل ۱۳ خوش آمدید! اگر مسیر ما را در طبقات کارخانه RAM دنبال کرده باشید، از قبل می‌دانید که کامپیوتر چگونه کارگاه خود را به میز کار سریع Stack و انبار گسترده Heap تقسیم می‌کند. اما تا به حال، ما داده‌ها را با استفاده از کانتینرهای حمل استاندارد ارائه‌شده توسط زبان‌های سطح بالا مدیریت می‌کردیم: اعداد صحیح ۴ بایتی، مقادیر بولی ۱ بایتی و مراجع شیء (Object References) سنگین.

اکنون، ما خود را تا لایه زیر اتمی محاسبات کوچک می‌کنیم. ما قدم به درون **معماری بیتی (Bitwise Architecture)** می‌گذاریم.

تصور کنید در حال طراحی یک بازی نقش‌آفرینی (RPG) جهان‌باز در Unity با هزاران شخصیت فعال هستید. هر شخصیت می‌تواند به‌طور هم‌زمان آتش بگیرد، مسموم شود، منجمد شود، نامرئی شود، پرواز کند، گیج (Stun) شود یا ده‌ها اثر وضعیت (Status Effects) دیگر را حمل کند. اگر این وضعیت‌ها را با استفاده از روش‌های سنتی مدیریت کنید، کارخانه خود را از جعبه‌های مقوایی حجیم و خالی ساخته‌اید. با اتخاذ یک سیستم فلگ‌های باینری بیتی (Bitwise Binary Flags)، داده‌های خود را به یک پنل کنترل فوق‌العاده متراکم تبدیل می‌کنید که در آن چندین وضعیت پیچیده در فضای میکروسکوپی یک عدد واحد بسته‌بندی می‌شوند.

---

### ۲. تاریخچه و افسانه علوم کامپیوتر: عصر تک‌بایت (The Era of the Single Byte)

در عصر مدرن محاسبات، گیگابایت‌های رم ارزان هستند، که این امر توسعه‌دهندگان را تا حدی دست‌ودل‌باز (و حتی ولخرج) کرده است. اما در اواسط قرن بیستم، در آغاز علوم کامپیوتر، هر سوئیچ الکترونیکی یک کالای لوکس فیزیکی بود. حافظه نه با گیگابایت، بلکه با *بایت* سنجیده می‌شد؛ گروه‌هایی متشکل از هشت کلید مکانیکی یا مغناطیسی کوچک.

برای مهندسی که روی کابینت‌های آرکید اولیه یا کامپیوتر راهنمای آپولو (Apollo Guidance Computer) کار می‌کرد، تخصیص یک بایت کامل از حافظه فقط برای ذخیره یک پاسخ ساده «بله یا خیر»، یک شکست مهندسی مطلق محسوب می‌شد. آن‌ها می‌دانستند که یک بایت واحد شامل **۸ بیت** است و هر بیت یک لامپ میکروسکوپی مستقل است که می‌تواند **۰ (خاموش/نادرست)** یا **۱ (روشن/درست)** باشد.

افسانه «دستکاری‌کنندگان بیت» (Bit-Twiddlers) در اینجا متولد شد. برنامه‌نویسان اولیه یک عدد صحیح را صرفاً به عنوان یک عدد مانند `42` نمی‌دیدند. آن‌ها آن را به عنوان آرایه‌ای از ۸، ۱۶ یا ۳۲ سوئیچ فیزیکی مجزا نگاه می‌کردند. با دستکاری مستقیم این سوئیچ‌ها با استفاده از ریاضیات سطح سخت‌افزار، آن‌ها می‌توانستند کل موجودی (Inventory)، وضعیت‌ها و پیکربندی‌های تجهیزات یک شخصیت بازی را در یک عدد واحد ذخیره کنند. این سنت همان چیزی است که موتورهای با کارایی بالا مانند Unity را امروزه فوق‌العاده سریع نگه می‌دارد.

### ۳. مسئله اصلی: انفجار وضعیت‌های بولی (The Boolean Status Explosion)

زمانی که توسعه‌دهندگان بدون پیش‌زمینه علوم کامپیوتر سیستم‌های حالت پیچیده می‌سازند، به‌طور طبیعی به سراغ چیزی می‌روند که از نظر بصری منطقی‌تر است: **مقادیر بولی (`bool`)**.

بیایید ببینیم وقتی اسکریپتی مانند این می‌نویسید، در پشت صحنه چه اتفاقی می‌افتد:

```csharp
public class StatusController : MonoBehaviour
{
    public bool isPoisoned;
    public bool isBurning;
    public bool isFrozen;
    public bool isStunned;
    public bool isInvisible;
    public bool isFlying;
    public bool isSlowed;
    public bool isInvulnerable;
}
```

#### هدررفت نامرئی (The Invisible Waste)

در زبان #C، یک `bool` واحد **۱ بایت (۸ بیت)** از حافظه را اشغال می‌کند. با وجود اینکه یک مقدار بولی تنها نشان‌دهنده یک `۱` یا `۰` است، کامپیوتر نمی‌تواند به‌راحتی در هر زمان به چیزی کوچک‌تر از یک بایت کامل آدرس‌دهی کند.

* ذخیره‌سازی این ۸ کلید وضعیت، **۸ بایت** داده به ازای هر شخصیت مصرف می‌کند.
* اگر ۱۰,۰۰۰ دشمن فعال در یک شبیه‌سازی عظیم داشته باشید، این یعنی **۸۰,۰۰۰ بایت** فقط برای کلیدهای حالت.
* وقتی کد شما می‌خواهد بررسی کند که آیا یک شخصیت می‌تواند حرکت کند یا خیر، باید زنجیره آشفته‌ای از شرایط را اجرا کند: `if (!isFrozen && !isStunned && !isSlowed)`. این کار پردازنده (CPU) را مجبور می‌کند بین نقاط حافظه مجزا و متعدد بپرد و تمرکز حافظه پنهان (Cache Focus) آن را مختل کند.

علاوه بر این، اگر بخواهید این وضعیت‌ها را از طریق شبکه برای همگام‌سازی بازی‌های چندنفره ارسال کنید، در هر فریم در حال ارسال ۸ متغیر مجزا در وب هستید. این امر باعث تورم شدید داده‌ها می‌شود که می‌تواند منجر به تاخیر شبکه (Network Lag) و کاهش کارایی سیستم شود.

### ۴. راهکار معماری: شابلون‌های بیتی یا ماسک‌گذاری (Bitwise Stencils / Masking)

معماری بیتی این مشکل را با استفاده از یک عدد صحیح واحد، مانند یک `int` استاندارد ۴ بایتی (که شامل **۳۲ بیت/سوئیچ** است) برای ذخیره تا ۳۲ فلگ مستقل True/False حل می‌کند. به جای ایجاد متغیرهای مجزا، ما هر وضعیت را به یک «خانه» یا ستون خاص در نمایش باینری آن تک‌عدد اختصاص می‌دهیم.

```text
Bit Position:  [7] [6] [5] [4] [3] [2] [1] [0]
Status Role:   Inv Fly Inv Stn Frz Brn Psn Dead
Current State:  0   1   0   0   0   1   1   0   (Value = 70)
```

در این بایت واحد، شخصیت در حال حاضر به‌طور هم‌زمان **درحال پرواز (Flying)**، **درحال سوختن (Burning)** و **مسموم (Poisoned)** است، در حالی که تمام وضعیت‌های دیگر کاملاً خاموش هستند.

برای خواندن و نوشتن روی این سوئیچ‌های میکروسکوپی مجزا بدون مزاحمت برای همسایگانشان، از عملگرهای باینری استفاده می‌کنیم. این عملگرها را مانند **شابلون‌ها (یا ماسک‌ها)** تصور کنید که سوئیچ‌هایی را که به آن‌ها اهمیت نمی‌دهیم مسدود می‌کنند و تنها سوئیچ هدف را در معرض دید قرار می‌دهند.

#### مجموعه ابزار اصلی بیتی (The Core Bitwise Toolset):

1. **عملگر یا بیتی OR (`|`) — کلید روشن‌کننده:** دو الگو را با هم ترکیب می‌کند. اگر یک لامپ در هر کدام از الگوها روشن باشد، روشن باقی می‌ماند. ما از این عملگر برای اعمال یک اثر وضعیت (Status Effect) استفاده می‌کنیم.
2. **عملگر و بیتی AND (`&`) — پنجره بازرس:** دو الگو را با هم مقایسه می‌کند. تنها در صورتی مقدار `1` را برمی‌گرداند که لامپ در *هر دو* الگو روشن باشد. ما از این عملگر برای بررسی اینکه آیا یک شخصیت وضعیت خاصی دارد یا خیر استفاده می‌کنیم.
3. **عملگر XOR بیتی (`^`) — کلید معکوس‌کننده:** وضعیت یک سوئیچ را تغییر می‌دهد (یا به اصطلاح Toggle می‌کند). اگر `1` باشد، به `0` تبدیل می‌شود و اگر `0` باشد، به `1` تبدیل می‌شود.
4. **عملگرهای شیفت بیت (`>>` یا `<<`):** همه لامپ‌ها را به سمت چپ یا راست می‌لغزانند و به ما این امکان را می‌دهند که بدون زحمت مشخص کنیم وضعیت ما کدام ستون را اشغال می‌کند.

### ۵. نمونه کدهای جامع

بیایید نگاهی بیندازیم به اینکه چگونه یک کد بهینه‌نشده و با تخصیص حافظه بالا (High-Allocation)، مدیریت حالت را انجام می‌دهد و در مقابل، چگونه یک سیستم پیشرفته فلگ‌های باینری بیتی آن را با صفر درصد سرریز (Zero Overhead) مدیریت می‌کند.

#### ❌ رویکرد تخصیص‌دهنده متورم (پهنای باند بالا و هدررفت حافظه)

این اسکریپت بهینه‌نشده، وضعیت‌های شخصیت را با استفاده از متغیرهای بولی استاندارد و به‌روزرسانی‌های تحت شبکه مبتنی بر رشته (String-based) مدیریت می‌کند. این کار باعث ایجاد تورم حافظه قابل‌توجهی می‌شود و CPU را مجبور می‌کند دائماً چندین آدرس حافظه مستقل را ارزیابی کند.

```csharp
using UnityEngine;
using System.Collections.Generic;

public class BloatedStatusManager : MonoBehaviour
{
    // ۸ بایت مجزا روی هیت یا استک به ازای هر موجود تخصیص می‌یابد
    public bool isDead;
    public bool isPoisoned;
    public bool isBurning;
    public bool isFrozen;
    public bool isStunned;
    public bool isInvisible;
    public bool isFlying;
    public bool isInvulnerable;

    public void ApplyPoison() => isPoisoned = true;
    public void CurePoison() => isPoisoned = false;

    // روشی ناکارآمد و با تخصیص بالا برای بررسی اینکه آیا بازیکن می‌تواند جادو بخواند یا خیر
    public bool CanCastSpell()
    {
        // پردازنده باید ۴ مکان حافظه مختلف را به صورت متوالی نگاه کند
        if (isDead || isStunned || isFrozen)
        {
            return false;
        }
        return true;
    }

    // شبیه‌سازی ارسال داده‌های وضعیت از طریق شبکه یا سیستم ذخیره‌سازی بازی
    public string GetStatusNetworkString()
    {
        // تولید زباله بالا! رشته‌ها را به صورت پویا روی حافظه Heap تخصیص می‌دهد
        List<string> activeStatuses = new List<string>();
        if (isPoisoned) activeStatuses.Add("Poisoned");
        if (isBurning) activeStatuses.Add("Burning");
        if (isFrozen) activeStatuses.Add("Frozen");
        
        return string.Join(",", activeStatuses.ToArray());
    }
}
```

#### 👑 قهرمان معماری (سیستم فلگ‌های بیتی با تخصیص حافظه صفر)

با استفاده از ویژگی `[System.Flags]` در #C در کنار عملگرهای شیفت بیتی، تمام آن متغیرهای بولی را در یک کانتینر تک ۴ بایتی فشرده می‌کنیم. پردازنده (CPU) می‌تواند ده‌ها حالت را تنها در یک چرخه کلاک (Clock Cycle) ارزیابی، پاک‌سازی یا ترکیب کند.

```csharp
using UnityEngine;
using System;

// ویژگی [System.Flags] به یونیتی و سی‌شارپ می‌گوید که این اینام یک ماسک بیتی است
[Flags]
public enum CharacterStates : uint
{
    None         = 0,        // 00000000 00000000
    IsDead       = 1 << 0,   // 00000000 00000001 (مقدار ۱)
    IsPoisoned   = 1 << 1,   // 00000000 00000010 (مقدار ۲)
    IsBurning    = 1 << 2,   // 00000000 00000100 (مقدار ۴)
    IsFrozen     = 1 << 3,   // 00000000 00001000 (مقدار ۸)
    IsStunned    = 1 << 4,   // 00000000 00010000 (مقدار ۱۶)
    IsInvisible  = 1 << 5,   // 00000000 00100000 (مقدار ۳۲)
    IsFlying     = 1 << 6,   // 00000000 01000000 (مقدار ۶۴)
    IsInvulnerable= 1 << 7   // 00000000 10000000 (مقدار ۱۲۸)
}

public class BitwiseStatusManager : MonoBehaviour
{
    // یک متغیر تک ۴ بایتی از نوع عدد صحیح که همه ۸ حالت را به‌طور هم‌زمان ذخیره می‌کند!
    [SerializeField] private CharacterStates currentStates = CharacterStates.None;

    private void Start()
    {
        Debug.Log("--- Initializing Bitwise Architecture ---");
        
        // ۱. اضافه کردن حالت‌ها با استفاده از عملگر بیتی OR (|)
        ApplyState(CharacterStates.IsPoisoned | CharacterStates.IsBurning);
 /*       
        IsPoisoned:    0  0  0  0  0  0  1  0    (خانه ۱ روشن است)
      | IsBurning:     0  0  0  0  0  1  0  0    (خانه ۲ روشن است)
        ----------------------------------------
        Result state : 0  0  0  0  0  1  1  0    (هر دو خانه اکنون ۱ هستند!)
*/
        PrintBinaryRepresentation();

        // ۲. بررسی حالت‌ها با استفاده از عملگر بیتی AND (&)
        if (HasState(CharacterStates.IsPoisoned))
        {
            Debug.Log("The entity is suffering from toxic poison!");
        }

        // ۳. بررسی چندین محدودیت ترکیب‌شده در تنها یک خط کد
        // ما یک شابلون (ماسک) برای شرایطی که شخصیت را ناتوان می‌کنند ایجاد می‌کنیم
        CharacterStates crowdControlled = CharacterStates.IsDead | CharacterStates.IsFrozen | CharacterStates.IsStunned;
        
        // اگر اشتراک بین حالت‌های فعلی و شرایط ناتوانی صفر نباشد، موجود ناتوان شده است
        bool canMove = (currentStates & crowdControlled) == CharacterStates.None;
        Debug.Log($"Can Entity Move? {canMove}");

        // ۴. حذف یک حالت با استفاده از عملگرهای بیتی NOT (~) و AND (&)
        RemoveState(CharacterStates.IsBurning);
        Debug.Log("Quenched the fire status.");
        PrintBinaryRepresentation();
        
        // ۵. تغییر وضعیت آسان یک حالت با استفاده از عملگر XOR (^)
        ToggleState(CharacterStates.IsInvisible);
        Debug.Log("Toggled invisibility field.");
        PrintBinaryRepresentation();
    }

    public void ApplyState(CharacterStates stateToApply)
    {
        // دستکاری مستقیم سخت‌افزار: وادار کردن بیت‌های مشخص‌شده به ۱ شدن
        currentStates |= stateToApply;
    }

    public void RemoveState(CharacterStates stateToRemove)
    {
        // معکوس کردن شابلون (روشن کردن همه چیز به جز وضعیت هدف)، سپس استفاده از AND
        // برای پاک کردن تمیزِ تنها بیت مشخص‌شده، در حالی که بقیه بیت‌ها حفظ می‌شوند.
        currentStates &= ~stateToRemove;
    }

    public bool HasState(CharacterStates stateToCheck)
    {
        // نگاه کردن از طریق پنجره شابلون. اگر نتیجه با وضعیت مورد بررسی برابر باشد، درست (True) است.
        return (currentStates & stateToCheck) == stateToCheck;
    }

    public void ToggleState(CharacterStates stateToToggle)
    {
        // تغییر وضعیت آنی بیت: اگر ۱ باشد ۰ می‌شود و اگر ۰ باشد ۱ می‌شود.
        currentStates ^= stateToToggle;
    }

    // متغیر کمکی برای نمایش بصری نحوه دیدن این عدد در حافظه توسط ماشین
    private void PrintBinaryRepresentation()
    {
        string binaryString = Convert.ToString((uint)currentStates, 2).PadLeft(8, '0');
        Debug.Log($"Current Integer Value: {(uint)currentStates} | Binary Configuration: {binaryString}");
    }
}
```

### ۶. خلاصه تحول معماری

| معیار معماری | آرایه بولی متورم / کلاس‌ها | سیستم فلگ‌های باینری بیتی |
| --- | --- | --- |
| **ردپای حافظه (Memory Footprint)** | ۱ بایت به ازای هر کلید وضعیت (۸ بایت برای ۸ وضعیت). | **۱ بیت به ازای هر کلید وضعیت** (در مجموع فقط ۴ بایت برای ذخیره ۳۲ وضعیت نیاز است). |
| **هزینه اجرای پردازنده (CPU Cost)** | کند. نیازمند بررسی چندین آدرس اشاره‌گر مجزا در طول شرط‌ها است. | **آنی و بدون وقفه**. چندین پیکربندی حالت ترکیبی را در یک چرخه کلاک ارزیابی می‌کند. |
| **تبادل داده و سریال‌سازی (Serialization)** | حجم تبادل بالا. نیازمند تبدیل به رشته‌ها، آرایه‌ها یا پیکربندی‌های JSON است. | **فوق‌العاده سبک**. کل بسته شبکه مربوط به وضعیت، تنها یک عدد ساده ارسالی روی کانال است. |
| **تولید زباله برای کامپایلر (GC Garbage)** | ریسک بالا در صورت استفاده از مرتب‌سازی لیست‌ها یا ساخت رشته برای ارزیابی شرایط ترکیبی. | **صفر مطلق**. صرفاً درون ثبات‌های سخت‌افزاری محلی (Hardware Registers) روی استک کار می‌کند. |
