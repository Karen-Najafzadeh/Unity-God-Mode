# پروفایل‌های کامپایل Just-In-Time (JIT): از زبان واسط (IL) تا سیلیکون بومی

---

### ۱. مقدمه: مترجم پویا (The Dynamic Translator)

در فصل‌های قبل، چیدمان فیزیکی حافظه و قدرت انتزاع‌های پارامتریک را بررسی کردیم. با این حال، یک مکانیزم خاموش اما قدرتمند بین انتزاع‌های سطح بالای C# شما و واحدهای اجرایی سخت‌افزار قرار دارد: **کامپایلر Just-In-Time (JIT)**.

هنگامی که برای یونیتی کد C# می‌نویسید، در حال نوشتن کد ماشین نیستید؛ بلکه در حال نوشتن **زبان واسط (Intermediate Language - IL)** هستید. IL یک نمایش بایت‌کد فشرده و مستقل از پلتفرم است. کامپایلر JIT، «مترجم پویایی» است که درون زمان اجرای بازی شما (به‌ویژه ماشین مجازی مونو - Mono) زندگی می‌کند. این کامپایلر متدهایی را که واقعاً در حال استفاده از آن‌ها هستید شناسایی کرده و آن‌ها را از IL به کد اسمبلی بومی و پرسرعت *در لحظه* (On the fly) و در حین اجرای بازی ترجمه می‌کند.

---

### ۲. تاریخچه و افسانه علوم کامپیوتر: جنگ کامپایل‌ها

برای درک JIT، باید تنش تاریخی بین سیستم‌های **Ahead-Of-Time (AOT)** و **تفسیرشده (Interpreted)** را درک کنیم.

* **پارادایم AOT (مانند C/C++):** کامپایلرهای AOT کد را *قبل* از اجرای برنامه به کد ماشین ترجمه می‌کنند. فایل باینری حاصل، به یک معماری CPU خاص (مانند x86 یا ARM) وابسته است. اگرچه این رویکرد بسیار پرسرعت است، اما نیازمند خطوط تولید (Pipeline) پیچیده و وابسته به معماری است.
* **پارادایم تفسیرشده (مانند پایتون/JS اولیه):** یک مفسر کد منبع را خط‌به‌خط می‌خواند و اجرا می‌کند. این روش بسیار قابل‌حمل (Portable) است اما به دلیل هزینه بالای تجزیه در زمان اجرا و نبودِ بهینه‌سازی‌های بومی، به‌طور وحشتناکی کند است.

**کامپایل JIT** به عنوان «جام مقدس» معماری طراحی شد. این کامپایلر قابلیت حملِ زبان‌های تفسیرشده را با سرعت اجرای کدهای کامپایل‌شده AOT ترکیب می‌کند. JIT کد ماشین را به‌صورت پویا تولید می‌کند و به آن اجازه می‌دهد تا در آخرین لحظه ممکن، خود را با معماری CPU میزبان تطبیق دهد.

---

### ۳. مسئله اصلی: مالیات زمان اجرا (The Runtime Tax)

در حالی که JIT برای قابلیت حمل عالی است، یک «مالیات زمان اجرا» تحمیل می‌کند.

اگر بازی شما پیچیده باشد، کامپایلر JIT تا زمانی که بازی در حال اجرا نباشد، نمی‌داند کدام متدها حیاتی هستند. هنگامی که در نهایت تصمیم می‌گیرد یک متد سنگین و کامپایل‌نشده را کامپایل کند:
1. **توقف CPU:** اجرای برنامه را برای تحلیل IL و تولید کد ماشین متوقف می‌کند.
2. **سربار تحلیل:** چرخه های CPU را برای اعمال پاس‌های بهینه‌سازی مصرف می‌کند.
3. **فشار حافظه:** کد ماشین تولیدشده را در یک منطقه حافظه اجرایی اختصاصی ذخیره می‌کند.

در بازی‌های پیچیده یونیتی، این فرآیند باعث ایجاد **میکرو-لرزش‌ها (Micro-stutters)** یا به اصطلاح "Hitch" می‌شود؛ به‌ویژه در لحظات حساس (مثلاً اولین باری که یک دشمن پیچیده ظاهر می‌شود).

---

### ۴. زیر کاپوت: خط لوله JIT (The JIT Pipeline)

هنگامی که بازی یونیتی شما برای اولین بار یک متد کامپایل‌نشده را فراخوانی می‌کند، ماشین مجازی این خط لوله پیچیده را فعال می‌کند:

1. **استخراج IL:** زمان اجرا بایت‌های IL متد را واکشی می‌کند.
2. **پروفایلینگ لایه‌ای (Tier-0 در برابر Tier-1):**
    * **Tier-0 (JIT پایه):** کد غیربهینه را سریعاً تولید می‌کند تا متد بلافاصله اجرا شود.
    * **Tier-1 (JIT بهینه‌ساز):** اگر متد به عنوان یک «مسیر داغ» (Hot Path - پرفراخوان) شناسایی شود، کامپایلر JIT پاس دوم را انجام داده و بهینه‌سازی‌های تهاجمی مانند باز کردن حلقه‌ها (Loop Unrolling)، جایگذاری مقادیر ثابت (Constant Folding) و درون‌خطی‌سازی (Inlining) را اعمال می‌کند.
3. **تخصیص ثبات (Register Allocation):** در اینجا عملکرد بازی برنده یا بازنده می‌شود. کامپایلر JIT تخصیص ثبات را انجام می‌دهد تا تصمیم بگیرد کدام متغیرها در ثبات‌های فیزیکی فوق‌سریع CPU (مانند RAX, RBX) زندگی کنند و کدام باید به RAM سیستم (که کندتر است) منتقل شوند.
4. **انتشار کد ماشین:** IR (نمایش واسط) به دستورات باینری برای معماری میزبان ترجمه می‌شود.
5. **مدیریت کش کد:** کد اسمبلی تولیدشده در منطقه «حافظه اجرایی» برای فراخوانی‌های بعدی ذخیره می‌شود.

### ۵. زمینه یونیتی: Mono JIT در برابر IL2CPP AOT

درک این دو Backend برای مهندسی با عملکرد بالا در یونیتی الزامی است:

#### رویکرد Mono JIT (توسعه / دسکتاپ)
* **رفتار:** کد را در زمان اجرا کامپایل می‌کند.
* **مزایا:** زمان بیلد سریع، پشتیبانی از Reflection و `System.Reflection.Emit` (تولید کد پویا).
* **معایب:** تاخیر در راه‌اندازی، وقفه‌های غیرقابل‌پیش‌بینی زمان اجرا، ردپای حافظه بالاتر به دلیل کش کد.

#### رویکرد IL2CPP AOT (تولید / موبایل / کنسول)
* **رفتار:** یونیتی کد C# شما را *قبل* از بیلد به C++ تبدیل کرده و سپس آن C++ را به کد ماشین (AOT) کامپایل می‌کند.
* **مزایا:** عملکرد فوق‌العاده سریع، زمان‌بندی دستورالعمل قابل‌پیش‌بینی، سربار زمان اجرای حداقلی.
* **معایب:** زمان‌های بیلد بسیار طولانی، نبود قابلیت تولید کد پویا، افزایش قابل توجه حجم فایل باینری.

### ۶. راهکارهای عملی: مشاهده و کاهش تاخیر

#### ❌ الگوی JIT-Heavy (زمان اجرای بهینه‌نشده)
```csharp
using UnityEngine;

public class JITStressTest : MonoBehaviour
{
    // کامپایلر JIT فقط در اولین اجرا به این متد سر می‌زند.
    // فراخوانی این متد در حین یک صحنه اکشن سنگین، یک فاجعه عملکردی است.
    public void MassiveCalculation(int n)
    {
        float result = 0;
        for (int i = 0; i < n; i++)
        {
            result += Mathf.Sqrt(i) * Mathf.Sin(i);
        }
        Debug.Log(result);
    }
}
```

#### 👑 رویکرد سازگار با AOT: گرم‌سازی JIT (Pre-warming)
برای کاهش وقفه‌ها در محیط Mono، ما «گرم‌سازی» (Pre-warming) را انجام می‌دهیم. ما متد را در حین صفحه بارگذاری یا توالی راه‌اندازی فراخوانی می‌کنیم تا کامپایلر JIT کارهای سنگین خود را *قبل* از شروع گیم‌پلیِ حساس به عملکرد انجام دهد.

```csharp
using UnityEngine;

public class JITPreWarmingManager : MonoBehaviour
{
    [SerializeField] private JITStressTest stressTest;

    private void Awake()
    {
        // ۱. کامپایلر JIT را مجبور می‌کنیم متد را همین الان کامپایل کند.
        // نتایج دور ریخته می‌شوند، اما کد ماشین تولیدشده در کش کد باقی می‌ماند.
        Debug.Log("Pre-warming JIT compilation...");
        stressTest.MassiveCalculation(0); 
        Debug.Log("JIT pre-warming complete.");
    }
}
```

#### ⚖️ مثال ۱: سربار Reflection (JIT در برابر AOT)
Reflection اغلب در محیط‌های JIT مورد سوءاستفاده قرار می‌گیرد. در AOT (IL2CPP)، ریفلکشن جنریک به طور قابل توجهی گران‌تر است، زیرا کامپایلر باید برای هر نمونه‌سازی تایپ (Type Instantiation) کد تولید کند.

```csharp
// بد: انجام این کار داخل حلقه Update در AOT
var field = typeof(MyClass).GetField("myValue"); 
field.SetValue(instance, 10);

// بهتر: کش کردن فیلد یا استفاده از Action/Func delegates برای دور زدن سربار ریفلکشن.
private FieldInfo _cachedField;
void Awake() => _cachedField = typeof(MyClass).GetField("myValue");
void Update() => _cachedField.SetValue(instance, 10);
```

#### ⚖️ مثال ۲: تولید کد پویا (انحصاری JIT)
اگر از `System.Reflection.Emit` برای تولید IL در زمان اجرا استفاده می‌کنید، پروژه شما **در IL2CPP کار نخواهد کرد**.

```csharp
// فقط JIT: این کد IL را به صورت پویا کامپایل می‌کند.
// IL2CPP (AOT) نمی‌تواند این را مدیریت کند زیرا کد ماشین را در زمان بیلد تولید می‌کند.
DynamicMethod method = new DynamicMethod("FastAdd", typeof(int), new[] { typeof(int), typeof(int) });
ILGenerator il = method.GetILGenerator();
il.Emit(OpCodes.Ldarg_0);
il.Emit(OpCodes.Ldarg_1);
il.Emit(OpCodes.Add);
il.Emit(OpCodes.Ret);
```

### ۷. معیارهای تصمیم‌گیری: JIT در برابر AOT

به عنوان یک مهندس Unity God Mode، بک‌اند خود را بر اساس مرحله پروژه و محدودیت‌های هدف انتخاب کنید.

| سناریو | بک‌اند پیشنهادی | چرا؟ |
| :--- | :--- | :--- |
| **نمونه‌سازی سریع** | **Mono (JIT)** | زمان‌های بیلد سریع‌تر امکان حلقه‌های تکرار سریع را فراهم می‌کند. |
| **دیباگ سنگین** | **Mono (JIT)** | دیباگرها به طور طبیعی‌تری با کامپایل JIT زمان اجرا تعامل دارند. |
| **انتشار محصول (Production)** | **IL2CPP (AOT)** | برای حداکثر عملکرد CPU و اجرای کد سازگار با کش (Cache-friendly) الزامی است. |
| **اسکریپت‌نویسی پویا/مادها** | **Mono (JIT)** | JIT اجازه استفاده از `System.Reflection.Emit` را می‌دهد که تزریق کد زمان اجرا را ممکن می‌کند. |
| **هدف موبایل/کنسول** | **IL2CPP (AOT)** | توسط بسیاری از پلتفرم‌ها برای عملکرد و محدودیت‌های بهینه‌سازی فروشگاه الزامی است. |

**قانون نهایی:** اگر پروژه شما به تولید IL پویا (`Reflection.Emit`) متکی است، شما به Mono JIT محدود هستید. اگر به ۶۰+ FPS در صحنه‌های موبایل پیچیده نیاز دارید، باید معماری خود را از روز اول با IL2CPP سازگار طراحی کنید.
