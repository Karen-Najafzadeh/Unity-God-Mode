# تغییرات کوواریانس (Covariance) و کنتراواریانس (Contravariance)

---

### ۱. مقدمه: ماتریس خط لوله تغییر شکل‌دهنده

به کوره معماری خوش آمدید. در فصل‌های قبلی، مفهوم **جنریک‌ها (Generics)** را بررسی کردیم؛ طرح‌هایی امن از نظر نوع (Type-safe) ایجاد کردیم تا کارخانه شما مجبور نباشد برای هر سلاح، زره یا معجون در دنیای RPG خود، ظرف‌های ذخیره‌سازی جداگانه‌ای تولید کند. متوجه شدیم که جنریک‌ها به ما اجازه می‌دهند کدی بسیار قابل استفاده مجدد بنویسیم، بدون اینکه متحمل مالیات سنگین حافظه یعنی "باکسینگ (Boxing)" (بسته‌بندی مقادیر درون اشیاء) شویم.

اما اکنون، مستقیماً با یک محدودیت طراحی پیچیده در معماری‌های جنریک مواجه می‌شویم.

تصور کنید در حال اداره یک کارخانه کشتی‌سازی مداری در آینده هستید. شما یک خط لوله بسیار بهینه برای بازرسی **سفینه‌های فضایی** طراحی کرده‌اید. طبیعتاً، از آنجا که یک **سفینه کلاس Dreadnought** نوعی سفینه فضایی است، ممکن است انتظار داشته باشید که خط لوله شما بتواند به طور یکپارچه ناوگانی از Dreadnoughtها را دریافت کند. با این حال، سیستم نوع (Type System) کامپیوتر شما را در دروازه متوقف می‌کند و یک خطای امنیتی سختگیرانه نمایش می‌دهد:
*"یک خط لوله از Dreadnoughtها، یک خط لوله از سفینه‌های فضایی نیست."*

این مانع با نام **اینواریانس (Invariance)** شناخته می‌شود. برای عبور از این محدودیت بدون شکستن امنیت نوع یا کرش کردن CPU، باید دو دریچه معماری نصب کنیم: **کوواریانس (Covariance)** و **کنتراواریانس (Contravariance)**. این‌ها سوئیچ‌های جهت‌داری هستند که به کامپایلر دقیقاً می‌گویند که چگونه روابط بین کلاس‌های پایه (مانند `Enemy` یا `Item`) به مجموعه‌ها یا پردازشگرهای آن نوع‌ها (مانند `IEnumerable<Enemy>` یا `IEventHandler<Item>`) منتقل می‌شوند.

### ۲. دانش علوم کامپیوتر: پارادوکس سلسله‌مراتب و اصل جایگزینی لیسکوف

برای درک اینکه چرا این مسئله اتفاق می‌افتد، باید به مبانی تاریخی **برنامه‌نویسی شی‌گرا (OOP)** نگاه کنیم. در اواسط دهه ۱۹۸۰، دانشمندان علوم کامپیوتر یک قاعده بنیادی به نام **اصل جایگزینی لیسکوف (LSP)** را فرموله کردند. این قاعده می‌گوید:

> *اگر $S$ یک زیرنوع از $T$ باشد، اشیاء از نوع $T$ را می‌توان بدون تغییر در هیچ‌یک از ویژگی‌های مطلوب برنامه، با اشیاء از نوع $S$ جایگزین کرد.*

به زبان ساده: اگر یک متد به عنوان آرگومان به یک `Enemy` نیاز دارد، باید اجازه داشته باشید یک `Zombie` را پاس دهید (چون یک زامبی *یک* دشمن است). حلقه بازی نباید منفجر شود چون یک زامبی به جای شکل شبح‌گونه انتزاعی عمومی، پا دارد.

```
       [ کلاس پایه Enemy ]  <--- قرارداد سطح بالا
               ▲
               │  (ارث‌بری از / Is-A)
       [ کلاس زیرمجموعه Zombie ]  <--- پیاده‌سازی مشخص
```

این برای اشیاء تکی عالی کار می‌کند. اگر یک شیء یک کلاس فرزند باشد، می‌تواند کاملاً به عنوان کلاس والد خود ظاهر شود.

پارادوکس زمانی رخ می‌دهد که این اشیاء را درون **جنریک‌ها** بپیچیم. وقتی یک اینترفیس یا کلاس جنریک مانند `IRepository<T>` ایجاد می‌کنید، کامپایلر C# کل ترکیب را به عنوان یک بلوک نوع متمایز در نظر می‌گیرد. به طور پیش‌فرض، حتی اگر `Zombie` از `Enemy` ارث‌بری کند، `IRepository<Zombie>` هیچ ارتباطی با `IRepository<Enemy>` ندارد. آن‌ها به عنوان ساختارهای کاملاً متفاوتی در نظر گرفته می‌شوند، مانند یک باغ سیب و یک پالایشگاه فولاد.

دانشمندان علوم کامپیوتر **تغییردهنده‌های واریانس (Variance Modifiers)** را اختراع کردند تا به طور ایمن این شکاف ساختاری را پر کنند و به سیستم‌ها اجازه دهند بدون کپی کردن منطق‌های تکراری در صدها نوع تخصصی، به شکلی تمیز مقیاس‌پذیر شوند.

### ۳. مشکل اصلی: پرده آهنین جنریک (اینواریانس)

چرا کامپایلر به طور پیش‌فرض اینقدر سختگیر است؟ چرا نمی‌توانیم یک `IRepository<Zombie>` را مستقیماً به متدی که به دنبال یک `IRepository<Enemy>` است پاس دهیم؟

بیایید ببینیم اگر کامپایلر بدون محدودیت اجازه این کار را می‌داد، پشت صحنه چه اتفاقی می‌افتاد. معماری بازی داستانی و خراب زیر را تصور کنید:

```csharp
// تنظیمات: یک سلسله‌مراتب ارث‌بری ساده
public class Enemy { public float hp; }
public class Zombie : Enemy { public void RiseFromGrave() { } }
public class Dragon : Enemy { public void BreatheFire() { } }

// یک اینترفیس واحد ذخیره‌سازی/موجودی ساده
public interface IBox<T>
{
    void PutIn(T item);
    T TakeOut();
}
```

حالا، فرض کنید متدی طراحی کرده‌اید که برای پردازش کانتینرهای دشمن جنریک است:

```csharp
public void MaliciousEngineHack(IBox<Enemy> enemyBox)
{
    // اگر کامپایلر اجازه می‌داد یک IBox<Zombie> به اینجا پاس داده شود...
    // ما می‌توانستیم به طور تصادفی یک DRAGON را در یک جعبه ZOMBIE بیندازیم!
    enemyBox.PutIn(new Dragon()); 
}
```

اگر کامپایلر اجازه می‌داد `IBox<Zombie>` به عنوان `IBox<Enemy>` ظاهر شود، کد بالا کامپایل می‌شد. سپس، در جای دیگری از حلقه بازی، اسکریپتی که فیزیک زامبی‌ها را مدیریت می‌کند، آیتمی را از همان جعبه بیرون می‌کشد، در حالی که انتظار یک `Zombie` را دارد:

```csharp
IBox<Zombie> zombieBox = new ZombieBox();
MaliciousEngineHack(zombieBox); // فرض کنیم این قانونی بود...

Zombie myZombie = zombieBox.TakeOut(); // کرش! این در واقع یک Dragon است!
myZombie.RiseFromGrave(); // CPU را منفجر می‌کند چون Dragons نمی‌توانند RiseFromGrave() را انجام دهند!
```

برای محافظت از RAM و جلوگیری از ناهنجاری‌های اجرایی، کامپایلر یک **پرده آهنین (اینواریانس)** می‌کشد. به شما می‌گوید که حتی اگر نوع‌ها در یک خانواده باشند، کانتینرهای جنریک آن‌ها کاملاً از یکدیگر قفل شده‌اند.

### ۴. راه‌حل‌ها: کوواریانس (`out`) و کنتراواریانس (`in`) ساده‌سازی شده

برای عبور ایمن از پرده آهنین، باید به کامپایلر قول دهیم که کانتینرهای جنریک ما فقط داده‌ها را در **یک جهت** حرکت می‌دهند.

#### جهت A: کوواریانس (`out`) — تولیدکننده خالص

اگر اینترفیس جنریک شما **فقط داده‌ها را به بیرون** به حلقه بازی می‌خواند و هرگز داده‌ای را **به درون** نمی‌پذیرد، کاملاً ایمن است که نوع‌های فرزند را با نوع‌های والد مطابقت دهید. ما این جایگاه را با کلمه کلیدی **`out`** علامت‌گذاری می‌کنیم.

* **طرح:** `IReadStream<out T>`
* **قانون:** می‌توانید یک استریم خاص را به یک استریم عمومی اختصاص دهید. `IReadStream<Enemy> stream = new ReadStream<Zombie>();`
* **تضمین ایمنی:** از آنجا که اینترفیس فقط متدهایی دارد که `T` (خروجی) را برمی‌گردانند، هیچ خطری وجود ندارد که یک اسکریپت خارجی یک نوع هم‌خانواده نامعتبر (مانند `Dragon`) را به مجموعه تزریق کند.

#### جهت B: کنتراواریانس (`in`) — مصرف‌کننده خالص

اگر اینترفیس جنریک شما **فقط داده‌ها را از درون** از حلقه بازی می‌پذیرد و هرگز داده‌ای را به بیرون برنمی‌گرداند، کاملاً ایمن است که نوع‌های والد را با نوع‌های فرزند مطابقت دهید. ما این جایگاه را با کلمه کلیدی **`in`** علامت‌گذاری می‌کنیم.

* **طرح:** `IReceiver<in T>`
* **قانون:** می‌توانید یک مصرف‌کننده عمومی را به یک مصرف‌کننده خاص اختصاص دهید. `IReceiver<Zombie> consumer = new Receiver<Enemy>();`
* **تضمین ایمنی:** از آنجا که یک مصرف‌کننده `Enemy` می‌داند چگونه با *هر* دشمن جنریک (شامل سلامت پایه، موقعیت‌ها و پاکسازی‌ها) کار کند، می‌تواند به طور ایمن یک `Zombie` را دریافت کرده و بدون مواجهه با متغیرهای گمشده یا بلوک‌های حافظه تخصیص نیافته، آن را پردازش کند.

### ۵. نمونه‌های پیاده‌سازی معماری در سطح تولید Unity

بیایید نگاهی بیندازیم که چگونه این را در حلقه‌های بازی Unity پیاده‌سازی کنیم. ما دو الگوی معماری را می‌سازیم که هر مهندس سیستم ارشد از آن‌ها استفاده می‌کند تا کدهای پایه را تمیز، سریع و کاملاً عاری از ساختارهای کلاس تکراری نگه دارد.

#### نمونه ۱: استریم کردن کوواریانت کوئست و لوت (`out`)

فرض کنید در حال برنامه‌نویسی یک موتور لوت RPG جهان‌باز عظیم هستید. شما پاداش‌های لوت تخصصی (`LegendarySword`) دارید که از یک طرح پایه آیتم مشترک (`Item`) ارث‌بری می‌کنند. شما می‌خواهید یک سیستم ردیابی موجودی واحد داشته باشید که همه کشوهای آیتم تخصصی را به یک لایه نمایش UI یکپارچه استریم کند.

```csharp
using System.Collections.Generic;
using UnityEngine;

// --- گام ۱: تعریف درخت نوع ---
public class Item 
{ 
    public string itemName; 
    public int goldValue;
}

public class LegendarySword : Item 
{ 
    public float holyDamageMultiplier; 
}

// --- گام ۲: ایجاد اینترفیس کوواریانت ---
// کلمه کلیدی 'out' به C# می‌گوید که 'T' فقط به عنوان مقادیر بازگشتی از متدها خارج می‌شود.
public interface ILootProducer<out T>
{
    T DispenseReward();
}

// --- گام ۳: پیاده‌سازی کارخانه‌های بتن (Concrete) ---
public class SwordVault : ILootProducer<LegendarySword>
{
    public LegendarySword DispenseReward()
    {
        return new LegendarySword 
        { 
            itemName = "Excalibur Prime", 
            goldValue = 5000, 
            holyDamageMultiplier = 3.5f 
        };
    }
}

// --- گام ۴: کنترل‌کننده کامپوننت Unity ---
public class LootStreamerDisplay : MonoBehaviour
{
    private void Start()
    {
        // بدون کوواریانس (out)، این انتساب یک خطای کامپایل ایجاد می‌کرد!
        // به لطف 'out'، یک اینترفیس تولیدکننده LegendarySwordها به عنوان 
        // یک اینترفیس معتبر تولیدکننده Itemهای پایه شناخته می‌شود.
        ILootProducer<Item> generalLootSource = new SwordVault();
        
        ProcessLootDrop(generalLootSource);
    }

    private void ProcessLootDrop(ILootProducer<Item> lootSource)
    {
        Item droppedItem = lootSource.DispenseReward();
        Debug.Log($"[موتور لوت] تولید شد: {droppedItem.itemName} به ارزش {droppedItem.goldValue} طلا.");
    }
}
```

#### نمونه ۲: سیستم ورودی/تعامل تخصصی کنتراواریانت (`in`)

حالا بیایید معکوس آن را بسازیم. یک سیستم بازی را تصور کنید که Raycastهای تعامل فیزیکی را مدیریت می‌کند. هنگامی که نشانگر بازیکن روی یک موجودیت تعاملی قرار می‌گیرد، سیستم یک پکیج اجرایی را *به درون* یک موتور تعامل پاس می‌دهد.

```csharp
using UnityEngine;

// --- گام ۱: تعریف درخت موجودیت ---
public class InteractableEntity { public string interactionPrompt = "بررسی"; }
public class TreasureChest : InteractableEntity { public bool isLocked = true; }

// --- گام ۲: ایجاد اینترفیس کنتراواریانت ---
// کلمه کلیدی 'in' تضمین می‌کند که 'T' فقط به عنوان پارامتر آرگومان ورودی استفاده می‌شود.
public interface IInteractionHandler<in T>
{
    void ExecuteInteraction(T target);
}

// --- گام ۳: پیاده‌سازی یک مدیریت‌کننده اصلی عمومی ---
public class GlobalInteractionSystem : IInteractionHandler<InteractableEntity>
{
    public void ExecuteInteraction(InteractableEntity target)
    {
        Debug.Log($"[تعامل] اجرای منطق جهانی برای اعلان هدف: {target.interactionPrompt}");
    }
}

// --- گام ۴: حلقه اجرایی Unity ---
public class PlayerInteractionRaycaster : MonoBehaviour
{
    private void Start()
    {
        // کنتراواریانس (in) به ما اجازه می‌دهد از یک مدیریت‌کننده بسیار عمومی 
        // برای انجام نقش یک مصرف‌کننده بسیار خاص استفاده کنیم.
        IInteractionHandler<TreasureChest> chestHandler = new GlobalInteractionSystem();
        
        TreasureChest spawnChest = new TreasureChest { interactionPrompt = "باز کردن صندوق باستانی" };
        
        // بدون نقص اجرا می‌شود!
        chestHandler.ExecuteInteraction(spawnChest);
    }
}
```

### ۶. نکات معماری و کارایی برای توسعه‌دهنده Unity در سطح God-Mode

اگر می‌خواهید ضمن استفاده از واریانس، کنترل مطلق بر بهینه‌سازی حافظه و طرح سخت‌افزاری داشته باشید، این محدودیت‌های معماری پیشرفته را در نظر داشته باشید:

* **محدودیت نوع مرجع (تله نوع مقدار):** کوواریانس و کنتراواریانس **فقط با انواع مرجع (Reference Types)** (کلاس‌ها و اینترفیس‌ها) کار می‌کنند. آن‌ها **با انواع مقدار (Value Types)** (استراکت‌ها، مقادیر اولیه مانند `int`، `float` یا موقعیت‌های سفارشی مانند `Vector3`) کار نمی‌کنند.
* *چرا؟* انواع مقدار فضای بایت خام و متغیری را مستقیماً روی استک اشغال می‌کنند (مثلاً یک `int` ۴ بایت است، در حالی که یک استراکت نقطه سه‌بعدی سفارشی ممکن است ۱۲ یا ۲۴ بایت باشد). اما انواع مرجع، همیشه اشاره‌گرهای ۶۴ بیتی یکسانی هستند که به هیپ اشاره می‌کنند. CPU می‌تواند به راحتی آدرس‌های مرجع کلاس‌های مختلف را جایگزین کند زیرا اندازه آدرس‌های اشاره‌گر همیشه دقیقاً یکسان است. اما نمی‌تواند این کار را با مقادیر با عرض بایت‌های متفاوت بدون شکستن فیزیکی تراز داده‌های استک انجام دهد.

* **دور زدن سربار اینترفیس با دلیگیت‌های توکار:** همیشه نیازی نیست برای بهره‌گیری از واریانس، اینترفیس‌های سفارشی اعلام کنید. C# شامل دلیگیت‌های جنریک توکار مانند `System.Func<out TResult>` (کوواریانت) و `System.Action<in T>` (کنتراواریانت) است. می‌توانید متدهایی را که با نوع‌های تخصصی کار می‌کنند، مستقیماً به استریم‌های رویدادی که نوع‌های عمومی را پردازش می‌کنند پاس دهید، که این کار کدهای کلیشه‌ای (Boilerplate) ساختاری را در سیستم‌های موتور شما حذف می‌کند.

* **حذف زباله‌روبی (GC) با کشینگ استاتیک:** هنگام ترکیب واریانس با رویدادها یا مدیران، از ایجاد اشیاء مدیریت‌کننده `new` در حلقه‌های تکرار شونده فریم مانند `Update()` خودداری کنید. ارجاع به اینترفیس‌های جنریک خود را در شروع سیستم کش (Cache) کنید تا از تولید تخصیص‌های هیپ که باعث می‌شود رفتگر خودکار حافظه (GC) وقفه‌های غیرمنتظره‌ای ایجاد کند، جلوگیری کنید.
