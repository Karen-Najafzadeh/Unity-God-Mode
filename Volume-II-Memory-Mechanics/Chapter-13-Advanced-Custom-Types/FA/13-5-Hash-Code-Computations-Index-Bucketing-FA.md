# محاسبات کد هش (Hash Code) و دسته‌بندی شاخص (Index Bucketing)

### ۱. مقدمه: انبار بی‌نهایت و کلاه مرتب‌سازی جادویی

تصور کنید در حال اجرای یک بازی نقش‌آفرینی آنلاین چندنفره عظیم (MMORPG) در Unity هستید. بازی شما شامل موجودی صدها هزار آیتم متمایز، یا پایگاه داده‌ای است که نام کاربری میلیون‌ها بازیکن جهانی را ردیابی می‌کند.

وقتی بازیکنی به دنبال آیتمی در مخزن خود می‌گردد—مثلاً یک شمشیر افسانه‌ای به نام `"ShadowSlayer_99"`—کامپیوتر چگونه آن را پیدا می‌کند؟

اگر کامپیوتر آیتم‌ها را در یک لیست سنتی ذخیره کند، باید از آیتم شماره ۱ شروع کند، بررسی کند که آیا `"ShadowSlayer_99"` است یا خیر، به آیتم شماره ۲ برود، دوباره بررسی کند و همین‌طور تا انتهای لیست (مثلاً ۵۰۰,۰۰۰) اسکن کند. اگر آیتم وجود نداشته باشد، یا در انتهای لیست باشد، کامپیوتر میلی‌ثانیه‌های ارزشمندی را در کل انبار هدر داده است. در یک بازی سریع، انجام این کار در هر فریم باعث لکنت و یخ‌زدگی بازی می‌شود.

برای حل این مشکل، دانشمندان علوم کامپیوتر ترکیبی از دو مکانیزم معماری درخشان را اختراع کردند: **محاسبات کد هش (Hash Code Computations)** و **دسته‌بندی شاخص (Index Bucketing)**. این دو با هم، پایه و اساس ساختار داده‌ای به نام **جدول هش (Hash Table)** (یا `Dictionary` در C#) را تشکیل می‌دهند.

به‌جای جستجو در یک لیست عظیم، کامپیوتر نام `"ShadowSlayer_99"` را به یک ماشین ریاضی به نام **تابع هش (Hash Function)** می‌دهد. این تابع مانند یک "کلاه مرتب‌سازی جادویی" عمل می‌کند. این تابع به طور آنی آن رشته متنی را به یک عدد خاص و قابل پیش‌بینی تبدیل می‌کند (کد هش یا **Hash Code**). سیستم دیگری آن عدد را می‌گیرد و مستقیماً آن را به یک اتاق یا کابینت خاص نگاشت می‌کند (**Index Bucket**).

دفعه بعد که به دنبال `"ShadowSlayer_99"` می‌گردید، انبار را اسکن نمی‌کنید. نام را دوباره به کلاه مرتب‌سازی می‌دهید، شماره کابینت دقیق را به شما می‌دهد و شما با یک حرکت مستقیم به سراغ آن می‌روید.

### ۲. دانش علوم کامپیوتر: طرح اداره پست و اصل لانه کبوتر

برای درک منشأ این مفهوم، باید به روزهای اولیه مرتب‌سازی مکانیکی و شبکه‌های پستی نگاه کنیم.

#### افسانه جدول آدرس مستقیم (Direct Address Table)

در دنیای ایده‌آل، اگر موجودی ۱۰۰ آیتم شماره‌گذاری شده از ۰ تا ۹۹ داشتید، می‌توانستید دقیقاً ۱۰۰ اسلات حافظه در RAM کامپیوتر خود اختصاص دهید. اگر آیتم شماره ۴۵ را می‌خواستید، مستقیماً به اسلات ایندکس ۴۵ می‌رفتید. این را **جدول آدرس مستقیم** می‌نامند. این فوق‌العاده سریع است—با زمان ثابت $O(1)، یعنی فرقی نمی‌کند ۵ آیتم داشته باشید یا ۵ میلیون، جستجو همیشه همان کسری از میکروثانیه طول می‌کشد.

اما وقتی چیزهایی که می‌خواهید ذخیره کنید اعداد ساده و متوالی نیستند چه اتفاقی می‌افتد؟ اگر کلیدهای شما رشته‌های متنی (مانند نام بازیکنان یا توضیحات آیتم) یا ترکیبات عظیمی از بردارها باشند چه؟

یک رشته متنی می‌تواند بی‌نهایت طولانی باشد. اگر سعی کنید برای هر ترکیب ممکنی از کاراکترهایی که بازیکن ممکن است برای نام کاربری تایپ کند، یک اسلات حافظه خاص از پیش اختصاص دهید، به اسلات‌های حافظه بیشتری نسبت به کل اتم‌های موجود در جهان قابل مشاهده نیاز خواهید داشت. RAM فیزیکی کامپیوتر شما بلافاصله آتش می‌گرفت.

#### اصل لانه کبوتر (Pigeonhole Principle) و برخوردها (Collisions)

این ما را به یک قاعده بنیادی ریاضی به نام **اصل لانه کبوتر** می‌رساند. این اصل می‌گوید اگر ۱۰ کبوتر دارید اما فقط ۹ لانه برای قرار دادن آن‌ها دارید، حداقل یکی از آن لانه‌ها *باید* حاوی بیش از یک کبوتر باشد.

در محاسبات، ما جهان بی‌نهایتی از ورودی‌های داده ممکن (نام کاربری، رشته‌ها، آیتم‌ها) داریم اما مقدار محدودی از سطل‌های (Buckets) حافظه فیزیکی. چالش اصلی علوم کامپیوتر فقط تبدیل متن به اعداد نبود؛ بلکه فهمیدن این بود که چگونه جهان بی‌نهایتی از احتمالات را در یک کابینت بایگانی کوچک و بسیار سازمان‌یافته فشرده کنیم، بدون اینکه داده‌ها روی هم بیفتند و خود را خراب کنند. این مشکل همپوشانی، **برخورد (Collision)** نامیده می‌شود.

### ۳. معماری اصلی: مشکل و راه‌حل‌ها

بیایید دقیقاً بررسی کنیم که چگونه این دو مکانیزم قدم‌به‌قدم برای حل مشکل حافظه بی‌نهایت کار می‌کنند.

#### گام ۱: محاسبه کد هش (فشرده‌سازی هویت)

مشکل اصلی این است که کامپیوتر فقط می‌تواند آدرس‌های حافظه را با استفاده از اعداد جستجو کند، اما انسان‌ها از اشیاء و متن‌های پیچیده استفاده می‌کنند.

یک **تابع هش** یک خط لوله ریاضی یک‌طرفه است. این تابع ورودی با هر اندازه‌ای (یک کاراکتر، یک فایل متنی عظیم، یا یک شیء بازی کامل) را می‌گیرد و بایت‌های داخلی آن را پردازش می‌کند تا یک عدد صحیح ۳۲ بیتی با اندازه ثابت (کد هش یا **Hash Code**) خروجی دهد.

در C#، هر شیء از یک تابع داخلی به نام `.GetHashCode()` ارث‌بری می‌کند.

یک تابع هش باکیفیت باید از سه قانون سختگیرانه پیروی کند:

۱. **قطعی (Deterministic):** اگر رشته `"Excalibur"` را به آن بدهید، باید هر بار که در طول اجرای آن برنامه فراخوانی می‌شود، دقیقاً همان عدد را برگرداند.
۲. **توزیع یکنواخت (Uniform Distribution):** تغییر حتی یک حرف کوچک (مثلاً از `"Excalibur"` به `"excalibur"`) باید منجر به یک کد هش بسیار متفاوت شود. این کار آیتم‌ها را به طور مساوی در حافظه پخش می‌کند نه اینکه آن‌ها را در یک نقطه جمع کند.
۳. **ارزیابی سریع (Fast Evaluation):** ریاضیات باید آنقدر ساده باشد که CPU بتواند در چند چرخه ساعت آن را محاسبه کند.

#### گام ۲: دسته‌بندی شاخص (Index Bucketing - قرار دادن در واقعیت)

عالی! ما `"ShadowSlayer_99"` را به یک کد هش تبدیل کردیم، فرض کنید عدد `2,147,483,647`. اما نمی‌توانیم آرایه‌ای با بیش از ۲ میلیارد اسلات فقط برای ذخیره چند صد آیتم ایجاد کنیم؛ این کار مقدار عظیمی از RAM را هدر می‌دهد.

اینجاست که **دسته‌بندی شاخص (Index Bucketing)** وارد عمل می‌شود. وقتی یک `Dictionary` ایجاد می‌کنید، کامپیوتر یک آرایه داخلی کوچک و قابل مدیریت با اندازه خاص اختصاص می‌دهد، فرض کنید **۱۰ اسلات** (با ایندکس ۰ تا ۹). این اسلات‌ها **سطل‌های (Buckets)** ما هستند.

برای تبدیل کد هش عظیم ۲ میلیاردی به شماره سطل بین ۰ تا ۹، از یک عمل ریاضی به نام **عملگر باقی‌مانده (Modulo Operator %)** استفاده می‌کنیم.

$$\text{شاخص سطل} = |\text{کد هش}| \pmod{\text{تعداد سطل‌ها}}$$

اگر کد هش ما `123456` باشد و `10` سطل داشته باشیم:

$$123456 \div 10 = 12345 \text{ با باقی‌مانده } 6$$

آیتم به طور مرتب در **سطل شماره ۶** قرار می‌گیرد.

#### مقابله با اجتناب‌ناپذیری: حل برخورد (Collision Resolution)

به دلیل اصل لانه کبوتر، دو نام کاملاً متفاوت در نهایت دارای کدهای هشی خواهند بود که به همان سطل اشاره می‌کنند. اگر `"ShadowSlayer_99"` و `"DragonRider_12"` هر دو به سطل شماره ۶ نگاشت شوند، سیستم چگونه آن را مدیریت می‌کند؟

دو روش کلاسیک برای حل این مشکل وجود دارد:

۱. **زنجیره‌سازی (Chaining - لیست‌های پیوندی):** هر سطل فقط یک اسلات نیست؛ بلکه یک سطل کوچک است که می‌تواند زنجیره‌ای از آیتم‌ها را نگه دارد. اگر برخوردی رخ دهد، آیتم جدید پشت سر آیتم اول در همان سطل پیوند می‌خورد. هنگام جستجو، کامپیوتر مستقیماً به سطل شماره ۶ می‌رود، سپس فقط زنجیره کوچک داخل آن سطل را اسکن می‌کند تا مطابقت دقیق را پیدا کند.
۲. **آدرس‌دهی باز (Open Addressing - کاوش خطی):** اگر سطل شماره ۶ پر باشد، کامپیوتر به سطل شماره ۷ نگاه می‌کند. اگر آن هم پر باشد، به سطل شماره ۸ نگاه می‌کند تا یک اسلات خالی پیدا کند.

### ۴. پیاده‌سازی کامل کد

بیایید از جادوی سطح بالای Unity فاصله بگیریم و جدول هش سفارشی، بدون تخصیص حافظه (Zero-Allocation) و با کارایی بالای خود را از صفر بسازیم. این معماری سفارشی نشان می‌دهد که چگونه یک رشته متنی هش می‌شود، از طریق عملیات Modulo فشرده می‌شود و با استفاده از **زنجیره‌سازی (Chaining)** برای مدیریت برخوردها، در یک شبکه دسته‌بندی شاخص قرار می‌گیرد.

```csharp
using System;
using System.Collections.Generic;
using UnityEngine;

public class HashArchitectureDemo : MonoBehaviour
{
    void Start()
    {
        // ۱. مقداردهی اولیه مخزن ذخیره‌سازی سفارشی با کارایی بالا با ۵ سطل
        CustomDictionary itemVault = new CustomDictionary(5);

        Debug.Log("=== پر کردن مخزن معماری ===");
        itemVault.Insert("ShadowSlayer_99", "Legendary Abyssal Greatsword +5");
        itemVault.Insert("DragonRider_12", "Mythic Wyvern Scale Chestplate");
        itemVault.Insert("Excalibur", "Holy Radiant Blade of Kings");
        itemVault.Insert("RustySpoon", "Common Trash Weapon");
        itemVault.Insert("PhoenixBow", "Ancient Firestrike Longbow");

        Debug.Log("\n=== بازیابی داده‌ها از سطل‌ها ===");
        itemVault.Search("Excalibur");
        itemVault.Search("ShadowSlayer_99");
        itemVault.Search("MegalodonTooth"); // آیتمی که وجود ندارد

        Debug.Log("\n=== مشاهده بصری سطل‌های داخلی ===");
        itemVault.PrintStorageLayout();
    }
}

/// <summary>
/// یک گره که یک آیتم فیزیکی ذخیره شده در زنجیره سطل را نشان می‌دهد.
/// </summary>
public class HashNode
{
    public string Key;         // شناسنامه قابل خواندن برای انسان (مثلاً نام بازیکن / شناسه آیتم)
    public string Value;       // داده‌های اصلی (مثلاً آمار آیتم / توضیحات)
    public int ComputedHash;   // کد هش محاسبه شده برای این کلید
    public HashNode Next;      // اشاره‌گر پیوند به آیتم بعدی در زنجیره این سطل (زنجیره‌سازی)

    public HashNode(string key, string value, int computedHash)
    {
        Key = key;
        Value = value;
        ComputedHash = computedHash;
        Next = null;
    }
}

/// <summary>
/// جدول هش مهندسی شده سفارشی ما که محاسبات کد هش و دسته‌بندی شاخص را به نمایش می‌گذارد.
/// </summary>
public class CustomDictionary
{
    private HashNode[] _buckets; // آرایه ایندکس اصلی که به عنوان لیست سطل‌های ما عمل می‌کند
    private int _bucketCount;    // اندازه کل کابینت بایگانی آرایه فیزیکی ما

    public CustomDictionary(int totalBuckets)
    {
        _bucketCount = totalBuckets;
        _buckets = new HashNode[_bucketCount]; // آرایه ثابت سطل‌ها را از پیش اختصاص دهید
    }

    /// <summary>
    /// تابع هش قطعی سفارشی (نوع الگوریتم DJB2).
    /// هر رشته متنی پیچیده را به یک عدد صحیح ۳۲ بیتی با توزیع بالا تبدیل می‌کند.
    /// </summary>
    private int ComputeDeterministicHash(string key)
    {
        unchecked // اجازه سرریز حسابی را با خیال راحت بدهید تا کارایی حفظ شود
        {
            int hash = 5381; // بذر شروع بهینه تاریخی
            for (int i = 0; i < key.Length; i++)
            {
                // بیت‌ها را ۵ واحد به چپ شیفت دهید و مقدار ASCII کاراکتر را اضافه کنید
                hash = ((hash << 5) + hash) + key[i];
            }
            return hash;
        }
    }

    /// <summary>
    /// درج یک آیتم در معماری سطل ما.
    /// </summary>
    public void Insert(string key, string value)
    {
        // گام ۱: محاسبه کد هش هویت ۳۲ بیتی خام
        int rawHash = ComputeDeterministicHash(key);

        // گام ۲: فشرده‌سازی هش به یک سطل شاخص معتبر با استفاده از ریاضیات Modulo
        // از Math.Abs استفاده می‌کنیم تا اگر هش به عدد منفی سرریز شد، شاخص مثبت بماند
        int bucketIndex = Math.Abs(rawHash) % _bucketCount;

        Debug.Log($"[درج] کلید: '{key}' -> هش خام: {rawHash} -> نگاشت شده به شاخص سطل: #{bucketIndex}");

        // گام ۳: مدیریت درج ساختاری یا زنجیره‌سازی برخورد
        if (_buckets[bucketIndex] == null)
        {
            // کابینت سطل کاملاً خالی است! گره را به عنوان پایه قرار دهید.
            _buckets[bucketIndex] = new HashNode(key, value, rawHash);
        }
        else
        {
            // برخورد شناسایی شد! سطل قبلاً اشغال شده است. 
            // آیتم جدید را به سر زنجیره اضافه می‌کنیم.
            Debug.LogWarning($"⚠️ [برخورد] سطل #{bucketIndex} قبلاً اشغال شده است! زنجیره‌سازی '{key}' پشت آیتم‌های موجود.");
            
            HashNode newNode = new HashNode(key, value, rawHash);
            newNode.Next = _buckets[bucketIndex]; // زنجیره قدیمی را پشت گره جدید ما پیوند دهید
            _buckets[bucketIndex] = newNode;      // این گره جدید را ورودی اصلی جلوی سطل کنید
        }
    }

    /// <summary>
    /// جستجوی آیتم مستقیماً از طریق شاخص سطل آن، با دور زدن هرگونه حلقه آرایه کامل.
    /// </summary>
    public void Search(string key)
    {
        int rawHash = ComputeDeterministicHash(key);
        int bucketIndex = Math.Abs(rawHash) % _bucketCount;

        Debug.Log($"[جستجو] شکار برای '{key}'. شاخص سطل هدف فوراً تعیین شد: #{bucketIndex}");

        HashNode currentElement = _buckets[bucketIndex];

        // فقط زنجیره باریک داخل این کابینت سطل خاص را پیمایش کنید
        while (currentElement != null)
        {
            if (currentElement.Key == key)
            {
                Debug.Log($"🎉 [پیدا شد] موفقیت! مطابقت در سطل #{bucketIndex} کشف شد. داده: {currentElement.Value}");
                return;
            }
            currentElement = currentElement.Next; // در زنجیره پیوندی پایین بروید
        }

        Debug.LogError($"❌ [پیدا نشد] کلید '{key}' در سطل #{bucketIndex} وجود ندارد.");
    }

    /// <summary>
    /// متد کمکی برای نمایش بصری محتویات کابینت‌های انبار ما.
    /// </summary>
    public void PrintStorageLayout()
    {
        for (int i = 0; i < _bucketCount; i++)
        {
            string bucketTrace = $"سطل #{i}: ";
            HashNode node = _buckets[i];
            
            if (node == null)
            {
                bucketTrace += "[قاب خالی]";
            }
            
            while (node != null)
            {
                bucketTrace += $"({node.Key} => {node.Value}) -> ";
                node = node.Next;
            }
            Debug.Log(bucketTrace);
        }
    }
}
```

### ۵. خلاصه تغییر معماری

| بعد عملکردی | طرح لیست ترتیبی سنتی | معماری محاسبه هش و دسته‌بندی شاخص |
| --- | --- | --- |
| **پیچیدگی زمان جستجو** | $O(N)$ زمان خطی: باید هر آیتم را تک‌تک از ابتدا بررسی کنید. | $O(1)$ زمان ثابت: شاخص را به صورت ریاضی محاسبه کرده و فوراً به آیتم هدف می‌پرد. |
| **مکانیک جستجو** | مقادیر شیء خام یا متن‌های رشته‌ای را به صورت ترتیبی در حافظه مقایسه می‌کند. | یک کد هش ساختاری ۳۲ بیتی سبک محاسبه کرده و آن را از طریق عملیات modulo در اندازه آرایه ثابت قرار می‌دهد. |
| **گلوگاه مقیاس‌پذیری** | با مقیاس‌گذاری داده‌های بازی به میلیون‌ها آیتم، جستجوها کندتر می‌شوند و افت فریم‌ها تشدید می‌شوند. | عملکرد بدون توجه به اندازه، سریع و صاعقه‌آسا باقی می‌ماند، به شرطی که اندازه آرایه شما به طور یکنواخت گسترش یابد تا برخوردهای داخلی محدود شود. |
| **جداسازی داده‌ها** | همه قطعات داده در یک خط بلوک مداوم عظیم مشترک هستند. | داده‌ها به صورت ریاضی طبقه‌بندی شده و در مسیرهای ایندکس تمیز و ایزوله گروه‌بندی می‌شوند (**سطل‌ها**). |
