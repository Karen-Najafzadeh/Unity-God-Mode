# معماری چیدمان داده‌ها (SoA در مقابل AoS)

---


### ۱. مقدمه: فروشگاه خواربارفروشی و خط مونتاژ

تصور کنید که یک انبار خواربارفروشی فانتزی عظیم و فوق‌سریع را اداره می‌کنید. وظیفه شما برآورده کردن سفارشات هزاران مشتری است که همگی دقیقاً یک چیز را در یک زمان می‌خواهند: یک عدد سیب قرمز تازه و فرد.

حالا، فکر کنید که چگونه این انبار می‌تواند سازماندهی شود:

* **گزینه الف (چیدمان سبد هدیه):** شما هزاران سبد هدیه حصیری زیبا بسته‌بندی می‌کنید. داخل هر سبد جداگانه، یک سیب، یک بطری شراب، یک قالب پنیر، یک شمع تزئینی و یک دستمال سفره چاپی سفارشی قرار می‌دهید. وقتی مشتری پشت پیشخوان شما می‌ایستد و فریاد می‌زند: *"یک سیب به من بده!"*، کارگر شما باید به قسمت عقب بدود، یک سبد حصیری عظیم و سنگین را روی شانه خود بیندازد، آن را تا پیشخوان حمل کند، بازش کند، سیب را بردارد تا به مشتری بدهد و سپس پنیر، شمع و شراب سنگین را تمام راه به قفسه برگرداند.
* **گزینه ب (چیدمان پالت):** شما سبدها را حذف می‌کنید. در عوض، سینی‌های مونتاژ صنعتی عظیم و طولانی را در یک صف قرار می‌دهید. یک سینی حاوی هیچ‌چیز جز ۱۰,۰۰۰ سیب نیست که کنار هم چیده شده‌اند. سینی بعدی حاوی هیچ‌چیز جز ۱۰,۰۰۰ قالب پنیر نیست. وقتی مشتری فریاد می‌زند: *"یک سیب به من بده!"*، کارگر شما قدم به سمت سینی سیب می‌گذارد و یک ردیف از ۱۰ سیب متوالی را فوراً به پایین خط می‌لغزاند.

در دنیای نرم‌افزارهای با کارایی بالا و معماری موتور بازی، **گزینه الف "آرایه ساختارها" (AoS)** و **گزینه ب "ساختار آرایه‌ها" (SoA)** نامیده می‌شود. این انتخاب چیدمان تعیین می‌کند که آیا پردازنده کامپیوتر شما مانند یک خط مونتاژ رباتیک سریع برق‌آسا عمل می‌کند یا مانند یک کارگر خسته انبار که سبدهای حصیری سنگین و ناخواسته را در حافظه می‌کشد.

### ۲. دانش کامپیوتر: افسانه حافظه آنی و خط کش حافظه پنهان (Cache Line) CPU

برای درک اینکه چرا این انتخاب مسئله مرگ و زندگی برای نرخ فریم بازی شماست، باید یک افسانه رایج را از بین ببریم: **توهم اینکه حافظه کامپیوتر (RAM) به طور آنی قابل دسترسی است.**

وقتی کد اولیه می‌نویسید، به شما آموزش داده می‌شود که RAM مانند یک شبکه جادویی از یادداشت‌های چسبان (Post-it) است و کامپیوتر می‌تواند هر یادداشت چسبان را در هر مختصاتی در زمان صفر بگیرد. **این یک دروغ کامل است.** در واقعیت، واحد پردازش مرکزی (CPU) کامپیوتر شما یک ماشین اسپرت فوق‌پیشرفته است که می‌تواند با سرعت ۳۰۰ مایل در ساعت حرکت کند، در حالی که RAM شما یک کامیون باربری بسیار کند است که با سرعت ۵ مایل در ساعت حرکت می‌کند. اگر CPU مجبور باشد متوقف شود و منتظر بماند تا RAM یک قطعه داده را تحویل دهد، کاملاً بیکار می‌ماند و چرخ‌هایش را برای صدها چرخه ساعت هرز می‌گرداند. این فاجعه **Cache Miss** نامیده می‌شود.

برای جلوگیری از این اتفاق، مهندسان کامپیوتر یک سالن VIP با سرعت بالا در داخل CPU طراحی کردند که **CPU Cache** نامیده می‌شود.

هر زمان که CPU شما یک قطعه داده را می‌خواهد - مثلاً موقعیت افقی `X` یک هیولای دشمن - فقط آن عدد را نمی‌گیرد. بلکه یک کانتینر حمل و نقل به انبار RAM می‌فرستد. این کانتینر یک بلوک ثابت و پیوسته ۶۴ بایتی از حافظه را می‌گیرد که شامل عدد هدف شما و *هر داده‌ای است که مستقیماً در کنار آن قرار دارد*. این کانتینر حمل و نقل ۶۴ بایتی **Cache Line** نامیده می‌شود.

CPU کل این خط کش حافظه پنهان را در سالن محلی فوق‌سریع خود بارگذاری می‌کند. اگر دستور بعدی به داده‌ای نیاز داشته باشد که در همان کانتینر حمل شده است، امتیاز **Cache Hit** را کسب می‌کند و داده را فوراً بدون انتظار برای RAM کند می‌خواند.

قانون اصلی بهینه‌سازی سطح سخت‌افزار ساده است: **اگر کانتینرهای حمل و نقل (Cache Lines) خود را با ۱۰۰٪ داده‌های مفید پر کنید، بازی شما مانند یک خدا اجرا می‌شود. اگر آن‌ها را با اقلام بی‌فایده پر کنید، بازی شما لکنت می‌گیرد.**

### ۳. مشکل اصلی: آرایه ساختارها (AoS) و قاتل عملکرد "گلوله سرگردان"

به‌طور سنتی، مهندسی نرم‌افزار به‌شدت به **برنامه‌نویسی شیءگرا (OOP)** متکی است. در OOP، دوست داریم فکر کنیم که جهان مانند زندگی واقعی است: یک شیء باید شامل همه چیزهایی باشد که متعلق به آن است.

اگر ما در حال ساخت یک شبیه‌ساز نبرد حماسی در یونیتی با ۲۰,۰۰۰ سرباز فعال باشیم، به‌طور طبیعی یک ساختار نقشه (struct) به نام `Soldier` تعریف می‌کنیم:

```csharp
public struct Soldier
{
    public Vector3 Position;     // ۱۲ بایت
    public Vector3 Velocity;     // ۱۲ بایت
    public int Health;           // ۴ بایت
    public int ShieldPoints;     // ۴ بایت
    public string SoldierName;   // ۸ بایت (اشاره‌گر)
    public int TotalKills;       // ۴ بایت
    public bool IsInCombat;      // ۱ بایت
}
```

سپس، همه سربازان خود را در یک لیست یا آرایه عظیم ذخیره می‌کنیم: `Soldier[] army;`. این چیدمان یک **آرایه ساختارها (AoS)** است.

#### تحلیل گلوگاه

حالا، بیایید ببینیم وقتی یک سیستم ردیابی حرکت در هر فریم اجرا می‌شود تا مکان‌های فیزیکی این ۲۰,۰۰۰ سرباز را به‌روزرسانی کند، در پشت صحنه چه اتفاقی می‌افتد. سیستم فقط می‌خواهد `Position` را بخواند و آن را با استفاده از `Velocity` اصلاح کند. اصلاً به نام‌ها، آمار سپر، تعداد کشتارها یا پرچم‌های نبرد اهمیت نمی‌دهد.

۱. CPU به `army[0]` نگاه می‌کند و `Position` آن را درخواست می‌کند.
۲. سخت‌افزار یک کانتینر Cache Line ۶۴ بایتی را به سمت RAM می‌فرستد.
۳. کانتینر روی `army[0]` قفل می‌شود. از آنجایی که داده‌ها به صورت متوالی در داخل شیء ذخیره می‌شوند، `Position` را بارگذاری می‌کند، اما *همچنین* به‌طور تصادفی `Health`، `ShieldPoints`، `SoldierName`، `TotalKills` و `IsInCombat` را فقط برای پر کردن جعبه ۶۴ بایتی برمی‌دارد.
۴. CPU محاسبات فیزیک را پردازش می‌کند. حالا، به `army[1]` حرکت می‌کند.
۵. از آنجایی که کانتینر اول با آن داده‌های اضافی (شمع، پنیر و شراب) پر شده بود، موقعیت `army[1]` نتوانست در همان کانتینر سالن VIP جا بگیرد.
۶. **بوم. Cache Miss.** CPU باید متوقف شود، حلقه فریم شما را مکث کند و منتظر بماند تا یک کانتینر کاملاً جدید به RAM سفر کند تا `army[1]` را بگیرد.

در معماری AoS، سخت‌افزار شما تا ۸۰٪ از زمان اجرای خود را صرف کشیدن وزن مرده در سراسر باس‌های سیستم می‌کند. با افزایش تعداد موجودیت‌ها از چند هزار، زمان فریم شما افزایش می‌یابد و باعث لکنت‌های شدید و ناگهانی می‌شود.

### ۴. راه نجات معماری: ساختار آرایه‌ها (SoA) و طراحی داده‌محور

برای رفع این نقص ساختاری، باید کل مدل ذهنی خود را زیر و رو کنیم. ما این ایده را که یک "سرباز" باید یک بسته محلی واحد باشد، رها می‌کنیم. در عوض، سرباز را از هم می‌پاشیم و داده‌ها را بر اساس **نحوه پردازش آن‌ها توسط سیستم‌ها** گروه‌بندی می‌کنیم.

در معماری **ساختار آرایه‌ها (SoA)**، مدیر بازی ما صاحب آرایه‌ای از اشیاء کامل سرباز نیست. در عوض، او صاحب یک کانتینر ساختاری اصلی واحد است که آرایه‌های متمایز، بسیار متمرکز و موازی از داده‌ها را مدیریت می‌کند:

[تصویر مقایسه چیدمان حافظه AoS و SoA]

```csharp
// یک موجودیت اکنون به سادگی یک ID عدد صحیح است (مثلاً سرباز شماره ۴۲)
public class SovereignSoldierSimulationManager
{
    public Vector3[] Positions;     // چیدمان: [Pos0, Pos1, Pos2, Pos3, Pos4...]
    public Vector3[] Velocities;    // چیدمان: [Vel0, Vel1, Vel2, Vel3, Vel4...]
    public int[]     HealthPools;   // چیدمان: [HP0,  HP1,  HP2,  HP3,  HP4...]
}
```

#### چرا این در لایه سخت‌افزار کار می‌کند

بیایید سیستم ردیابی حرکت خود را دوباره روی این چیدمان جدید اجرا کنیم:

۱. CPU می‌خواهد حرکت ارتش شما را به‌روزرسانی کند. او `Positions[0]` را درخواست می‌کند.
۲. کانتینر سخت‌افزار به RAM سفر می‌کند و یک Cache Line ۶۴ بایتی می‌گیرد.
۳. از آنجایی که آرایه `Positions` حاوی **هیچ‌چیزی جز مختصات خام Vector3 نیست که محکم کنار هم چیده شده‌اند**، آن کانتینر ۶۴ بایتی واحد، `Positions[0]`، `Positions[1]`، `Positions[2]`، `Positions[3]` و `Positions[4]` را به‌طور همزمان می‌گیرد!
۴. CPU این جعبه بسته‌بندی شده واحد را در سالن حافظه پنهان L1 خود بارگذاری می‌کند. او سرباز ۰ را فوراً به‌روزرسانی می‌کند. سپس، سرباز ۱ را بررسی می‌کند. **Cache Hit!** او قبلاً در سالن است. سرباز ۲؟ **Cache Hit!** سرباز ۳؟ **Cache Hit!**
۵. CPU هزاران عنصر را به صورت متوالی بدون توقف حتی یک بار جریان می‌دهد.

با تغییر ساختار چیدمان داده‌های خود از AoS به SoA، خط لوله حافظه خود را از یک اتاق زیر شیروانی به‌هم‌ریخته به یک جریان داده با قدرت بالا و بدون تخصیص تبدیل کرده‌ایم.

### ۵. مثال‌های جامع کد: رویارویی معماری

بیایید یک مقایسه در سطح تولید در یونیتی انجام دهیم. ما یک چارچوب مبارزه AoS ساده‌لوحانه و قاتل عملکرد می‌سازیم و سپس آن را به یک طراحی داده‌محور SoA بهینه‌شده و حاکم بازنویسی می‌کنیم.

#### ❌ معماری AoS ساده‌لوحانه (نابودکننده کش)

این اسکریپت یک ساختار شیءگرای کلاسیک را مدل می‌کند که در آن یک حلقه ذرات یا موجودیت‌های حاوی کیسه‌ای مخلوط از بارهای داده را به‌روزرسانی می‌کند.

```csharp
using UnityEngine;

public class NaiveAoSPhysicsSimulator : MonoBehaviour
{
    // سبد حصیری حاوی تعاریف داده مخلوط
    public struct CosmicEntity
    {
        public Vector3 Position;
        public Vector3 Velocity;
        public int BaseHealth;
        public int ManaPoints;
        public string EntityIdentifier;
        public Matrix4x4 TransformationMatrix; // بار اضافی سنگین برای یک عبور موقعیت ساده!
        public bool IsActive;
    }

    [SerializeField] private int entityCount = 20000;
    private CosmicEntity[] entityArray;

    private void Start()
    {
        entityArray = new CosmicEntity[entityCount];
        for (int i = 0; i < entityCount; i++)
        {
            entityArray[i].Velocity = Random.insideUnitSphere * 2f;
            entityArray[i].IsActive = true;
        }
    }

    private void Update()
    {
        // قاتل عملکرد: جریان‌سازی از طریق گام‌های بزرگ اشیاء برای به‌روزرسانی ویژگی‌های کوچک
        for (int i = 0; i < entityArray.Length; i++)
        {
            if (entityArray[i].IsActive)
            {
                entityArray[i].Position += entityArray[i].Velocity * Time.deltaTime;
            }
        }
    }
}
```

#### 👑 معماری SoA حاکم (جریان داده دوستدار کش)

این معماری بازنویسی‌شده، فیلدهای داده را به مسیرهای حافظه خالص و به‌طور محکم گروه‌بندی‌شده تقسیم می‌کند. توجه کنید که چگونه حلقه به‌روزرسانی آرایه‌های متوالی را بدون زحمت پردازش می‌کند، هیچ زباله‌ای تولید نمی‌کند و به حداکثر استفاده از سخت‌افزار می‌رسد.

```csharp
using UnityEngine;

public class SovereignSoAPhysicsSimulator : MonoBehaviour
{
    [SerializeField] private int entityCount = 20000;

    // آرایه‌های ذخیره‌سازی موازی (چیدمان پالت)
    private Vector3[] positions;
    private Vector3[] velocities;
    private bool[]    activeStates;

    // داده‌های سنگین کمکی کاملاً در مسیر خاص خود زندگی می‌کنند
    private Matrix4x4[] transformationMatrices;
    private int[]       healthPools;

    private void Start()
    {
        positions = new Vector3[entityCount];
        velocities = new Vector3[entityCount];
        activeStates = new bool[entityCount];
        transformationMatrices = new Matrix4x4[entityCount];
        healthPools = new int[entityCount];

        for (int i = 0; i < entityCount; i++)
        {
            velocities[i] = Random.insideUnitSphere * 2f;
            activeStates[i] = true;
        }
    }

    private void Update()
    {
        float deltaTime = Time.deltaTime;

        // قهرمان سخت‌افزار با کارایی بالا: ۱۰۰٪ کارایی خط کش!
        // هیچ ماتریس سنگین یا عدد صحیح سلامت هرگز مسیرهای پردازنده را در طول این اجرا آلوده نمی‌کند.
        for (int i = 0; i < positions.Length; i++)
        {
            if (activeStates[i])
            {
                positions[i].x += velocities[i].x * deltaTime;
                positions[i].y += velocities[i].y * deltaTime;
                positions[i].z += velocities[i].z * deltaTime;
            }
        }
    }
}
```

### ۶. خلاصه تغییرات معماری

| بعد عملکرد | آرایه ساختارها (AoS) | ساختار آرایه‌ها (SoA) |
| --- | --- | --- |
| **پارادایم مدل ذهنی** | **طراحی شیءگرا:** داده‌ها حول "موجودیت‌ها" یا "چیزها"ی مفهومی گروه‌بندی می‌شوند. | **طراحی داده‌محور:** داده‌ها مستقیماً بر اساس زمینه استفاده و برنامه‌های پردازش سیستم گروه‌بندی می‌شوند. |
| **کارایی خط کش** | **بسیار پایین:** خطوط کش متغیرهای داده همراه ناخواسته را جذب می‌کنند و سالن CPU را آلوده می‌کنند. | **حداکثر (۱۰۰٪):** خطوط کش کاملاً متراکم هستند و با ساختارهای داده متوالی و یکسان بسته‌بندی شده‌اند. |
| **پروفایل مقیاس‌پذیری** | به‌سرعت کاهش می‌یابد. افزودن متغیرهای جدید به یک ساختار، به‌طور خودکار حلقه‌های موجود را جریمه می‌کند. | به‌راحتی صاف می‌شود. می‌توانید آرایه‌های داده جدید را به شبیه‌سازی اضافه کنید بدون اینکه سیستم‌های قدیمی را کند کنید. |
| **هم‌ترازی سخت‌افزار** | پردازنده را مجبور می‌کند در فواصل حافظه ناهموار بپرد (**رفتار تعقیب اشاره‌گر**). | به‌طور تمیز در مسیرهای جریان مستقیم برش می‌خورد و CPU را برای عملیات برداری موازی پیشرفته آماده می‌کند. |
