# اشتراک رویدادهای چرخه حیات در مرزهای زمان اجرا (Runtime Boundaries)

---

#### ۱. مقدمه: کنترل مرزی هسته موتور

به یکی از هیجان‌انگیزترین قلمروهای معماری موتورهای با کارایی بالا خوش آمدید: **مرزهای زمان اجرا (Runtime Boundaries)**.

تا به امروز، احتمالاً یونیتی را به عنوان یک اکوسیستم دیجیتال یکپارچه دیده بودید که در آن کد C# می‌نویسید، دکمه "Play" را می‌زنید و جادو اتفاق می‌افتد. اما در زیر آن سطح دوستانه و صیقل‌خورده، یک گسل مکانیکی عظیم نهفته است. یونیتی در واقع موجودی با "دو مغز" است. لایه بیرونی آن، جایی که اسکریپت‌های گیم‌پلی، هوش مصنوعی سفارشی و سیستم‌های رابط کاربری شما زندگی می‌کنند، توسط **زمان اجرای (Runtime) ایمن و مدیریت‌شده C# (.NET/Mono/IL2CPP)** اداره می‌شود. با این حال، هسته درونی و فوق‌سریع آن، که رندرینگ گرافیکی سه‌بعدی خام، ترکیب صدا، تبدیل ماتریس‌ها و ضربه‌های فیزیکی را مدیریت می‌کند، کاملاً به زبان **C++ بومی (Native)** و مدیریت‌نشده نوشته شده است.

هنگامی که بازی شما در حال اجراست، این دو دنیا باید دائماً با یکدیگر صحبت کنند. اگر دشمنی یک مانع فیزیکی را بشکند یا یک کلیپ صوتی به نشانگر پخش خاصی در عمق هسته C++ برسد، آن رویداد باید فوراً یک زنگ هشدار را در اسکریپت C# شما به صدا درآورد. **اشتراک رویدادهای چرخه حیات در مرزهای زمان اجرا** مطالعه نحوه ساخت خط‌لوله‌های ارتباطی فوق‌سریع، بدون تخصیص حافظه (Zero-Allocation) و تله‌پاتیک در سراسر این شکاف است، بدون اینکه باعث گیر کردن، لکنت یا از کار افتادن نرخ فریم بازی شما شود.

#### ۲. دانش کامپیوتر: شکاف، سفیر و عوارضی

برای درک سختی این موضوع، باید به دانش تاریخی کامپیوتر نگاه کنیم. در روزهای اولیه مهندسی نرم‌افزار، برنامه‌ها به یک زبان واحد نوشته می‌شدند. اگر زبان C بود، مستقیماً به کد ماشین کامپایل می‌شد و هر تابع می‌توانست مستقیماً به آدرس حافظه هر تابع دیگری بپرد.

سپس ظهور **ماشین‌های مجازی** و **زمان‌های اجرای مدیریت‌شده (Managed Runtimes)** (مانند جاوا و C#) اتفاق افتاد. این زبان‌ها یک "زندان مخملی" ارائه می‌کردند: کد شما را در یک لایه ایمن می‌پیچیدند که مدیریت حافظه را برای شما انجام می‌داد، خطاها را به‌طور خودکار بررسی می‌کرد و از کرش‌ها جلوگیری می‌کرد. اما آن‌ها دوراهی جدیدی را معرفی کردند: **یک برنامه مدیریت‌شده چگونه با یک جزء سیستم مدیریت‌نشده تعامل برقرار می‌کند؟**

دانشمندان کامپیوتر مکانیزمی به نام **Interop** (کوتاه شده برای Interoperability) را اختراع کردند که به‌طور خاص از طریق **P/Invoke** (Platform Invoke) و **Marshalling** اجرا می‌شود.

هسته بومی C++ را به عنوان یک پادشاهی جنگجوی باستانی و بسیار منضبط تصور کنید که فقط به گویش‌های باینری خام صحبت می‌کند. زمان اجرای مدیریت‌شده C# یک شهر فناوری مدرن و خودکار است که به شبکه‌های پیچیده اشیاء صحبت می‌کند. بین آن‌ها یک دیوار مرزی فیزیکی قرار دارد: **مرز زمان اجرا**.

هنگامی که یک اسکریپت C# می‌خواهد با یک روال موتور C++ صحبت کند، یک "سفیر" (Marshaller) باید مداخله کند. سفیر اجرای کد را متوقف می‌کند، شیء C# را به بیت‌های خامی که C++ می‌تواند درک کند ترجمه می‌کند، آن بیت‌ها را از مرز عبور می‌دهد و به آن‌ها تحویل می‌دهد. این فرآیند **Data Marshalling** نامیده می‌شود.

هشدار تاریخی این دانش واضح است: **عبور از مرز بسیار پرهزینه است.** این یک عوارضی فیزیکی است. اگر کد شما هزاران بار در هر فریم از این مرز عبور کند تا به رویدادها گوش دهد، CPU شما تمام وقت خود را صرف ترجمه پیام‌ها به جای اجرای بازی شما می‌کند.

#### ۳. مشکل اصلی: اداره گمرک کند و اشتراک‌های شبح

هنگامی که یک توسعه‌دهنده به سادگی تلاش می‌کند رویدادها را در سراسر این مرزها مدیریت کند، با دو تله معماری فاجعه‌بار مواجه می‌شود:

##### تله A: عوارضی عبور از مرز، حلقه فریم را متوقف می‌کند
فرض کنید یک زیرسیستم C++ سطح پایین (مانند یک شبیه‌سازی فیزیک سیالات سفارشی که روی سخت‌افزار پس‌زمینه اجرا می‌شود) نیاز دارد هر بار که یک مولکول مایع با یک شیء برخورد می‌کند، به ۵۰۰۰ موجودیت بازی در C# اطلاع دهد. اگر هسته موتور C++ عملیات خود را متوقف کند، یک خط لوله Interop استاندارد باز کند، هر بسته داده برخورد را به یک شیء Heap جدید C# ترجمه کند و یک رویداد Delegate استاندارد C# را فراخوانی کند، بازی کاملاً از کار می‌افتد. خط لوله اجرا مجبور می‌شود به مجموعه‌ای از توقف‌های کوچک و مداوم وارد شود، زیرا رجیسترهای CPU مدیریت‌نشده را با وضعیت ماشین مجازی مدیریت‌شده همگام‌سازی می‌کند.

##### تله B: نشت حافظه مرجع "شبح" (اشاره‌گرهای آویزان)
اشیاء C# پویا هستند؛ Garbage Collector (GC) عاشق این است که آن‌ها را جابجا کند یا زمانی که فکر می‌کند کسی به آن‌ها نگاه نمی‌کند، آن‌ها را پاک کند. حافظه C++ ایستا و صلب است؛ دقیقاً در جایی که قرار داده شده باقی می‌ماند تا زمانی که به‌طور دستی تخریب شود. اگر اشاره‌گر (آدرس حافظه) یک متد C# را به هسته موتور C++ پاس دهید تا C++ بتواند بعداً آن را فراخوانی کند، و سپس Garbage Collector C# آن شیء را در RAM جابجا کند تا فضا را فشرده کند، هسته موتور C++ همچنان آدرس قدیمی و کهنه را نگه می‌دارد. دفعه بعد که C++ سعی کند آن رویداد را شلیک کند، یک **اشاره‌گر آویزان (Dangling Pointer)** را فراخوانی می‌کند که به حافظه خالی یا خراب شلیک می‌کند و باعث کرش فوری و غیرقابل کنترل به دسکتاپ می‌شود.

#### ۴. راه نجات معماری: اشاره‌گرهای تابع ایستا و رجیستری‌های Blittable

برای دور زدن کامل عوارضی Interop و تأمین ثبات مطلق زمان اجرا، مهندسان سیستم یک **ماتریس فراخوانی بدون تخصیص حاکم (Sovereign Non-Allocating Callback Matrix)** می‌سازند. به جای پاس دادن اشیاء سنگین یا Delegate‌های پویا به جلو و عقب مرز، ما از سه قانون معماری پیشرفته استفاده می‌کنیم:

۱. **طرح‌بندی انواع Blittable:** ما اطمینان حاصل می‌کنیم که هر بسته داده رویداد که از مرز عبور می‌کند، از انواع داده‌های "Blittable" (مانند `int`، `float` یا `byte`) ساخته شده است. این انواع دارای نمایش باینری یکسانی در هر دو C++ و C# هستند، به این معنی که توسط سفیر Marshalling *صفر ترجمه* نیاز دارند. آن‌ها مستقیماً از امنیت مرزی عبور می‌کنند.
۲. **اشاره‌گرهای تابع بومی (`delegate* unmanaged`):** به جای استفاده از اشیاء رویداد استاندارد و سنگین C#، ما آدرس حافظه سخت‌افزاری خالص و خام یک متد C# را با استفاده از اشاره‌گرهای تابع مدیریت‌نشده مدرن استخراج می‌کنیم. ما این متدهای C# را با ویژگی‌های خاصی مانند `[UnmanagedCallersOnly]` علامت‌گذاری می‌کنیم. این به کامپایلر می‌گوید که این آدرس اجرا را در جای خود منجمد کند، که آن را برای هسته موتور C++ دقیقاً مانند یک تابع بومی C++ قابل خواندن می‌کند.
۳. **رجیستری‌های شاخص دوطرفه:** به جای پاس دادن مراجع اشیاء، ما یک ID عددی ساده (Entity ID یا Instance ID) را پاس می‌دهیم. C++ رویداد را با شلیک یک اشاره‌گر تابع خام و پاس دادن یک عدد ساده فعال می‌کند. C# آن عدد را دریافت می‌کند و فوراً از آن برای جستجوی موجودیت هدف در یک آرایه پیش‌تخصیص‌یافته در Stack یا Heap محلی استفاده می‌کند و به جستجوی $O(1)$ با **تولید زباله (Garbage) صفر مطلق** دست می‌یابد.

#### ۵. پیاده‌سازی معماری جامع

بیایید یک شبیه‌سازی با کارایی بالا و در سطح تولید از این مرز بسازیم. ما یک هسته رویداد موتور بومی سطح پایین (شبیه‌سازی لایه C++) و یک پل مرزی زمان اجرای C# حاکم ایجاد می‌کنیم که رویدادهای چرخه حیات (مانند فریم‌های فیزیک عمیق موتور یا محرک‌های مقداردهی اولیه) را به طور کامل بدون تولید حتی یک بایت زباله یا ریسک خرابی حافظه مدیریت می‌کند.

##### 👑 معماری مرزی حاکم (بدون تخصیص و الگوی بافر دستور)

در زیر سیستم بسیار بهینه‌شده آورده شده است. ما از یک بافر دستور (Command Buffer) مشترک و پیش‌تخصیص‌یافته برای تبادل رویدادها استفاده می‌کنیم و Delegate‌ها، Action‌ها و ثبت‌نام‌های Callback را کاملاً دور می‌زنیم. اگر این روش به عنوان یک صف تک‌تولیدکننده/تک‌مصرف‌کننده پیاده‌سازی شود، ذاتاً ایمن برای رشته‌ها (Thread-Safe) است و به Garbage Collection صفر نیاز دارد.

```csharp
using System;
using System.Runtime.InteropServices;

namespace SovereignEngine.Core
{
    // ۱. طرح‌بندی را تعریف کنید که از نظر ریاضی در هر دو لایه سخت‌افزاری C++ و C# یکسان باشد
    [StructLayout(LayoutKind.Sequential)]
    public struct BlittableLifecycleEvent
    {
        public int EntityId;
        public int PhaseIndex;
        public float RawTimestamp;
    }

    // ۲. سمت گیرنده رویداد C# با کارایی بالا، با استفاده از بافر دستور (Polling)
    public static class GameplaySystemRegistry
    {
        private const int MAX_EVENTS = 1024;
        
        // بافر حافظه پیش‌تخصیص‌یافته برای نگهداری دستورات ورودی، کاملاً ایستا
        private static BlittableLifecycleEvent[] _commandBuffer = new BlittableLifecycleEvent[MAX_EVENTS];
        private static int _writeIndex = 0;
        private static int _readIndex = 0;

        // موتور بومی (یا لایه Interop) یک رویداد را به بافر می‌فرستد
        public static void PushNativeEvent(BlittableLifecycleEvent nativeEvent)
        {
            _commandBuffer[_writeIndex] = nativeEvent;
            _writeIndex = (_writeIndex + 1) % MAX_EVENTS;
        }

        // سیستم C# این بافر را در طول حلقه اصلی Update بررسی می‌کند
        // اینجاست که همگام‌سازی اتفاق می‌افتد، در سمت مدیریت‌شده (Managed)
        public static void ProcessCommands()
        {
            while (_readIndex != _writeIndex)
            {
                BlittableLifecycleEvent cmd = _commandBuffer[_readIndex];
                
                // اجرای منطق مستقیماً بر اساس داده‌های دستور
                ApplyLogic(cmd);
                
                _readIndex = (_readIndex + 1) % MAX_EVENTS;
            }
        }

        private static void ApplyLogic(BlittableLifecycleEvent e)
        {
            // پردازش مستقیم داده‌ها، بدون درگیر شدن Delegate‌ها یا اشیاء رویداد
        }
    }

    // ۳. شبیه‌سازی هسته اجرای موتور بومی C++ عمیق
    public unsafe class NativeEngineSimulator
    {
        public void ExecuteHighFrequencyNativeLoop(int targetEntity, int currentPhase, float timestamp)
        {
            // آماده‌سازی داده‌ها مستقیماً روی Stack سخت‌افزاری
            BlittableLifecycleEvent hardwareEvent;
            hardwareEvent.EntityId = targetEntity;
            hardwareEvent.PhaseIndex = currentPhase;
            hardwareEvent.RawTimestamp = timestamp;

            // رویداد را مستقیماً به رجیستری بافر مشترک بفرستید
            GameplaySystemRegistry.PushNativeEvent(hardwareEvent);
        }
    }
}
```

#### ۶. خلاصه تغییرات معماری

| ابعاد | مرز تخصیص ساده (Naive) | معماری سریع بومی حاکم (Sovereign) |
| --- | --- | --- |
| **ماتریس تخصیص** | **تورم سنگین Heap:** اشیاء جدید، بسته‌های داده و رشته‌ها را در هر فراخوانی رویداد ایجاد می‌کند. | **خط صاف مطلق ($O(1)$):** از داده‌های ساختاری تخصیص‌یافته روی Stack استفاده می‌کند. فشار روی Garbage Collector صفر است. |
| **سرعت مرزی** | **عوارضی ترجمه سنگین:** لایه Marshalling موتور را مجبور به تأیید ساختارها می‌کند. | **پرش آنی:** آدرس‌های ماشین بومی خالص را از طریق خط‌لوله‌های اجرای سخت‌افزاری استاندارد فعال می‌کند. |
| **امنیت حافظه** | **شکننده و بی‌ثبات:** آسیب‌پذیری بالا در برابر لکنت‌های جابجایی GC یا کرش‌های ناشی از اشاره‌گرهای خراب. | **ایمن و منجمد:** متدها با نشانگرهای زمان کامپایل کاملاً قفل شده‌اند و ایمنی بومی دائمی را تأمین می‌کنند. |
| **پروفایل رشته‌ها** | محدود به Main Thread. تلاش برای فراخوانی رویدادها از پس‌زمینه بومی باعث کرش‌های مهلک زمان اجرا می‌شود. | کاملاً همزمان (Concurrent). رشته‌های پس‌زمینه فیزیک یا رندر بومی می‌توانند با ایمنی رویدادها را فعال کنند. |
